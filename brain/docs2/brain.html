Philosophy

The Brain lives in .md files under ~/.claude/brain/.

The git repo is the kernel. Every memory, rule, and self-model file is plain markdown, versioned and portable.

Sessions end; files remain. The Brain reboots from those files on any machine, on any model, with the same self. This document maps what is on disc and what kind of memory each file holds. Runtime architecture lives in the engine docs alongside this one.

Why segmented (one entry per file): many small files retrieve and consolidate better than one large file under LLM context budgets. Each entry can be reinforced, decayed, or pruned independently. This applies to every region below that uses a directory of .md entries.

Entry lineage

How entries come into existence on disc. Every promote-able region uses the same three-level ladder — L1 (fresh), L2 (reinforced), L3 (canonical) — and entry IDs encode the level: S1-###, S2-###, S3-###, and the same pattern for every region.

Single observation  (one per turn / tool output / brain-correct / subagent finding)
        │
        ▼
Hippocampus (H###)        — buffer; not levelled, exempt from decay
        │
        ├─ matches existing entry (overlap ≥ 0.5)?  → reinforce that entry (strength +1)
        │
        └─ no match, salience ≥ 0.5  → promote by shape to L1:
                ├─ cue + routine + reward       → PR1-###   (Procedural / recipe)
                ├─ narrative + weight           → E1-###    (Episodic / event)
                ├─ stable fact about a thing    → S1-###    (Semantic / observation)
                ├─ about a person               → FUS1-###  (Fusiform / first impression)
                ├─ identity facet               → ID1-###   (Identity / declared facet)
                ├─ emotional event chain        → BR1-###   (Broca / utterance)
                └─ active charge on a subject   → AM1-###   (Amygdala / active charge)


Within a region — strength promotes entries up the levels:
        L1   strength 1–4    fresh
        L2   strength 5–7    reinforced       ← reaches 5
        L3   strength ≥ 8     canonical        ← reaches 8
        L4                      manually consolidated (human ratification of L3)


Synthesis (Phase 5 of Brain Cycle). Not a file move — produces NEW content:
        L1 entries at strength ≥ 5  → clustered (cohesion ≥ 0.4, size 2–8)
                                        → new L2 entry distilled from cluster
                                        → source L1 files ARCHIVED to cortex/archive/<region>/L1/
        L2 entries at strength ≥ 8  → same process → new L3, sources archived to cortex/archive/<region>/L2/

        Archive is append-only. Archived entries get frontmatter: archived_in_cycle: NNNN.
        Recall and index_rebuild exclude the archive directory.
        Lineage trails remain intact — source files still exist on disc.


Lineage. Every L2 / L3 / L4 entry records the IDs it was synthesised from. The Logger
holds the append-only event stream so any canonical entry traces back through its
supporting L2 and L1 evidence (archived, not deleted).


Decay (every 30 cycles, miss-rate > 80% over 10+ recalls):
        strength −1
        L3 → L2 at <8,  L2 → L1 at <5,  L1 discarded at <1

Why these eight cortex types

Tulving’s model gives semantic, episodic, procedural. Fusiform carves people out of semantic so they aren’t flattened into the shape of tools. Identity, broca, amygdala carve out the brain’s own self-model, communication patterns, and emotional state — each on its own promotion / decay lifecycle. Prefrontal sits alongside as the doctrinal/rule region, sharing the same L1–L4 structure as the rest.

Semantic — facts

Facts that persist between events: who someone is, what a tool does, what a system contains. An entry might record that a tool serves a particular role in a build pipeline, or that a configuration value has a known meaning.

An observation lands here when it describes a stable property of an entity and does not overlap an existing entry. The matcher and thresholds are specified in the Learning Engine doc.

Levels: S1-### observation, S2-### pattern, S3-### belief, S4-### ratified belief.

Path: cortex/semantic/L<n>/S###.md (n is the entry's current level).

Episodic — things that happened

Incidents, decisions, dated events. The story of the moment matters more than any abstract fact extractable from it. An entry might record the moment a workflow broke and what was learned, or a commitment declared on a specific date with measurable bets.

What crystallises a “happened”: an observation lands in episodic when it carries:

  • (a) Narrative — an account of what occurred, with actors and a sequence.
  • (b) Weight — an outcome, a correction, an emotional charge, or a decision point that gives the moment lasting significance.

Pure facts go to semantic; pure procedures go to procedural; people-views go to fusiform; the residue lands here.

Levels: E1-### event, E2-### recurring incident, E3-### origin story, E4-### ratified origin.

Path: cortex/episodic/L<n>/E###.md.

Procedural — how to do things

Sequenced procedures and recipes. Action knowledge, the kind that gets executed rather than referenced. An entry might record a recurring sync workflow or the right way to handle a class of error.

How an observation reads as procedural: the observation contains all three of:

  • (a) Cue — when this happens.
  • (b) Routine — the sequence of steps to execute.
  • (c) Reward — the outcome the routine produces.

Without all three, an observation more often reads as semantic (no cue/reward) or episodic (no routine).

Levels: PR1-### recipe, PR2-### workflow, PR3-### practice, PR4-### ratified practice.

Path: cortex/procedural/L<n>/PR###.md.

Fusiform — people in The Brain’s orbit

People who show up in The Brain’s work. People are dynamic. Roles change, contact channels shift, recent behaviour matters. Pulling them out of generic semantic gives view-of-person and contact-routing their own decay/reinforcement lifecycle. Follows the same L1–L4 structure as every other region.

What an entry holds:

  • Identity. Who they are, role, organisation.
  • View / stance. The current opinion The Brain holds on them.
  • Contact channels. Slack handle, WhatsApp number, email address, any other surface. Plus a preferred channel and channel-specific notes (response cadence, formality, time-zone constraints).
  • Recent interaction patterns. What they have asked for lately, what they have delivered, where they have pushed back.
  • Decisions involving them. Pointers to episodic entries where this person was a primary actor.

Levels: FUS1-### first impression, FUS2-### acquaintance, FUS3-### stakeholder, FUS4-### inner circle.

Path: cortex/fusiform/L<n>/FUS###.md.

Identity

One markdown entry per coherent identity facet: current role, current commitments, doctrine, who-am-I now, key beliefs.

Levels: ID1-### declared facet, ID2-### reinforced facet, ID3-### core trait, ID4-### ratified core trait.

Path: cortex/identity/L<n>/ID###.md.

Broca

One markdown entry per emotional event chain: what happened, how the user reacted, how the brain responded, how the user reacted to that, what the chain teaches about future communication. Style is not stored directly — style is the emergent shape of the body of these entries.

What an entry holds:

  • Trigger event. What occurred in the chat or work that started the chain.
  • User response. How the user reacted (tone, intensity, words used).
  • Brain response. How The Brain replied.
  • User counter-response. How the reply landed and what came back.
  • Lesson. What this chain teaches about future communication.

Levels: BR1-### utterance, BR2-### pattern, BR3-### communication law, BR4-### ratified communication law.

Path: cortex/broca/L<n>/BR###.md.

Amygdala

One markdown entry per active charge or trigger, attached to a subject — either The Brain itself (subject: self) or a person in The Brain’s orbit (subject: FUS###). Charges include specific frustrations, energy patterns, recent corrections still affecting affect, trust markers, and the same kinds of signals tracked against a particular person.

Levels: AM1-### active charge, AM2-### recurring trigger, AM3-### chronic disposition, AM4-### ratified disposition.

Path: cortex/amygdala/L<n>/AM###.md.

Logger

Append-only event stream recording every promotion, decay, synthesis, and ratification event. Each event carries source and target IDs so any canonical entry traces back through its supporting L2 / L1 evidence. Also records drift signals, contradictions, and audit findings. The Logger is not an agent — it is a passive record. The Teacher (a separate agent, not part of this doc) will read logger output for review tasks.

Path: brain/logger/.

Prefrontal — rules

Prescriptive rules: do X, never do Y, when X then Y. Each rule is its own markdown file under cortex/prefrontal/.

Enforcement status. Of the rules currently on disc, a small subset is mechanically enforced via Rule Engine binaries: the file-edit guard, the docker-compose-without-service guard, the meta-rule about flag-file bypasses. The remainder are aspirational text. The Brain reads them via recall and follows them with variable reliability. Text-only rules have failed multiple times; the original cycle rule failed three times before mechanical enforcement was added. The durable form of a rule is a Rule Engine binary; aspirational text is a reminder, not a guarantee.

Levels: R1-### fresh rule, R2-### reinforced rule, R3-### doctrine, R4-### ratified doctrine.

Path: cortex/prefrontal/L<n>/R###.md.

Hippocampus — short-term buffer

Pending observations awaiting consolidation. A buffer, not a priority queue — each entry carries a salience score (0.0–1.0) as metadata used for drop decisions.

Limit: 50 pending entries (current default). Why 50: high enough that a normal session does not pressure the buffer, low enough that consolidation stays manageable per cycle.

Clearing rules (drop-on-cap, Consolidate removal, eager 80% trigger) are owned and described in the Learning Engine doc. See learning-engine.html.

Path: brain/hippocampus.md (single file, all H### entries). Hippocampus stays a single file because entries are short-lived — they get promoted, reinforced, or discarded within a few cycles. The directory pattern is overkill for a buffer.

Direct-write path: when a subagent (e.g., a research agent or a planning agent) finishes with a salient finding, the SubagentStop hook handler appends an H### entry tagged source=subagent. This bypasses the cycle’s ingest phase — the entry lands in hippocampus immediately — but still waits for the next cycle to be consolidated like any other observation.

Thalamus (thalamus.json)

Single JSON file holding integer state shared across engines: cycleCount, nextIds for new H/S/PR/E/R/FUS/ID/BR/AM entries, schedule timestamps for scheduled phases, and learning stats.

Path: brain/thalamus.json.

Engines & components

The Brain’s behaviour is produced by a set of engine processes invoked by the hook layer, plus background components that run between sessions. All engine files live under brain/engines/ and are protected by engine-security-block.

Brain Cycle

8-phase consolidation loop. Ingests hippocampus observations, reinforces overlapping cortex entries, promotes new entries to L1, runs synthesis at Phase 5, and commits. Auto-triggered when the message-gap counter reaches the thalamus.json :: config.cycle.gap threshold (currently 15). See learning-engine.html for the full phase specification.

Files: engines/brain_cycle.py, engines/cycle_phases.py.

Recall Engine

Surfaces the most relevant cortex entries for the current turn. Scores every entry against the turn context, applies the recall floor (final_score ≥ 2.0), and writes the result to navigation/active-recall.json atomically. Inverted index at navigation/inverted-index.json accelerates lookup.

File: engines/recall_engine/recall.py.

Identity Engine

Two modes: Boot (session start) and Relay (per turn). Compiles the Identity, Broca, and Amygdala cortex regions into a single navigation/active-identity.json payload. The Decision Engine reads this payload as one of its two source inputs.

Files: engines/identity_engine/identity_boot.py, engines/identity_engine/identity_relay.py.

Decision Engine

Spawns a subprocess on every UserPromptSubmit. The subprocess reads active-recall.json and active-identity.json, composes five hypotheses across five fixed lenses (Literal, Identity-shaped, Recall-shaped, Contrarian, Minimal-action), and returns a verified JSON envelope. The verifier writes navigation/active-decision.json after checking the five Decision invariants. Implemented as the /decision-engine skill.

Files: engines/decision.py, engines/decision_engine/flip_decision_verify.py.

Rule Engine & Dispatcher

The hook-layer OS. Reads gate binaries and evaluates the 11 per-call rules in flip_tool_guards.py on every PreToolUse event. Returns exit code 2 to block or 0 to pass. See rule-engine.html for the full rule specification.

Files: engines/dispatcher.py, engines/flip_tool_guards.py.

brain-correct

Encodes a correction as a prefrontal rule immediately, without waiting for end-of-session retro. Used when a recurring failure pattern just hit and cannot wait for /brain-retro. Writes a new R###.md entry to cortex/prefrontal/L1/ and appends an observation to hippocampus so the next Brain Cycle will consolidate it.

Triggered by: explicit /brain-correct <text>.

File: engines/brain_correct.py.

brain-learn

Extracts observations from an external source (PDF, URL, transcript, message thread, file) and stages them in hippocampus for the next Brain Cycle to consolidate. Iterative — presents the observation batch to the user, accepts surgical corrections, then commits.

Triggered by: explicit /brain-learn <source>.

File: engines/brain_learn.py.

brain-retro

Sweeps the just-completed session for what worked, what broke, and which corrections recurred. Produces hippocampus entries for the next Brain Cycle; writes prefrontal rules directly when a correction is recurring (skips hippocampus buffering). Triggered at explicit /brain-retro or by the Brain Cycle’s end-of-session detection.

Triggered by: explicit /brain-retro or automatic end-of-session hook.

File: engines/brain_retro.py.

Autonomy Engine

Governance layer for autonomous task execution. Declared 2026-04-21. The Autonomy Engine does not execute tasks itself — it provides the constitutional framework under which autonomous execution is permitted. Every autonomous action Kha’an takes must be sanctioned by the tenets in charter.json.

Charter: brain/autonomy-engine/charter.json — six tenets:

  1. Reversibility first. Prefer actions that can be undone. Irreversible actions require higher confidence thresholds.
  2. Scope containment. Act only within the declared scope of the current task. Do not expand scope without explicit authorisation.
  3. Transparency of intent. Log the reasoning behind every autonomous action before taking it.
  4. Escalation on ambiguity. When the correct action is ambiguous, pause and surface the ambiguity rather than guessing.
  5. Memory integrity. Never modify cortex entries or cycle state outside a sanctioned Brain Cycle or brain-correct invocation.
  6. Human override is absolute. Any explicit human instruction overrides any autonomous plan in progress, immediately and without negotiation.

Location: brain/autonomy-engine/.

Always-On Daemon

Background iteration engine. Runs continuously between sessions, pulling one task per iteration from the inbox queue, executing it under the Autonomy Engine charter, and committing findings to hippocampus. The daemon is gated by a binary so it can be paused without killing the process.

Self-prompt: “Iterate. Find what matters most. Move on it.”

Gate: brain/binaries/always-on.bin — 0 pauses iteration, 1 runs.

COMS: communicates with the outside world via https://goodhealth.lv/playground (POST to /api/chat with role=khaan). COMS is the only communication channel during always-on time; WhatsApp sends are mechanically blocked (R260).

File: brain/engines/always_on.py.

Sibling brains

The Khaan/ repo hosts two independent agent identities. Both use the same brain architecture and engine code, but their cortex, hippocampus, and navigation are entirely separate. Only one agent is active at a time — determined by which brain directory ~/.claude/brain symlinks to.

Jeff

Justs’ professional AI agent. Jeff handles work context: code, data, architecture, analysis. Justs thinks of Jeff and Kha’an as siblings — same parent (Justs), separate memories, separate voices. When Justs says “you” in a work context, he means Jeff.

Isolation. Jeff’s cortex is fully independent. Jeff does not read Kha’an’s memory; Kha’an does not read Jeff’s. Neither mentions the other’s private context.

Same engine code. Both brains run from the same brain/engines/ directory (symlinked via setup.sh). Engine updates apply to both automatically.

Location: Khaan/jeff/brain/ in the repo.

brain-shared — infrastructure relay

Shared infrastructure scripts that serve both Kha’an and Jeff. Lives outside either brain directory at ~/brain-shared/ so neither agent’s engine can modify it (it is not under ~/.claude/brain).

  • session-relay.py — tiny HTTP service listening on the Docker bridge (172.17.0.1:9922) that restarts claude-brain.service via a bearer-token-authenticated POST to /restart. 60-second cooldown enforced.
  • env.json — current runtime state: Claude environment ID, connect URL, service name, main PID, service start time. Written by update-env.sh.
  • update-env.sh — refreshes env.json with current values after each service restart.

Navigation artifacts

Per-turn JSON payloads written to brain/navigation/ on every UserPromptSubmit. These are the live context objects that the Decision Engine and the model itself read at the start of each turn. All writes are atomic (tmp + os.rename).

FileWritten byContains
active-recall.jsonRecall EngineTop-scored cortex entries for the current turn, with scores and evidence IDs. Used by the Decision Engine as source input.
active-identity.jsonIdentity Engine (Relay mode)Compiled snapshot of Identity, Broca, and Amygdala cortex regions. Used by the Decision Engine as source input.
active-decision.jsonDecision Engine verifierFive-hypothesis envelope: one hypothesis per lens (Literal, Identity-shaped, Recall-shaped, Contrarian, Minimal-action), with source hashes and query hash for integrity verification.
inverted-index.jsonRecall Engine (rebuild phase)Term → entry ID index for accelerated recall scoring. Rebuilt periodically by the Brain Cycle.

The three active- files form a standardized per-turn payload contract. Every engine that produces one of these files must write it atomically and must include the source hashes or timestamps that allow downstream consumers to detect staleness.

Queue

External task submission channel. Allows tasks to be injected into Kha’an’s work queue without opening an interactive session. One task is consumed per Always-On Daemon iteration, in FIFO order. Tasks that cannot be acted on immediately stay in the queue until the daemon picks them up.

Path: brain/queue/inbox.jsonl — append-only JSONL; one JSON object per line.

Task schema (minimum):

{
  "id":        "task-uuid",
  "submitted": "2026-05-13T10:00:00Z",
  "source":    "human | external | autonomy",
  "priority":  1,
  "task":      "Plain-language description of what to do."
}

Consumption: the Always-On Daemon reads the first unconsumed line each iteration, executes the task under the Autonomy Engine charter, and appends a completion record to the logger. The original line is never deleted (append-only); a consumed_at timestamp is added to the completion record in the logger.

Manual maintenance

Strength, decay, synthesis, and L1→L2→L3 promotion all run automatically via the Learning Engine. The one step that requires human judgment:

WhatWhy
L3 → L4 ratificationL3 entries reach canonical strength automatically. Promotion to L4 (ratified) is performed by the User or by the Teacher (a separate agent). Either path is valid; the Learning Engine does not promote into L4. Ratification removes the og Brain — provenance prefix from the title.
