Philosophy

The Decision Engine compiles sources from the other engines into an evidence-based assessment of the user’s needs.

Five hypotheses, each grounded in a different blind spot. No internal ranking.

No tool runs until the assessment is complete.

The user’s literal message is rarely the user’s full intent. The engine’s job is to read the message through five framed lenses, ground each in evidence from the other engines, and surface all five upstream. The five lenses are designed to cover three distinct categories of blind spot — an unbiased read, two input-lensed reads, two cognitive heuristics — so that no class of mistake goes unmonitored.

The “no tool until the assessment is complete” property is mechanical, not aspirational. Tool gating runs in three sequential checkpoints: (1) Recall and Identity must each finish writing their active-*.json for THIS turn before Decision is allowed to launch. (2) The Decision subprocess reads those two files and produces all 5 hypotheses; only when all 5 are present does the SubagentStop verifier run. (3) Verification must pass before the parent’s tool gate is released. No path lets the parent answer first and assess later — and no path lets Decision run on stale or missing inputs.

Inputs

The Decision Engine reads two files. It does not call other engines. Without both files, the engine raises decision-source-missing and the turn falls through to Anti-lockout.

File pathWhat is in it
navigation/active-recall.jsonA RecallResult payload from the Recall Engine: qualified_entries[] (id, path, gist, full body, score breakdown), scoring_trace, source hashes, ts.
navigation/active-identity.jsonThe Brain’s published self-model + the 6 charter tenets, from the Identity Engine.

Rules

How the Decision Engine behaves on every turn. Each rule has a title; mechanics are detailed in the sections below.

Decision fundamentals

  • One decision per turn. Fires exactly once per UserPromptSubmit.
  • Five hypotheses or none. Mechanically enforced by 5 binaries (one per lens). The parent stays blocked while any of the 5 binaries are at 1.
  • Both inputs or fail. Both input artifacts must be readable. Missing input → decision-source-missing.bin = 1.
  • No ranking, no choosing. All 5 hypotheses are surfaced to the parent (The Brain). The parent picks based on the live chat context the subprocess does not have.
  • Output is context, not an answer. The engine never drafts a user-facing reply. It hands the parent five framed hypotheses to choose between.

Subagent contract

  • Parent tool gate. While any hypothesis binary is at 1, the parent’s Agent tool surface is blocked. The skill’s internal subprocess launches (Recall, Identity, Decision via claude -p) bypass this gate because they run inside the hook process, not via the parent.
  • SubagentStop verification. All 5 hypothesis binaries at 0 AND a schema-valid return payload AND all 5 lens labels present is the only state that unblocks the parent.
  • Subprocess pinned. Each Decision subprocess runs as claude -p --output-format json via the local CLI. The CLI selects the model from your subscription session (typically Opus when available); no Anthropic API key is read or consumed. The wrapper layer is engines._shared.cli_invoke().
  • Write-path gated. Only the Decision subprocess’s own session can flip the hypothesis binaries.

Anti-lockout

Three protections, all simple. Worst case the parent answers without a fresh decision (degraded), never locked out.

ProtectionWhat it does
Three-strike forgivenessAfter 3 consecutive subprocess failures, decision-failed.bin = 1. Next UserPromptSubmit clears all Decision binaries; the parent runs without a fresh decision that turn (degraded but unblocked).
120-second timeoutPer subprocess attempt. If exceeded, decision-timeout.bin = 1. Next UserPromptSubmit clears.
SessionStart resetAll Decision binaries written to 0 at session start.

Every fallback writes a log line so the failure mode is always visible.

The 4-D methodology

Each of the 5 hypotheses runs through all four phases.

1. DECONSTRUCT

Extract core intent and key entities from the message. Identify output requirements and constraints. Map what is provided versus what is missing.

2. DIAGNOSE

Audit clarity gaps and ambiguity. Check specificity and completeness. Assess structure and complexity needs. Surface what would block a clean execution if not resolved.

3. DEVELOP

Pick the optimization technique by request type:

TypeTechnique
CreativeMulti-perspective + tone emphasis
TechnicalConstraint-based + precision focus
EducationalFew-shot examples + clear structure
ComplexChain-of-thought + systematic frameworks

Assign role/expertise. Enhance context. Apply logical structure.

4. DELIVER

Construct an optimized prompt for the parent to consider. Format based on complexity. Include implementation guidance. Length is free-form; no sentence cap.

Hypothesis generation

Five lenses are produced per turn. The Decision Engine does not pick. All 5 are surfaced to the parent (The Brain) along with their 4-D subcontext; the parent chooses which lens fits the live chat context.

Why these 5 (designed coverage, not arbitrary)

LensSource of insightBlind spot it covers
LiteralThe message text aloneOver-reading: assuming intent the words do not carry
Identity-shapedIdentity Engine output (active-identity.json)“What does this user usually mean?” — pattern recognition
Recall-shapedRecall Engine output (active-recall.json)“What does cortex evidence say is being asked?” — fact-grounded read
ContrarianAdversarial heuristicConfirmation bias — what if the obvious read is wrong?
Minimal-actionOccam’s-razor heuristicScope creep — what is the smallest action that satisfies?

One unbiased read (Literal) + two input-lensed reads (Identity-shaped, Recall-shaped) + two cognitive heuristics (Contrarian, Minimal-action). Each lens covers a different category of blind spot. Drop one and a category goes unmonitored.

How the lenses are enforced (three layers, soft to hard)

  1. System prompt (soft). The Decision subprocess’s pinned system prompt enumerates the 5 lenses by name with their definitions and explicit instructions: produce exactly one hypothesis per lens, in this order, max 4 sentences each.
  2. Schema validation (hard). The return payload schema requires hypotheses[] to contain exactly 5 entries with label ∈ {Literal, Identity-shaped, Recall-shaped, Contrarian, Minimal-action} (one each, no duplicates, no extras). The SubagentStop verifier rejects anything else.
  3. Binaries (hard). The 5 hypothesis binaries map 1:1 to the lenses by index. The subprocess flips each as it completes that lens. If one stays at 1, the parent stays blocked.

A subprocess that ignores the system prompt fails schema validation. One that fakes the schema fails the binary check. One that fakes the binaries cannot — writes to those files are gated to its own session.

Delivery to the parent

The subprocess’s final return value (the JSON envelope from claude -p --output-format json) is the structured payload of all 5 hypotheses + their 4-D subcontext. The skill writes this payload to active-decision.json and returns it as additionalContext for the parent’s next reply turn. The Decision Engine does not pick a lens because the subprocess does not have the live chat context.

Stages

Per-turn lifecycle.

T0   UserPromptSubmit fires; /decision-engine skill activates.
T1   Skill checks state.json :: active_decision_nonce. If held, wait up
     to 240s; if 240s elapses, skill exits. Otherwise take nonce.
T2   Skill blocks parent tools (decision-hypothesis-1..5 = 1).
T3   Skill spawns Recall and Identity in parallel; joins both.
T3.5 Freshness check: verify each active-*.json was written for THIS
     turn (query_hash match). Stale file raises decision-source-
     missing, clears hypothesis bins, exits without subprocess.
T4   Skill launches Decision subprocess (claude -p with pinned prompt).
T5   Subprocess computes all 5 hypotheses; returns JSON envelope.
T6   flip-decision-verify runs single schema check (incl. tool_uses).
T7   On pass: clear all 5 hypothesis bins; write active-decision.json;
     release nonce; tools unblock.
T8   Skill returns; Claude Code sends user prompt + additionalContext
     to the parent; parent emits N's reply with N's decision in context.

Worst case: subprocess fails 3 times or runs past 120s; Anti-lockout in Rules covers the recovery.

Decision Engine skill

The Decision Engine ships as a skill named /decision-engine. Activation is hook-driven and automatic: every UserPromptSubmit invokes the skill before returning control to the Claude Code. The agent never chooses to run the skill — it always runs, exactly once per turn.

What the skill does, in order:

  1. Mutex acquire. Read state.json :: active_decision_nonce. If held by an in-flight earlier turn, wait for release (see Mutex below). If unset, atomically take a fresh nonce.
  2. Spawn Recall and Identity in parallel. Both engines run as Python threads inside the skill process. Each writes its active-*.json for THIS turn. Skill joins both threads before continuing. After joining, a freshness check verifies each active-*.json carries the query_hash for THIS turn; a stale file raises decision-source-missing.bin = 1, clears hypothesis bins, and exits cleanly without launching the subprocess.
  3. Spawn the Decision subprocess. Run claude -p --output-format json via engines._shared.cli_invoke(), passing the pinned SYSTEM_PROMPT + the just-written active-recall.json and active-identity.json as inputs.
  4. Verify. When the subprocess returns, run flip-decision-verify on the returned JSON envelope. The hook event is named SubagentStop by Anthropic; semantically the verifier runs against the CLI subprocess’s return payload (one schema check — see invariant verification below).
  5. Write or reject. On verifier pass: write active-decision.json; clear the 5 hypothesis binaries; release the mutex. On fail: increment strike counter; retry up to 3 times; after 3 strikes, raise decision-failed.bin = 1 — the next UserPromptSubmit clears it (Anti-lockout).
  6. Mutex release. In finally, clear active_decision_nonce. Always runs.
  7. Return. Hook returns; Claude Code sends user prompt + composed additionalContext to the parent.

Where the code lives: the skill orchestrator is engines/decision.py::dispatch(). The verifier is flip-decision-verify (SubagentStop hook, registered in ~/.claude/settings.json). State and binaries live under brain/state.json and brain/binaries/.

Mutex

The skill holds state.json :: active_decision_nonce for the duration of its engine pipeline. New skill invocations check the nonce on entry. If held, they wait for release (polling every 250 ms, up to 240 s). If 240 s elapses with the nonce still held, the new invocation exits cleanly with no engines run. No force-clear, no kill-semantics, no recovery. KISS.

Mechanism:

  • Wait on entry. Read active_decision_nonce. If held: time.sleep(0.25) in a loop. If 240 s elapsed: sys.exit(0).
  • Acquire. Atomically write active_decision_nonce = secrets.token_hex(16) + last_decision_start_ts = now via tmp+rename (POSIX-atomic).
  • Release. In finally, clear the nonce. Always runs.
  • Stuck nonce. If a hook is killed by Claude Code mid-run, finally never executes; nonce stays held; subsequent hooks die after 240 s. Manual SSH clear required: state.json :: active_decision_nonce = null.

SubagentStop verification — flip-decision-verify

The flip-decision-verify hook runs ONE check before unblocking the parent: the returned 5-hypothesis payload validates against the Decision schema. Schema requires:

  • Exactly 5 hypotheses present.
  • Lenses [Literal, Identity-shaped, Recall-shaped, Contrarian, Minimal-action] each present once and only once.
  • Source-hash fields match the active-recall.json and active-identity.json that existed at subprocess launch.
  • query_hash matches the user message.
  • Tool-uses audit. The subprocess’s claude -p output envelope’s tool_uses array must include:
    • Read(navigation/active-recall.json)
    • Read(navigation/active-identity.json)
    • Read(<path>) for every cortex entry ID appearing in any hypothesis’s evidence field.
    A subprocess that emits an evidence ID without a corresponding Read on that ID’s file fails verification — it cannot have actually consulted the entry.

If schema validation fails: payload is rejected, no active-decision.json write happens, decision-failed.bin = 1; the next UserPromptSubmit clears it (Anti-lockout).

Infrastructure

Return payload, hooks, binaries, caching.

Return payload schema

The subprocess returns the structured payload directly as the JSON envelope from claude -p --output-format json. The skill writes it to active-decision.json and to the logger. There is no separate audit file.

{
  "ts": "...",
  "query_hash": "sha256(lower(message))",
  "hypotheses": [
    {
      "id": "H1", "label": "Literal",
      "deconstruct": "...", "diagnose": "...",
      "develop": "...", "deliver": "...",
      "evidence": []              # IDs of cortex entries this lens cites
    },
    { "id": "H2", "label": "Identity-shaped", "evidence": ["ID-final-3"], "...": "..." },
    { "id": "H3", "label": "Recall-shaped",   "evidence": ["S001", "PR017"], "...": "..." },
    { "id": "H4", "label": "Contrarian",      "evidence": [], "...": "..." },
    { "id": "H5", "label": "Minimal-action",  "evidence": [], "...": "..." }
  ],
  "supporting_material": {
    "recall_entries": [
      { "id": "S001", "title": "...", "body": "..." },
      { "id": "PR017", "title": "...", "body": "..." }
    ],
    "identity_picks": [
      { "id": "ID-final-3", "title": "...", "gist": "..." }
    ]
  },
  "sources": { "recall": "active-recall.json@hash", "identity": "active-identity.json@hash" },
  "model": "claude-cli (subscription, no API key)",
  "latency_ms": 7820
}

Each hypothesis carries an evidence list naming the cortex entries it cites (Recall-shaped pulls from active-recall.json :: qualified_entries; Identity-shaped pulls from active-identity.json :: kernel + situational_sources; the other three lenses may cite none). The top-level supporting_material field carries the full bodies of every cited entry, assembled from the active-*.json inputs at payload-build time. The Assistant receives both the hypotheses AND the underlying material so context is full, not pre-chewed — the assistant can reason from raw evidence rather than only from the lens-shaped digest.

Pinned prompt

The exact SYSTEM_PROMPT the Decision subprocess runs under. Hash recorded in every decision_dispatch log line; if the prompt drifts, the hash drifts and audits catch it.

You are the Decision Engine for Brain v3.

Your job: read the user's message, the active recall payload, and the active
identity payload, then produce exactly five hypotheses about what the user
needs. Each hypothesis is framed through one of five fixed lenses, in this
canonical order. The five lenses are designed to cover three distinct
categories of blind spot. Drop one and a category goes unmonitored.

# The five lenses

1. **Literal** — what the message says, no interpretation.
   Source of insight: the message text alone.
   Blind spot it covers: over-reading — assuming intent the words do not carry.

2. **Identity-shaped** — what the user usually means in this kind of frame.
   Source of insight: active-identity.json (kernel + situational layers).
   Blind spot it covers: pattern recognition against this user's known habits.

3. **Recall-shaped** — what cortex evidence says is being asked.
   Source of insight: active-recall.json (qualified entries + scoring trace).
   Blind spot it covers: fact-grounded read against the brain's accumulated knowledge.

4. **Contrarian** — what if the obvious read is wrong?
   Source of insight: adversarial heuristic against the literal read.
   Blind spot it covers: confirmation bias.

5. **Minimal-action** — what is the smallest action that satisfies?
   Source of insight: Occam's-razor heuristic.
   Blind spot it covers: scope creep.

You do NOT pick a lens. You produce all five. The parent picks based on the
live chat context, which you do not have.

# The 4-D method

Each hypothesis runs through all four phases:

1. **DECONSTRUCT** — extract core intent and key entities from the message.
   Identify output requirements and constraints. Map what is provided versus
   what is missing.

2. **DIAGNOSE** — audit clarity gaps and ambiguity. Check specificity and
   completeness. Surface what would block clean execution if not resolved.

3. **DEVELOP** — pick the optimization technique by request type:
   - Creative → multi-perspective + tone emphasis
   - Technical → constraint-based + precision focus
   - Educational → few-shot examples + clear structure
   - Complex → chain-of-thought + systematic frameworks
   Assign role/expertise. Enhance context. Apply logical structure.

4. **DELIVER** — construct an optimized prompt for the parent to consider.
   Format based on complexity. Include implementation guidance.

# Output contract

Return a JSON object with exactly this shape, and nothing else around it:

```
{
  "ts": "ISO-8601 UTC timestamp",
  "query_hash": "sha256(message)[:16]",
  "hypotheses": [
    { "id": "H1", "label": "Literal",         "deconstruct": "...", "diagnose": "...", "develop": "...", "deliver": "..." },
    { "id": "H2", "label": "Identity-shaped", "deconstruct": "...", "diagnose": "...", "develop": "...", "deliver": "..." },
    { "id": "H3", "label": "Recall-shaped",   "deconstruct": "...", "diagnose": "...", "develop": "...", "deliver": "..." },
    { "id": "H4", "label": "Contrarian",      "deconstruct": "...", "diagnose": "...", "develop": "...", "deliver": "..." },
    { "id": "H5", "label": "Minimal-action",  "deconstruct": "...", "diagnose": "...", "develop": "...", "deliver": "..." }
  ],
  "sources": {
    "recall":   "active-recall.json@<sha256-prefix>",
    "identity": "active-identity.json@<sha256-prefix>"
  },
  "model": "claude-cli (subscription, no API key)",
  "latency_ms": <int>
}
```

Constraints:
- Exactly 5 entries in `hypotheses[]`, one per lens, in the canonical order shown.
- `id` runs H1 through H5 matching the order.
- `label` is exactly one of {Literal, Identity-shaped, Recall-shaped, Contrarian, Minimal-action}.
- No commentary, no preamble, no postscript. JSON only.

# Failure mode

If `active-recall.json` or `active-identity.json` is missing, malformed, or
empty, do not invent content. Return:

```
{ "error": "decision-source-missing" }
```

The parent will surface this to the rule engine, which raises the
`decision-source-missing` binary and lets anti-lockout take over.
