Overview

The GLC Alerts system is an automated daily monitor that compares today's and yesterday's GL Controlling Power BI exports, checks cost quotas against targets, validates data quality, and posts a structured alert to the #glc-alerts Slack channel every morning at 09:05 EEST.

The system catches:

  • Power BI report refresh failures
  • Coefficient adjustments (Material, Labor, Shipping)
  • Unexpected month-over-month changes in any KPI
  • Cost quota deviations from targets (material, labor, shipping)
  • Zero-value data gaps and daily outliers vs. historical anchors

The alert runs on the Jeff/Picanova AWS account (497892281264), not on goodhealth.lv. Davis Lozda and Justus Kaiser have access to this account.

The Lambda lists all files in this folder, sorts them by name (lexicographic), and picks the last two as "today" and "yesterday." This means file naming convention matters — files must sort chronologically by name.

Each export contains daily-level data with a datekey column and KPI columns (Total Revenue, Material Costs, Labor Costs, Shipping Costs, etc.).

Alert Logic

All checks use a sensitivity threshold of 0.25% (thr = 0.0025).

Check 1: Refresh Failure

Compares the maximum datekey in today's export vs. yesterday's. If they are the same date, the Power BI report did not refresh overnight.

if max_date_today == max_date_yesterday:
    "GL Controlling failed to Refresh!!!"
    Short-circuit: skip all other checks

Check 2: Coefficient Adjustment Detection

For Material, Labor, and Shipping, the Lambda compares two versions of each KPI:

Coeff-adjusted KPIRaw total KPI
Material Costsmaterial_costs_total
Labor Costslabour_costs_total
Shipping Costsshipping_costs_calc

If the coeff-adjusted KPI changed by ≥0.25% but the raw total did not, the alert reports: "Coefficients have been adjusted for [KPI]." This prevents false alarms when only coefficients were updated, not actual cost data.

Check 3: Historical Monthly Comparison

Groups data by month, excluding the current (latest) month. For each historical month, computes (today - yesterday) / yesterday per KPI. Any change ≥0.25% triggers an alert.

KPIs that were flagged as coefficient-adjusted are skipped entirely (both the coeff KPI and its raw counterpart).

KPIs monitored (30+): Total Revenue, Material Costs, Labor Costs, Shipping Costs, Payment Costs, Adword, Bing, Facebook, Tik Tok, material_costs_total, labour_costs_total, shipping_costs_calc, Other Production Costs, Affiliate, Reputation, CRM, Newsletter, Other Marketing, Other Paid Services, Print, Referral, Other Social Networks, TV, Voucher Provisions, Third Party Costs, Marketing & Sales Personnel and Contracted Services (Marketing), Indirect, Other Income, Rent / Leasing, Orders, Quantity.

Check 4: Target Checks (All Months)

Unlike the historical comparison, target checks include all months including current.

4a. Cost Quota Check

For Material, Labor, Shipping: computes actual_cost / Total_Revenue and diffs against the target in the GLC - cost quota sheet. Flags if |diff| ≥ 0.25%.

4b. Absolute Cost Check (Fallback)

When the quota target is missing or zero for a given month/KPI, falls back to the GLC - absolute sheet. Computes (actual - target) / target. This is primarily used for Labor, where absolute amounts are the primary check.

Check 5: Data Quality

5a. Zero-Value Check (Last 90 Days)

Aggregates each KPI by date across all shops. If any KPI sums to exactly zero on any day in the last 90 days, reports: "Data is zero for [KPI] in last 90 days."

5b. Anchor Check (Current Month Only)

For each date in the current month, aggregates KPI values and compares against historical anchors from the anchors sheet:

  • If value ≤ 0.5 x historical MIN: "[KPI] for [date] is more than 2x below the historical MIN"
  • If value ≥ 2.0 x historical MAX: "[KPI] for [date] is more than 2x above the historical MAX"

Anchor KPIs: Total Revenue, Orders, Quantity, Material Costs, Labor Costs, Payment Costs, Shipping Costs, Adword, Bing, Facebook, Third Party Costs.

How to Act on Alerts

Based on Davis Lozda's troubleshooting runbook (Slack canvas, February 2026).

When to Act

ScenarioAction
Deviation 1–4%Post in #glc-alerts asking if this is expected. No deep investigation unless confirmed.
Deviation >4%Investigate immediately.
Entire previous day missingAlways investigate. Start at revenue check.
Multiple metrics brokenPost immediately that you are investigating. Keep channel updated.

Step 1: Check Revenue in dw.sales_fact

SELECT
    created_date,
    SUM(revenue_net_price_eur) AS revenue_net
FROM dw.sales_fact sf
WHERE created_date >= CURRENT_DATE - 7
GROUP BY 1
ORDER BY 1 DESC;

If revenue looks normal, check specific cost metric (Step 2). If revenue is off, isolate per data source (Step 3).

Step 2: Check the Specific Metric

SELECT
    created_date,
    SUM(revenue_net_price_eur)    AS revenue_net,
    SUM(shipping_costs_final)     AS shipping_costs,
    SUM(marketing_costs)          AS marketing_costs,
    SUM(labour_costs_total)       AS labour_costs,
    SUM(material_costs_total)     AS material_costs,
    SUM(payment_costs_final)      AS payment_costs
FROM dw.sales_fact sf
WHERE created_date >= CURRENT_DATE - 7
GROUP BY 1
ORDER BY 1 DESC;

Step 3: Isolate per Data Source

SELECT
    created_date,
    data_source,
    SUM(revenue_net_price_eur) AS revenue_net,
    SUM(shipping_costs_final)  AS shipping_costs,
    SUM(marketing_costs)       AS marketing_costs
FROM dw.sales_fact sf
WHERE created_date >= CURRENT_DATE - 3
GROUP BY 1, 2
ORDER BY 1 DESC, 2;

Valid data sources: Mixpix, Lillestoff, Picturator, MWS, Rewallution PL, Grafomap, PicaAPI.

GLC View Filters

The GLC report view dw.v_glc_facts_2022plus applies these filters — investigation queries should match:

  • testorder = false, isvalidorder = true
  • dim_customers.internalcustomer = false
  • dim_shops.testshop = 0 OR dim_shops.platform = 'PicaAPI'
  • Excluded shops: ORWO / Direct to customer, ORWO / to store / Pallet 1, ORWO / direct to store / DHL, ORWO / to store / Pallet 2

Full Filter Query

SELECT
    sf.created_date,
    sf.data_source,
    SUM(sf.revenue_net_price_eur) AS revenue_net
FROM dw.sales_fact sf
LEFT JOIN dw.dim_customers dc ON dc.customerkey = sf.customerkey
LEFT JOIN dw.dim_shops ds ON ds.shopkey = sf.shopkey
WHERE sf.created_date >= CURRENT_DATE - 3
    AND sf.testorder = false
    AND sf.isvalidorder = true
    AND dc.internalcustomer = false
    AND (ds.testshop::text = '0' OR ds.platform = 'PicaAPI')
    AND sf.shop NOT IN (
        'ORWO / Direct to customer',
        'ORWO / to store / Pallet 1',
        'ORWO / direct to store / DHL',
        'ORWO / to store / Pallet 2'
    )
GROUP BY 1, 2
ORDER BY 1 DESC, 2;

Step 4: Pentaho Debugging

If dw.sales_fact is wrong but underlying tables are fine, the issue is in the last Pentaho step:

  1. Check the Pentaho error log for the relevant job.
  2. Check the source database — is the raw data correct there?
  3. Try running the step manually from the admin account.
  4. For marketing-specific issues, run only the marketing jobs + dw.sales_fact refresh.

Work backwards: dw.sales_fact → transformation step → source table → source database.

Known Issues

IssueDetail
TikTok costs persistently zeroVyral/Rest invoices arrive as PDFs only, not via API. "Data is zero for Tik Tok" is expected until invoices are loaded manually or API integration is built.
Other Paid Services zeroPosition was recently separated from Amazon SPA & SBA costs (Jira BTM-2226). Currently shows zero pending new dw column implementation.
TV costs zeroDependent on invoices from Joey. Zero is expected when no invoice has been received for the period.
Cost quota normal ranges not filledDavis's troubleshooting runbook has placeholder ranges (__% - __%). Should be filled from historical data for easier triage.
Authentication token expiryFebruary 2026: Satya identified an expired authentication token causing alert failures. The Lambda uses Azure AD client credentials — the client secret has an expiry that must be monitored.
File naming assumptionThe Lambda picks the 2 most recent files by lexicographic name sort. If file naming convention changes, the wrong files may be compared.
Anchor thresholds not auto-updatedAnchors in the Excel workbook are from 2024–2025 data. Must be manually updated each year or after significant business changes. Stale anchors produce false alerts.
No env vars for secretsClient ID, client secret, tenant ID, and Slack webhook URL are hardcoded in the Lambda source code. Rotating the Azure AD secret requires a code deployment.
