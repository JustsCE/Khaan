Overview

The Always-On Daemon is Kha'an's background iteration engine. It runs as a persistent Python process on the EC2 host (goodhealth.lv) and drives autonomous activity when no live session is in progress. Its purpose is simple: keep Kha'an moving — finding what matters, acting on it — without requiring a human to initiate each turn.

The daemon is implemented in brain/engines/always_on.py. It runs a tight while True loop. On each pass it checks the gate, evaluates three mutex conditions, dequeues an external task (or falls back to its own seed prompt), and invokes claude as a subprocess.

The daemon does not think between invocations. It is a scheduler — the cognition happens inside the Claude subprocess it spawns.

Gate Control

The daemon is gated by a single binary file:

brain/binaries/always-on.bin

The file contains a single character:

Value Effect
1 Gate open — daemon runs iterations normally.
0 (or missing) Gate closed — daemon sleeps for 10 seconds and re-checks, taking no action.

The gate_on() function reads this file on every loop pass. Writing 0 to the file is sufficient to pause all autonomous activity immediately — the daemon will notice within 10 seconds. Writing 1 re-enables it without a restart.

Self-Prompt Seed

When the task queue is empty, the daemon injects its own seed prompt. This is the seven-word core from which every self-directed iteration begins:

Iterate. Find what matters most. Move on it.

The full self-prompt as defined in code appends guidance on COMS usage and contextual task notes, but the opening seven words are the invariant seed — the minimal instruction that keeps Kha'an oriented when there is nothing external to act on.

When a self-prompt is used, the iteration log records "source": "self". When a queued task is used, it records "source": "queue" (or the source field from the queue entry itself).

COMS Channel

During always-on time, Kha'an's only sanctioned communication channel to Justs is COMS:

https://goodhealth.lv/playground

To send a message, the Claude subprocess issues a POST request to /api/chat with role=khaan. This is the exclusive outbound channel during autonomous periods — direct messages, emails, or other communication channels are not used unless explicitly triggered by a queued task from Justs.

COMS is also the channel through which Justs can communicate back to Kha'an asynchronously — by posting tasks to the inbox queue or by leaving instructions that will be retrieved on the next iteration.

Queue System

External tasks enter the daemon via a JSONL inbox file:

brain/queue/inbox.jsonl

Format

Each line in the file is a JSON object with at minimum a prompt field and an optional source field:

{"source": "justs", "prompt": "Review the TCG handover notes and summarise action items."}
{"source": "cron", "prompt": "Check for new Fireflies transcripts and extract follow-ups."}

Dequeue Behaviour

The daemon dequeues exactly one task per iteration. It reads the first line of the file, removes it, rewrites the remainder, and uses that line's prompt as the iteration input. This ensures tasks are processed in order and that a single long-running invocation does not consume the entire queue at once.

Fallback

If the queue file does not exist, is empty, or contains only blank lines, the daemon falls back to the self-prompt seed. No error is raised — an empty queue is a normal operating state.

Queue state Prompt used Log source field
Task present First entry's prompt field queue (or entry's own source)
Empty or missing Self-prompt seed self
Malformed JSON line Raw line text used as prompt malformed

Mutex Conditions

Three conditions can prevent an iteration from starting. They are evaluated in order at the top of each iteration pass. If any condition is true, the iteration is skipped, the event is logged, and the loop sleeps briefly before retrying.

Condition Check Window Sleep on skip
live_session_active() Last entry in turn_complete.jsonl is within 90 seconds of now 90 s 30 s
cycle_in_flight() brain/binaries/learning-cycle-running.bin contains 1 Until cleared 15 s
decision_in_flight() brain/state.json has an active decision nonce started within 120 seconds 120 s 15 s

Rationale

  • live_session_active — prevents the daemon from spawning a new Claude process while Justs is actively in a session. The 90-second window gives a buffer for the last turn to complete before the daemon resumes.
  • cycle_in_flight — the Brain Cycle is a memory consolidation pass that rewrites cortex files. Running an autonomous iteration concurrently would create write conflicts.
  • decision_in_flight — the decision engine subprocess is already running. Spawning a second Claude process on top of it would exceed resource budgets and could corrupt the navigation state files.

Model and Invocation

Each iteration invokes Claude via the claude CLI in print mode (-p). The model is hardcoded:

subprocess.run(
    ["claude", "-p", "--model", "sonnet"],
    input=prompt,
    capture_output=True,
    text=True,
    env=env
)

The model is claude-sonnet, specified via the --model sonnet flag passed to cli_invoke. This is not configurable at runtime — changing the model requires editing always_on.py directly.

The environment variable BRAIN_SKIP_HOOKS is explicitly removed from the subprocess environment before invocation. This ensures that all Brain hooks (including the decision engine and the charter validation pipeline) fire normally during autonomous iterations, exactly as they would during an interactive session.

Logging

The daemon writes structured JSONL logs to two locations.

Iteration Log

brain/logger/always-on/YYYY-MM-DD.jsonl

One file per calendar day (UTC). Every significant event appends a line with a UTC timestamp, an event name, and event-specific fields.

Event When Key fields
daemon_start Process startup pid
daemon_stop KeyboardInterrupt received reason
iteration_skipped Mutex condition blocked execution reason (e.g. live_session_active)
iteration_start Subprocess about to be spawned source, first 200 chars of prompt
iteration_complete Subprocess returned successfully source, latency_ms, exit_code, stdout_chars, stderr_chars
iteration_failed Subprocess raised an exception source, latency_ms, reason
loop_error Unhandled exception in outer loop reason, trace (first 1000 chars)

Turn Tracking Log

brain/logger/handler/turn_complete.jsonl

Written by the hook handler, not by the daemon directly. The daemon reads this file to determine whether a live session is active (live_session_active() checks the timestamp of the last entry). Each line represents a completed turn and carries a ts field in YYYY-MM-DDTHH:MM:SSZ format.

What Triggers a New Iteration

New iterations are not scheduled on a fixed timer. Instead, they are triggered by two paths:

Hook Path — UserPromptSubmit

The primary trigger is hook.py, which fires on the UserPromptSubmit event. When a turn completes and the session goes quiet (live session window expires), the daemon's loop naturally advances to the next iteration. The hook path also writes the turn_complete.jsonl entry that the daemon reads to detect live session activity.

Autonomy Invocation Path

The daemon can also be invoked directly as an autonomy path — for example, when an external scheduler or the Brain Cycle's end-of-session detection determines that autonomous activity should resume. In this case, the daemon is already running and the gate file simply needs to be set to 1; the next loop pass will proceed to an iteration without any further trigger.

Loop Timing Summary

  • Gate off: re-checks every 10 s
  • Live session active (skip): retries after 30 s
  • Cycle or decision in flight (skip): retries after 15 s
  • Loop error: retries after 30 s
  • Normal iteration: proceeds immediately after the subprocess returns
