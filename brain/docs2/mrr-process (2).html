1. Overview & Timeline

Every month, the MRR (Monthly Revenue Reporting) data is processed by loading a new snapshot of order and revenue data in the database (bi_stage_dev), then refreshing both QlikView and Power BI reports for finance stakeholders.

Deadline: Always complete the data load before the 3rd working day of the month.

Monthly Sequence

  • Previous month closes on the last day of the month — data becomes available
  • Step 0: Perform data quality checks (refunds, Groupon, Sendmoments)
  • Step 1: Create new tables & stored procedures in the database
  • Step 2: Call procedures, build MRR snapshot in the database
  • Step 3: Refresh QlikView report on the virtual machine, validate, backup
  • Step 4: Update Power BI views in the database, refresh Power BI report
  • Notify stakeholders (LEN, CHU, JKA, MFR, MNA)

4. Step 0: Data Quality Checks

Before starting the load, verify that the upstream data sources have complete data for the month being loaded. Run these three checks in the database:

i) Refunds

Verify that refund data is current and includes the full previous month:

SELECT refund_date, *
FROM dw.refund_items_fact
ORDER BY refund_date DESC
LIMIT 100;

Check that refund_date values extend through the end of the previous month. If the latest refund dates are more than a few days old, the refund pipeline may be stalled.

ii) Groupon Vouchers

Verify the Groupon voucher staging data is fresh:

SELECT dw_timestamp, *
FROM asa.b2b_grouponvouchers
ORDER BY dw_timestamp DESC
LIMIT 100;

Then check the aggregated Groupon revenue by month and source to spot any gaps:

SELECT
    source,
    TO_CHAR(redeemed_date, 'YYYY-MM') AS year_month,
    SUM(netreveur) AS total_netreveur
FROM dw.groupon_vouchers_fact
GROUP BY TO_CHAR(redeemed_date, 'YYYY-MM'), source
ORDER BY year_month DESC;

The previous month should appear with reasonable revenue figures. Compare against prior months for sanity.

iii) Sendmoments

Verify Sendmoments revenue data is loaded for the target month:

SELECT *
FROM poc_landing.sendmoments_revenue
ORDER BY created_date DESC
LIMIT 100;

Confirm that created_date values cover the full previous month.

5. Step 1: Create Tables & Procedures (Database)

Each month, new versioned tables and stored procedures are created in the database. They are copies of the previous month's objects with updated date references.

i) Copy Previous DDL

Open the DDL of the previous month's stored procedures and tables. Use them as templates for the new month.

ii) Create New Stored Procedures

Two procedures are created each month:

ProcedureNaming ConventionExample
Monthly Financedw.sp_monthlyfinance[YYYYMMDD]dw.sp_monthlyfinance20260305
Monthly Finance All Salesdw.sp_monthlyfinanceallsales[YYYYMMDD]dw.sp_monthlyfinanceallsales20260305

The [YYYYMMDD] date suffix is the date when the procedure is created (not the reporting month).

Three places to update inside each procedure:
  1. The CREATE PROCEDURE statement name
  2. The TRUNCATE TABLE target table name
  3. The INSERT INTO target table name
All three must reference the same new date suffix. Missing one will cause the procedure to write to last month's table.

iii) Create New Tables

Three tables are created each month:

6. Step 2: Call Procedures & Build the Snapshot (Database)

This is the core data processing step. It runs the two finance procedures and then builds the MRR snapshot by combining historical data with new data.

i) Call the Monthly Finance Procedure

CALL dw.sp_monthlyfinance20260327();

This populates dw.monthly_finance_20260327 with the current month's order/revenue data.

ii) Call the Monthly Finance All Sales Procedure

CALL dw.sp_monthlyfinanceallsales20260327();

This populates dw.monthly_finance_allsales20260327 with a broader sales dataset (used by the QlikView “MRR Full” tab).

iii) Build the MRR Snapshot

The snapshot is built by combining the previous snapshot (historical data) with new orders only from the current monthly_finance table. This is a UNION ALL with a LEFT JOIN deduplication pattern.

Truncate the new snapshot table first:

TRUNCATE dw.mrrsnapshot20260331;  -- new SNAPSHOT

Insert the combined data:

INSERT INTO dw.mrrsnapshot20260331  -- new SNAPSHOT
SELECT * FROM (
    -- Part A: All historical data from the PREVIOUS snapshot
    SELECT dw_id, serviceprovider, paymentmethod, ordernumber,
           invoicenumber, createddate, archiveddate, status,
           shipcountry, province, shop, company,
           invoicecompany, invoicevatid, invoiceaddress,
           invoicecity, invoicezipcode, invoicecountry,
           articlenumber, sku, NULL AS productionsite,
           couponcode, coupondescription, coupongroupid,
           shippingprovider, vatrate, vatamount1local,
           vatamount1eur, vatrate1name, vatamount2local,
           vatamount2eur, vatrate2name, currencycode, quantity,
           grosssourcerevlocal, grosssourcereveur,
           netsourcerevlocal, netsourcereveur, datasource,
           grossgrpnreveur, grossgrpnrevlocal,
           netgrpnreveur, netgrpnrevlocal,
           grpncosteur, grpnfixedcosteur, grpnearneur,
           dw_batch, dw_sourcecode, dw_timestamp
    FROM dw.mrrsnapshot20260228  -- old SNAPSHOT
    WHERE CreatedDate > '2019-12-31'

    UNION ALL

    -- Part B: NEW orders only (not in previous snapshot)
    SELECT mfnew.dw_id, mfnew.serviceprovider, ...
    FROM dw.monthly_finance_20260327 mfnew  -- new NORMAL TABLE
    LEFT JOIN dw.mrrsnapshot20260228 mfold   -- old SNAPSHOT
        ON mfold.ordernumber = mfnew.ordernumber
       AND mfold.DataSource  = mfnew.DataSource
    WHERE mfold.ordernumber IS NULL          -- only NEW orders
      AND mfnew.CreatedDate  > '2019-12-31'
      AND mfnew.ArchivedDate >= '2026-03-01' -- first day of load month
) aa
WHERE aa.ArchivedDate > '2019-12-31'
  AND aa.ArchivedDate < '2026-04-01';        -- first day of NEXT month

Deduplication Logic

The LEFT JOIN on ordernumber + DataSource with WHERE mfold.ordernumber IS NULL ensures only orders that do NOT already exist in the previous snapshot are inserted. This prevents double-counting of historical orders. Both columns must match for an order to be considered a duplicate — the same ordernumber can appear from different data sources (e.g., Groupon vs. direct).

Refresh the Power BI Report

  1. Open the MRR Power BI report.
  2. Refresh all data connections.
  3. Wait for refresh to complete.
  4. Save the report (Ctrl+S).

Data Reconciliation (Power BI)

After refresh, verify data integrity in Power BI the same way as QlikView:

  • All historical monthly amounts must remain unchanged.
  • Only the newly loaded month should appear as an addition.
  • Cross-reference against the QlikView export for parity.

Backup

  1. Copy the MRR Power BI file to the archive folder.
  2. Rename the archived copy with the current date.

10. Snapshot Logic — Deep Dive

Understanding the snapshot build pattern is critical for debugging issues.

How Snapshots Accumulate

Each month's snapshot is a superset of all previous snapshots plus new orders. The chain looks like:

mrrsnapshot20260131 = all orders through January
mrrsnapshot20260228 = mrrsnapshot20260131 + new Feb orders
mrrsnapshot20260331 = mrrsnapshot20260228 + new Mar orders
...

The snapshot is append-only by design. Once an order enters a snapshot, it stays in all future snapshots. The deduplication LEFT JOIN ensures no double-counting.

Date Filters Explained

FilterPurpose
CreatedDate > '2019-12-31'Excludes legacy data before 2020 (applies to both historical and new parts)
ArchivedDate >= '2026-03-01'Only include new orders archived in the target month (Part B only)
ArchivedDate < '2026-04-01'Exclude orders archived in future months (outer WHERE, applies to all)

Archive Date Filter Change

JEN and CHU requested including all created orders, not just archived ones. The archive date filter in the outer WHERE clause may be adjusted or removed in future iterations. If you are instructed to remove this filter, ensure consistency — remove it from both the main snapshot query AND the Sendmoments insert.

11. Sendmoments Data Integration

Sendmoments revenue data is inserted into the snapshot separately because it comes from a different source (poc_landing.sendmoments_revenue) and has a different column structure. This is the last step of the snapshot build — the Sendmoments data is unioned into the snapshot by inserting it at the end of Step 2.

Insert Query

INSERT INTO dw.mrrsnapshot20260331 (  -- new SNAPSHOT
    dw_id, serviceprovider, paymentmethod, ordernumber,
    invoicenumber, createddate, archiveddate, status,
    shipcountry, province, shop, company,
    invoicecompany, invoicevatid, invoiceaddress,
    invoicecity, invoicezipcode, invoicecountry,
    articlenumber, sku, productionsite,
    couponcode, coupondescription, coupongroupid,
    shippingprovider, vatrate, vatamount1local,
    vatamount1eur, vatrate1name, vatamount2local,
    vatamount2eur, vatrate2name, currencycode, quantity,
    grosssourcerevlocal, grosssourcereveur,
    netsourcerevlocal, netsourcereveur, datasource,
    grossgrpnreveur, grossgrpnrevlocal,
    netgrpnreveur, netgrpnrevlocal,
    grpncosteur, grpnfixedcosteur, grpnearneur,
    dw_batch, dw_sourcecode, dw_timestamp
)
SELECT
    sm.id, sm.service_provider, sm.payment_method,
    sm.order_number, sm.invoice_number,
    sm.created_date, sm.archived_date, sm.status,
    sm.ship_country, sm.province, sm.shop, sm.company,
    sm.invoice_company, sm.invoice_vat_id,
    sm.invoice_address, sm.invoice_city,
    sm.invoice_zip_code, sm.invoice_country,
    sm.article_number, sm.sku, sm.production_site,
    sm.coupon_code, sm.coupon_description,
    sm.coupon_group_id, sm.shipping_provider,
    sm.vat_rate, sm.vat_amount,
    NULL, NULL, NULL, NULL, NULL,  -- vatamount1eur..vatrate2name
    sm.currency_code, sm.quantity,
    sm.gross_rev_local, sm.gross_rev_eur,
    sm.net_rev_local, sm.net_rev_eur,
    sm.datasource,
    NULL, NULL, NULL, NULL,        -- grpn columns
    NULL, NULL, NULL,              -- grpn cost columns
    NULL, NULL, NULL               -- dw_batch, dw_sourcecode, dw_timestamp
FROM poc_landing.sendmoments_revenue sm
WHERE sm.archived_date >= '2026-03-01'
  AND sm.archived_date <= '2026-04-01';

Column Mapping Notes

The Sendmoments source table uses snake_case column names while the snapshot uses camelCase/lowercase. The column mapping in the INSERT handles this translation. Key differences:

Sendmoments ColumnSnapshot ColumnNotes
sm.iddw_id—
sm.vat_amountvatamount1localOnly local VAT; EUR and rate2 fields are NULL
(not available)grossgrpnreveur etc.All Groupon-specific columns are NULL for Sendmoments
(not available)dw_batch, dw_sourcecode, dw_timestampAll NULL — Sendmoments has no DW metadata

Date Filters

The Sendmoments insert uses the same date range as the main snapshot: archived_date >= first day of month AND archived_date <= first day of next month.

Boundary condition: Note the use of <= (less-than-or-equal) for the upper date bound in the Sendmoments insert vs. < (less-than) in the main snapshot query. This means orders archived exactly on the first day of the next month will be included in Sendmoments but excluded from the main snapshot. Verify this is intentional for your month’s load.
