Rules

How recall works, grouped by concern. Each rule has a title; mechanics are detailed in the sections below.

Recall fundamentals

  • Reads five cortex regions. cortex/semantic/L<n>/, cortex/episodic/L<n>/, cortex/procedural/L<n>/, cortex/fusiform/L<n>/, cortex/prefrontal/L<n>/. Identity, broca, amygdala are out of scope.
  • One floor. Score floor = 2.0. Below the floor, an entry has only weak partial signals and is dropped.
  • No top-N truncation. Every entry above the floor advances. Width is enforced by floor + cap, never by quota.

Stage flow (5 stateless stages)

  • Normalize. Regex-extract excluded_domains, time_hint, compare_targets, intent_targets, msg_words, msg_stems from the query.
  • Match domains. Pass A substring-matches domain names; Pass B keyword-matches the rest. Bundle detection runs in two tiers (registered vs. keyword-triggered).
  • Expand scope when domains are thin. If matched domains together hold < 50 entries, expand scan to all domains.
  • Score with 7 signals. domain_bonus (+3.0), type_bonus (+1.5), level_bonus (L1 +0.5 / L2 +1.5 / L3 +3.0), lex_bonus (cap 4.0), co_ref_bonus (+2.0), bundle_bonus (+1.5), reuse_bonus (cap 2.0; −0.5 if last access > 7d). Heuristic weights.
  • Lex rescue. Entries with lex_score ≥ 2.0 outside scan_domains are added with score = min(lex_score, 4.0) + level_bonus.
  • Empty short-circuit. If candidate count is zero, route to the empty-recall endpoint. No rerank, no injection.
  • Rerank with 3 components. specificity (cap 2.0), diversity (−1.0 per same-type duplicate), temporal (asymmetric: −1.5 stale, +1.0 verified ≤ 5 cycles, +0.5 ≤ 20 cycles, date fallback when cycle metadata is absent).
  • Graph expansion. For every qualified entry, follow related_to and add neighbours at 50% of parent rerank score, subject to the 2.0 floor.
  • Hand off in full. Every entry above the 2.0 floor is written to navigation/active-recall.json as a structured payload for the Decision Engine to consume.

Philosophy

The Recall Engine surfaces context. It does not answer queries.

Width over selectivity — every cortex entry that could plausibly inform the next move enters the qualified list.

Recall is the find step. Pruning, hypothesis-construction, and final decision happen in the Decision Engine.

Recall surfaces. The Decision Engine reads the qualified set and chooses what reaches Brain. The Recall Engine never writes to additionalContext directly.

Stage 1 — NORMALIZE

Input: raw query string. Operation: regex-extract structural cues from the query.

  • excluded_domains — phrases like “not the X / exclude X / ignore X / skip X / without X” collected into a set. Purpose: a query like “everything about Slack but not the channel topology” tells us to pull the slack domain in but exclude the channel-topology entries.
  • time_hint — “yesterday / last night” → recent_1d; “last week / this week” → recent_7d; “last month / this month” → recent_30d. Purpose: “what did we ship last week” should weight recent entries higher in rerank’s temporal axis.
  • compare_targets — “X vs Y / X versus Y / X or Y / X compared to Y / X against Y” forces both X and Y into scan domains. Purpose: “Slack vs WhatsApp” needs both communication domains in the candidate pool, even if one side hits no domain keyword.
  • intent_targets — verbs like “deploy X / build X / fix X / debug X / connect to X” collected as nouns. Purpose: “deploy lambda” routes to the deploy bundle even if no domain keyword matches.
  • msg_words = [a-z0-9]+ tokens from the lowercased query. msg_stems = stemmed variants of words longer than 3 chars. Purpose: a query with “scoring” finds entries containing “scored” or “scores”.

Stems are stored alongside direct terms in the inverted index; lex_bonus picks them up transparently with no extra lookup. A small protected-acronym set bypasses stemming so tokens like SQL, AWS, S3 keep their full IDF weight.

Output: (excluded, time_hint, compare_targets, intent_targets, msg_words, msg_stems).

Stage 2 — DOMAIN MATCH

Input: query + excluded set + the domain dictionary. Operation: two-pass match.

  • Pass A: substring match the domain name itself (e.g., "vericity" in msg).
  • Pass B: for each unmatched domain, check its keyword list (e.g., the deployment domain matches on "docker", "deploy", "acr", "container app", "ec2", "nginx", …).
  • Bundle detection. A bundle is a hand-curated preload list for a known recurring workflow — when its trigger fires, every entry in the preload list joins the qualified set with a +1.5 bundle_bonus, so the engine doesn’t rediscover relevance from scratch each time the workflow appears. Bundles live at brain/bundles/<name>.json with a name, a trigger phrase, a keyword list, and a manifest of cortex IDs to surface together. Detection runs in two tiers: tier-1 = bundles registered in matched domains’ bundles field; tier-2 fires only if tier-1 is empty: keyword-match against the bundle’s keywords + trigger, requiring ≥ 2 matches OR (≥ 1 match AND ≥ 1 intent hit).
  • Scope decision: if matched domains hold < 50 total entries, expand to all domains. Narrow domains starve coverage; expanding still preserves the domain_bonus signal for ranking.

Output: (matched_domains, active_bundles, bundle_entry_ids, scan_domains).

Stage 3 — SCORE

Input: scan_domains, matched_domains, lexical_hits, entity_linked, bundle_entry_ids, access_log, eval_penalties. Operation: walk every entry in the scan domains. For each, compute the 7 signals (table below). Sum to a base score. Keep entries with score > 0.

The 7 signals

SignalFormulaCapType
domain_bonus+3.0 if entry’s domain is in matched_domains3.0binary
type_bonus+1.5 if entry’s memory type matches query_type1.5binary
level_bonus+0.5 if entry is at L1, +1.5 at L2, +3.0 at L3 (folder = level)3.0tiered
lex_bonusmin(IDF-weighted hit count, 4.0)4.0continuous
co_ref_bonus+2.0 if entry id appears in entity.linked2.0binary
bundle_bonus+1.5 if entry is in an active bundle’s preload1.5binary
reuse_bonusmin(voluntary_access * 0.5, 2.0) minus 0.5 if last access > 7d ago2.0 / -0.5continuous

All weights are heuristic.

Lexical rescue path. The inverted index (navigation/inverted-index.json) is a posting-list map term→[entry IDs] plus IDF weights computed via log(N / posting_list_len) + 1.0. After the main scan, the rescue looks up entries with lex_score ≥ 2.0 that fell outside the scan domains and adds them with score = min(lex_score, 4.0) + level_bonus. The 2.0 floor requires at least two distinct rare-term hits; the 4.0 cap matches the main lex_bonus cap so a routing-failed entry can’t outrank a routing-perfect entry purely on word overlap.

The inverted index is rebuilt by the brain cycle, so the rescue path always reads a current index. Rescue thresholds are static; the index is the moving part.

Output: candidates list [{id, path, gist, score, read=False}, ...] sorted by score descending. No top-N truncation — every entry above the score floor advances.

Stage 4 — RERANK

Input: the full candidate list from Stage 3 + entry_meta + access_log. entry_meta is the recall-index.json records for the scanned domains, scoped per-query.

Operation. When candidate_count == 0 the dispatcher routes to the empty-recall endpoint (Recall.dispatch_empty()) immediately — no rerank, no handoff, just a marker so the Decision Engine knows there is no recall context this turn. Otherwise every candidate is reranked with three components added to the base score:

ComponentFormulaCapPurpose
specificitymin(len(query_words ∩ gist_words) * 0.5, 2.0)2.0Reward entries whose gist literally overlaps the query, not just domain-tagged. Capped so a verbose query can’t let a single entry’s overlap dominate.
diversity0.0 first time a memory type is seen, -1.0 for each subsequent same-type-1.0 per dupPrevent five procedurals from monopolising the result set when a mix of memory types would be more useful.
temporalsee breakdown below-1.5 ... +1.0Answer “is this knowledge still true?” — prefer current truth over decayed knowledge.

Temporal breakdown. Asymmetric — the world doesn’t suddenly become true (positive bonuses small) but it can suddenly become wrong (penalties larger).

  • -1.5 — stale: cycleCount - verified_at_cycle > 50.
  • +1.0 — verified within the last 5 cycles. Just confirmed.
  • +0.5 — verified within the last 20 cycles. Recently relevant.
  • Date fallback when cycle metadata is absent: +1.0 if learned ≤ 7 days ago, +0.5 if ≤ 30 days ago, -0.5 if > 30 days ago.

1-hop graph expansion. Sort by rerank_score descending. For every qualified entry, follow related_to links and add neighbours to the qualified list at 50% of the parent’s rerank_score (subject to the 2.0 floor). No cap on seeds, no cap on neighbours per seed — Recall widens, the Decision Engine prunes. The 50% discount keeps direct hits ranked above their neighbours without removing the neighbours from consideration.

Output: reranked qualified list, single tier.

Stage 5 — HANDOFF

Input: reranked candidates + min_req_score = 2.0 floor.

Operation: filter to rerank_score ≥ 2.0. Every entry that clears the floor is qualified. Below 2.0 an entry has only weak partial signals — a single binary match alone (domain or type) does not clear the floor; an entry needs at least domain-plus-something, or strong lexical evidence, or a combination. The 2.0 cut keeps the qualified set grounded in real evidence rather than incidental keyword overlap.

Handoff format. Every qualified entry is written in full — rules, procedurals, semantic, episodic, all of it — into navigation/active-recall.json as a structured payload (id, path, gist, full body, score breakdown, links). The Decision Engine reads this file. Recall never writes to additionalContext and never speaks to Brain directly.

Output: a RecallResult payload containing qualified_entries (full bodies + metadata) and scoring_trace (signal breakdown for each entry). Returned to the dispatcher and persisted to active-recall.json. The Decision Engine consumes either form.

Infrastructure

How recall plugs into the rest of the brain. Grouped by which engine drives or consumes each piece.

Driven by Rule Engine — Recall.dispatch(message) → RecallResult | None

The dispatcher calls this exactly once per UserPromptSubmit.

  • Args: raw user message string.
  • Returns: a RecallResult payload with qualified_entries (full bodies + metadata) and scoring_trace, or None if no candidate qualified (empty-recall route).
  • Side effects: writes navigation/active-recall.json; updates the access log; writes modes/bundle_mode.json when a bundle activates.

Read by Recall, written by Learning Engine

The Brain Cycle (Learning Engine, Phase 4) rebuilds navigation/inverted-index.json and navigation/recall-index.json from the cortex on disk every cycle. Recall reads these files; it does not write them. See learning-engine.html for the rebuild contract.

Consumed by Decision Engine — active-recall.json

Recall’s only piece of cross-event state. Written at the end of Stage 5 (HANDOFF) and consumed by the Decision Engine. Lives until either:

  1. Overwrite by the next UserPromptSubmit’s dispatch.
  2. 2-hour expiry from the timestamp.

Writes are atomic via tmp-then-rename, so a concurrent read sees either the old payload or the new payload, never a half-written one.

Read by Rule Engine — binaries

Four alerts the Rule Engine reads. Same file shape as every other binary in the system: 0 silent, 1 alert, atomic writes.

BinaryTrigger to raise (1)Opener (writes 0)
recall-staleactive-recall.json older than 2 hoursnext successful dispatch
recall-dispatch-failedlast dispatch threw or returned an errornext successful dispatch
recall-failed3rd consecutive dispatch failurenext UserPromptSubmit
recall-timeoutdispatch exceeds 30snext UserPromptSubmit

Recall raises and clears its own binaries; the Rule Engine just reads them.

Anti-lockout

Three protections, all simple. Worst case the parent answers without fresh recall (degraded), never locked out.

ProtectionWhat it does
Three-strike forgivenessAfter 3 consecutive dispatch failures, recall-failed.bin = 1. Next UserPromptSubmit clears every recall binary; the parent runs without a fresh qualified set that turn (degraded but unblocked).
30-second timeoutPer dispatch attempt. If exceeded, recall-timeout.bin = 1. Next UserPromptSubmit clears.
SessionStart resetAll recall binaries written to 0 at session start.

Every fallback writes a log line so the failure mode is always visible.

Logs

Every recall event logs to logger/recall/. The writer is logger/api.py::write_recall_log(event, payload); the recall engine doesn’t manage the file, it just calls the writer. Append-only.

Each event is enumerated below with the exact payload it carries and the question it lets us answer.

EventPayloadWhat it lets us measure
dispatchts, query_hash, scan_domains, candidate_count, qualified_count, qualified_ids, route (normal | empty)Qualified-set size and composition between scoring variants. The primary surface for recall comparisons.
lex_rescuets, query_hash, rescued_ids, lex_scoresHow often the rescue path fires and whether tuning its floor changes precision.
bundle_activationts, query_hash, tier (1 | 2), bundles_active, bundle_entry_idsWhich bundles ever fire. Dormant ones get pruned; load-bearing ones earn maintenance attention.
graph_expansionts, query_hash, seed_ids, neighbour_ids, parent_scoresWhether 1-hop expansion adds value the Decision Engine actually consumes.
handoffts, query_hash, qualified_ids, total_charsPayload size handed to the Decision Engine. The cost side of the recall ledger.
index_rebuildts, terms_count, entries_count, phrases_count, orphans_removedThat the index changed across runs — a sanity check before reading any other diff.

All events share query_hash (sha256 of the lowercased message), so a single query’s full trajectory — dispatch → lex_rescue → bundle_activation → graph_expansion → handoff — can be reconstructed by joining on that key.
