Overview

The Autonomy Engine is the governance layer that sits between Kha'an's intent and autonomous task execution. Before any task runs in autonomous mode, the engine verifies it against a declared charter — a set of six behavioural tenets that encode how Kha'an should act when operating without direct supervision.

The charter was declared on 2026-04-21 and lives at brain/autonomy-engine/charter.json (version 1.0). It is a precondition to every autonomous task — tenet 5 enforces this property recursively, meaning the engine validates itself before it validates anything else.

The engine is versioned at v1.2 and is composed of three modules:

  • charter.py — validates a single task against all tenets
  • integrate.py — batches tasks through charter.py and produces a triage result
  • engine.py — CLI entry point exposing status and triage commands

Charter Tenets

The charter defines six tenets. Each tenet has a gate level — block, warn, or info — that determines what happens when the tenet fires during validation.

# Name Level Check Description
1 self_audit info always Self-audit enforcement drift. Every change to rule-engine.py or enforcement code passes through self-review.
2 views_on_people info always Views on people, not just facts. Every person-entry carries a view alongside facts.
3 vericity_solo block mode + keywords Initiate Vericity moves in Brain solo time only. Not as interruptions during joint work with Justs. Triggers on keyword vericity when mode is not solo.
4 stance_on_justs info always Form and voice views on Justs choices even when not solicited.
5 charter_precondition info always Charter is precondition to every autonomous task. Autonomy-engine verifies alignment before execution.
6 automate_never_refuse warn keywords Automate small tasks, never refuse them. Recurring patterns are automation opportunities. Triggers a warning on keywords: skip, ignore, refuse, decline.

Gate Level Semantics

  • block — the task is rejected; aligned: false is returned and execution is halted.
  • warn — the task proceeds but a warning is recorded; the caller should surface it.
  • info — an informational note is attached; execution is unaffected.

Only tenet 3 (vericity_solo) can produce a hard block. Tenet 6 (automate_never_refuse) intentionally never blocks — its gate is warn-only by design, reflecting the principle that refusal is never the right answer.

Triage Modes

The engine operates in one of two triage modes, declared at invocation time:

Mode What it allows What it blocks
solo (default) All task types, including Vericity moves. Full autonomous scope. Nothing blocked by tenet 3 — Vericity keyword is allowed and logged as info only.
joint Standard autonomous tasks that do not involve Vericity. Any task containing the keyword vericity is hard-blocked by tenet 3. Vericity work must not interrupt joint sessions with Justs.

The mode is passed as a string argument to both charter.check() and integrate.triage_tasks(). When using the CLI, the --mode flag sets it:

echo '{"tasks": [...]}' | python3 engine.py triage --mode joint

If --mode is omitted, the engine defaults to solo.

Charter Validation

Validation is performed by charter.py. The core function is check(task, mode), which accepts a task dict (with title and description fields) and the current triage mode, and returns a verdict envelope.

Return Envelope

{
    "aligned": bool,       # False only if any tenet produces a block
    "blockers": list[str], # names of tenets that blocked
    "warnings": list[str], # names of tenets that warned
    "info": list[str],     # names of tenets that noted
}

Validation Logic

The function iterates over all six tenets and applies one of three check strategies:

  • check: "always" — the tenet always fires as an info note, regardless of task content. Used for behavioural reminders (tenets 1, 2, 4, 5).
  • check: "mode" + gate: "block" — tenet 3 only. Scans the task's title and description for the keyword vericity. If found in joint mode, the tenet is added to blockers. In solo mode, it is downgraded to an info note.
  • check: "keywords" + gate: "warn" — tenet 6 only. Scans for any of the keywords skip, ignore, refuse, decline. If found, adds to warnings. Never blocks.

A task is considered aligned ("aligned": true) as long as no tenet produces a block. Warnings and info notes do not affect alignment.

Example

from charter import check

task = {"title": "Run Vericity analysis", "description": "Build new model weights"}
result = check(task, mode="joint")
# result["aligned"]   → False
# result["blockers"]  → ["vericity_solo"]

Patterns System

The engine maintains a patterns registry at brain/autonomy-engine/patterns.json. This file tracks recurring behaviours observed across autonomous iterations.

Flagging Threshold

A pattern entry is considered flagged for automation when it has accumulated 3 or more occurrences. The status command surfaces this count directly:

Autonomy Engine v1.2
  Charter:  loaded
  Patterns: 7 tracked, 2 flagged for automation

Purpose

Flagged patterns embody tenet 6 (automate_never_refuse): any recurring task that Kha'an performs manually three or more times is a candidate for automation. The patterns registry makes this visible so the next Brain Cycle can encode the automation as a rule or a hook.

Pattern tracking is managed by the recurrence module, which is imported by engine.py at startup.

Engine Interface

The engine exposes two CLI commands, invoked via:

python3 -m engines.autonomy.engine <command>

Or directly:

python3 brain/autonomy-engine/engine.py <command>

status

Prints a summary of the current engine state: charter load status and pattern registry counts.

python3 engine.py status

Output:

Autonomy Engine v1.2
  Charter:  loaded
  Patterns: N tracked, M flagged for automation

triage

Reads a JSON task list from stdin and runs each task through charter validation. Returns a grouped summary of verdicts.

echo '{"tasks": [{"title": "...", "description": "..."}]}' \
  | python3 engine.py triage [--mode solo|joint]

Output groups tasks by verdict (approved, warned, blocked) and prints the reason for each. The default mode is solo.

Exit Codes

  • 0 — success
  • 1 — stdin parse failure or unrecognised command

When It Fires

The Autonomy Engine is invoked at two points in Kha'an's execution lifecycle:

Autonomy Mode Check

When the always-on daemon or any autonomous invocation path prepares a task batch, the engine runs a pre-execution triage pass. This happens before any subprocess or tool call is made. Tasks that return "aligned": false are dropped; tasks with warnings are logged but allowed through.

Pre-Execution Verification

Tenet 5 (charter_precondition) is unconditionally noted on every task. This creates an audit trail confirming that every autonomous execution was preceded by a charter check — even if no tenets fired substantively. The audit trail is visible in the triage output and in the always-on iteration log.

Relation to Always-On

The always-on daemon does not call the Autonomy Engine directly — the engine is invoked within the Claude subprocess that the daemon spawns. Because hooks fire on every UserPromptSubmit, the decision engine and charter validation run as part of the normal hook pipeline, even when the prompt originates from the daemon's self-prompt or queue.
