Philosophy

Skills are the five operational verbs of the Brain: consolidate memory, model intent, capture lessons, ingest sources, encode corrections.

Brain Cycle and Decision Engine run automatically. Brain Retro, Brain Learn, and Brain Correct write to hippocampus.md or directly to prefrontal — whichever path urgency demands.

The five skills divide into two tiers: infrastructure skills that run on every turn without user invocation (Brain Cycle, Decision Engine), and learning skills that are triggered explicitly or at session end (Brain Retro, Brain Learn, Brain Correct). The learning skills feed hippocampus; Brain Cycle consolidates it. None of the learning skills write directly to L2/L3/L4 — only the cycle promotes entries between levels.

Execution model. All five skills run via cli_invoke() or directly via claude -p as a Bash subprocess — not as Anthropic native subagents. The SubagentStop hook is a no-op for all five skills (see hook.py::handle_subagent_stop). Schema verification happens inline inside each skill’s run() function, not via the hook layer.

Brain Cycle — 8-phase consolidation loop

Trigger: auto-fires when learning-cycle-overdue.bin = 1 (message counter gap ≥5 since last cycle). Also explicitly invokable via /brain-cycle for on-demand consolidation.

Blocking behaviour. From the moment learning-cycle-running.bin = 1 until the cycle commits, the dispatcher blocks all other tools. The only exception is the Skill:brain-cycle call itself (to prevent lockout). A running cycle holds state.json :: fsm = CYCLE_RUNNING and a unique nonce; no other process may write cortex during this window.

Gate clearing. On every exit path (success, timeout, 3-strike failure) the cycle unconditionally clears: learning-cycle-overdue, learning-cycle-running, learning-cycle-failed, learning-cycle-timeout, cycle-empty. Outcome is recorded in state.json :: last_cycle_outcome.

The 8 phases

  1. Ingest — scan hippocampus.md and any staged sources; classify observations by cortex region and category.
  2. Hippocampus — write classified observations as candidate entries; stage them for promotion evaluation.
  3. Consolidate — compute overlap coefficients; reinforce duplicates (strength +1), promote entries meeting the threshold, discard entries below the floor. Reports: N reinforced / N promoted / N discarded.
  4. Index rebuild — rebuild the full-text search index over all cortex entries. Reports: total terms / total entries.
  5. Synthesis — create L2 bundles from dense L1 clusters; create L3 summaries from mature L2 groups. Archive consumed source entries. Reports: L2 created / L3 created / sources archived.
  6. Eval (conditional, every eval_window cycles; default 30) — drift detection across cortex regions.
  7. Hygiene (conditional, same window) — prune stale, low-strength entries that have not been reinforced.
  8. Commit — bump thalamus.json :: cycleCount, write the ledger, git-commit the brain repo. Reports: cycle N.

Retry and timeout

Up to 3 attempts on transient failure. Timeout: 900 seconds (15 minutes). On 3-strike failure: state.json :: fsm = MAINTENANCE_FAILED. On timeout: last_cycle_outcome = timeout, gates cleared, returns None.

Manual invocation

python3 -c "import sys; sys.path.insert(0, BRAIN_PATH); \
from engines.brain_cycle import run_cycle; import json; \
r = run_cycle(); print(json.dumps(r, indent=2) if r else 'cycle returned None')"

Reads: hippocampus.md, all cortex entries.
Writes: cortex entries (promote/discard), search index, thalamus.json, git commit on brain repo.
Budget: 900 seconds, 3 retry attempts. Runs synchronously inside the UserPromptSubmit hook when overdue.

Decision Engine — per-turn hypothesis engine

Trigger: auto-fires on every UserPromptSubmit via hook.py::handle_user_prompt_submit. Manual invocation supported for debugging.

What it does: composes 5 hypotheses about user intent through fixed lenses (Literal, Identity-shaped, Recall-shaped, Contrarian, Minimal-action). Each hypothesis has four fields: deconstruct (what the message literally says), diagnose (what the real gap is), develop (agent role and technique), deliver (the concrete action). The parent session reads all five and picks based on live context the subprocess lacks.

Pipeline (per turn)

  1. Mutex acquire — locks state.json :: active_decision_nonce; 240s wait, 250ms poll; exits clean if exceeded.
  2. Recall + Identity Relay — two Python threads run in parallel. Recall writes navigation/active-recall.json (top cortex entries for this turn). Identity Relay writes navigation/active-identity.json (situational identity layer). Both threads join before the next step.
  3. Decision subprocess — cli_invoke() with a pinned SYSTEM_PROMPT; returns a JSON envelope with 5 hypothesis objects.
  4. Verify envelope — 5 hypotheses present, lens labels match canonical order, query_hash matches, source hashes match, tool_uses include reads of all cited evidence IDs.
  5. On pass: write navigation/active-decision.json, clear decision-hypothesis-{1..5} bins, release mutex.
    On fail: increment strike counter; retry up to 3 times; after 3 strikes set decision-failed.bin = 1. Anti-lockout clears it on the next UserPromptSubmit.
  6. Mutex release in finally.

Blocking side-effect

During the ~30s decision subprocess window, decision-hypothesis-{1..5} bins are set to 1. Because these are in the dispatcher’s BLOCKING_BINARIES, any tool call from a concurrent session (e.g. the always-on daemon) will be blocked until the decision completes. The _compose_turn_context() call in hook.py clears all five bins after consuming them into the UserPromptSubmit additionalContext.

Manual invocation

python3 -c "import sys; sys.path.insert(0, BRAIN_PATH); \
from engines.decision import dispatch; import json; \
r = dispatch('YOUR_MESSAGE'); print(json.dumps(r, indent=2) if r else 'None')"

Reads: cortex (via Recall thread), identity ledgers (via Identity Relay), user message.
Writes: navigation/active-recall.json, navigation/active-identity.json, navigation/active-decision.json.
Budget: 240 seconds per turn. Spec: docs/decision-engine.html. Orchestrator: decision.py::dispatch().

Brain Retro — session-end retrospective

Trigger: explicit /brain-retro invocation at session end. Not called by Brain Cycle — the cycle is a memory consolidator, not a session analyst.

What it does: sweeps the just-completed session for what worked, what broke, and which corrections recurred. Produces hippocampus entries for the next Brain Cycle to consolidate, and writes prefrontal rules directly when a correction is recurring.

Steps:

  1. Read the session transcript end-to-end (from logger/handler/turn_complete.jsonl).
  2. Identify three categories of finding:
    • Worked — intended outcome achieved with no correction.
    • Broke — user correction received; capture the original, the correction, and the resulting behavior.
    • Recurring — same correction or class of correction received N+ times across the session or recent cycles; flag as a candidate for a prefrontal rule.
  3. For each Broke finding, write an episodic-target hippocampus entry (the lesson, the moment, the correction).
  4. For each Recurring finding, file a candidate R### directly to cortex/prefrontal/L1/ at strength 1. Bypasses the cycle’s promote-scan because correction-driven rules need to land immediately; subsequent reinforcements promote through L2/L3 normally.
  5. For each Worked finding, reinforce the existing procedural entry that drove the success (strength +1) if one exists, or write a procedural-target hippocampus entry if the path was novel. Successes are evidence; ignoring them is a memory leak.

Execution model

Runs via claude -p --output-format json as a Bash subprocess. Not an Anthropic native subagent — SubagentStop does not fire. Schema verification happens inline inside the orchestrator’s run() function: payload must validate against the Brain Retro schema (required fields, ID format, target region valid, source citations present). On schema fail: no entries land, retro_verification_failed log event emitted.

Reads: session transcript (logger/handler/turn_complete.jsonl).
Writes: hippocampus.md (H### entries), cortex/prefrontal/L1/R###.md (direct rule writes for recurring corrections).
Budget: 80 tool calls, 300 seconds.
Ledger: logger/learning/retro-NNNN.json per invocation.

Brain Learn — on-demand learning

Trigger: explicit /brain-learn <source>. The source is an external artifact: PDF, transcript, URL, message thread, or any file. (Reading the current conversation is Brain Retro’s job.)

What it does: extracts observations from the source and stages them in hippocampus for the next Brain Cycle to consolidate. Iterative — presents the batch to the user, accepts surgical corrections, then commits.

Procedure

  1. Read the source. Plan first. If multi-part, segment into chunks. Non-trivial extractions enter plan mode. User approves before observations land.
  2. Extract candidate observations. No fabrications. Classify each by memory category, assign salience, cite source span. Unknowns marked TBD. Surface contradictions; do not pick a side silently.
  3. Match against existing cortex. Compute overlap coefficient (same formula as Brain Cycle Phase 3). Pre-flag duplicates and reinforce-candidates.
  4. Present the batch. Iterate, not one-shot. Show staged observations. Wait for reaction.
  5. Apply pushback as surgical edits. Edit only observations called out. Wholesale rewrites are signal violations.
  6. Verify before commit. Grep for fabricated IDs, ungrounded claims, missing source spans.
  7. Promote, then encode lessons. Write verified observations to hippocampus.md tagged source=brain-learn. If the same correction lands N+ times across batches, file a prefrontal rule via the Brain Retro mechanism.
  8. Honesty over narrative. Low-confidence extractions say so in frontmatter. Do not paper over uncertainty with confident prose.

Execution model

Runs via claude -p --output-format json as a Bash subprocess. SubagentStop does not fire. Verification is inline: every observation must have id, target category, salience, source-span citation. On schema fail: no entries land, learn_verification_failed log event emitted.

Reads: the source; existing cortex (for duplicate detection).
Writes: hippocampus.md (H### entries staged for the next cycle).
Budget: 160 tool calls, 1800 seconds.
Ledger: logger/learning/learn-<source-hash>.json per batch.

Brain Correct — immediate correction encoding

Trigger: explicit /brain-correct <text>. Used when a correction MUST be encoded immediately and cannot wait for /brain-retro at session end.

Why it exists. Brain Retro fires at session end, which can be hours away. Recurring corrections (same mistake N+ times) need to be captured the moment the user types them. /brain-correct is the emergency-encode path.

What it does:

  1. Read the correction text and any arguments from the user’s invocation.
  2. Identify the failure pattern: what did the agent do, what should it have done, what rule would have prevented this.
  3. Write a prefrontal entry directly to cortex/prefrontal/L1/R###.md at strength 1. Frontmatter records shape: brain-correct, source message hash, recurring_count: 1.
  4. If the correction text matches an existing R### entry (overlap coefficient ≥0.5), increment recurring_count and bump strength by 1 instead of creating a duplicate. Repeated invocations reinforce one rule rather than scatter many.
  5. Emit a learn_correct_applied log event with the new or reinforced rule ID.

Execution model

Runs via claude -p --output-format json as a Bash subprocess. SubagentStop does not fire. Verification is inline: proposed R### entry must validate against the prefrontal-rule schema (required fields, ID format, body present, source-message-hash captured). On schema fail: rule does NOT land, correct_verification_failed log event emitted.

Reads: the user’s correction text, optionally recent conversation for context.
Writes: cortex/prefrontal/L1/R###.md (new entry or strength bump on existing).
Budget: 30 tool calls, 60 seconds.
Ledger: logger/learning/correct-NNNN.json per invocation.
