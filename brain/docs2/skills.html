Philosophy

Skills produce inputs; the Learning Engine consolidates them.

Each skill writes either to hippocampus.md (staged for the next Brain Cycle) or directly to a long-term region (when correction urgency demands it).

The Learning Engine’s Brain Cycle runs every 5 messages and consolidates whatever has accumulated in hippocampus into the cortex. The three skills below are the paths by which observations and corrections enter that pipeline. Each is invokable independently; each has a documented trigger, scope, and verifier. None of them write directly to L2/L3/L4 — only the cycle promotes entries between levels.

The three skills

Brain Retro — session-end retrospective

Trigger: explicit /brain-retro invocation. Brain Cycle may also call this skill internally when it detects the session winding down (closing patterns in the latest user message).

What it does: sweeps the just-completed session for what worked, what broke, and which corrections recurred. Produces hippocampus entries for the next Brain Cycle to consolidate, and writes prefrontal rules directly when a correction is recurring.

Steps:

  1. Read the session chat transcript end-to-end.
  2. Identify three categories of finding:
    • Worked — intended outcome achieved with no correction.
    • Broke — user correction received; capture the original, the correction, and the resulting behavior.
    • Recurring — same correction or class of correction received N+ times across the session or across cycles; flag as a candidate for a prefrontal rule.
  3. For each Broke finding, write an episodic-target hippocampus entry (the lesson, the moment, the correction).
  4. For each Recurring finding, file a candidate R1-### directly to prefrontal at L1 (fresh rule). This bypasses Phase 4 (promote scan) because correction-driven rules need to be immediate; waiting for the next cycle would let the same mistake recur. Subsequent reinforcements promote it through L2 / L3 normally.
  5. For each Worked finding, reinforce the existing procedural entry that drove the success (strength +1) if one exists, or write a procedural-target hippocampus entry if the path was novel. The successful path is always reinforced or promoted; never discarded. Successes are evidence; ignoring them is a memory leak.

Runs as a subagent — SubagentStop verifies

Brain Retro runs in a subagent. The parent invokes it with the session transcript. The subagent classifies findings and returns a structured payload (Worked / Broke / Recurring lists, each with target hippocampus or prefrontal entries).

The SubagentStop hook then runs a single verifier check: the returned payload validates against the Brain Retro schema (required fields, ID format, target region valid, source citations present). If schema validation fails, the payload is rejected, no entries land in hippocampus or prefrontal, and a retro_verification_failed log event is emitted. The parent must re-launch the subagent to retry.

Reads: the session transcript.
Writes: hippocampus.md (H### entries), cortex/prefrontal/L1/R###.md (direct rule writes for recurring corrections, at L1).
Budget: 80 tool calls, 300 seconds (matching Brain Cycle).
Ledger: written independently to logger/learning/retro-NNNN.json (does not require a Brain Cycle to run; Brain Retro may complete in a session where the next cycle is many sessions later). The same retro_summary event is also emitted to the running event stream.

Brain Learn — on-demand learning

Trigger: explicit /brain-learn <source>. The source is an external artifact: PDF, transcript, URL, message thread, conversation log, or any other file. (Reading the current conversation is the Brain Retro skill’s job.)

What it does: extracts observations from the source and stages them in hippocampus for the next Brain Cycle to consolidate.

Runs as a subagent — SubagentStop verifies

Brain Learn runs in a subagent. The parent invokes the subagent with the source. The subagent extracts observations and returns a structured payload of staged H### entries.

Subprocess pinned: the Brain Learn subagent runs as claude -p --output-format json via the local CLI (subscription, no API key). The CLI selects the model from your subscription session. Recorded in every learn_batch log entry as "model": "claude-cli".

The SubagentStop hook then runs a single verifier check: the returned payload validates against the Brain Learn schema (every observation has id, target category, salience, source-span citation). If schema validation fails, no entries land in hippocampus.md and a learn_verification_failed log event is emitted. The parent must re-launch the subagent to retry.

Procedure

Each step embeds the pattern that governs it. These patterns are not optional.

  1. Read the source. Plan first. If multi-part, segment into manageable chunks. Non-trivial extractions enter plan mode — the plan lists what is being extracted, from where, with what confidence. The user approves the plan before observations land in hippocampus.
  2. Extract candidate observations. No fabrications. Per chunk, classify each by memory category and assign salience. Every observation cites a span of the source. Unknowns are marked TBD, never invented. If the source contradicts itself, surface the contradiction; do not pick a side silently.
  3. Match against existing cortex. Compute the overlap coefficient (same formula as Brain Cycle Phase 3). Pre-flag duplicates and reinforce-candidates.
  4. Present the batch. Iterate, not one-shot. Show the staged observations to the user. Wait for reaction.
  5. Apply pushback as surgical edits. Edit just the observations the user calls out. Do not rewrite the batch. Wholesale rewrites are signal violations.
  6. Verify before commit. Grep the batch for fabricated IDs, ungrounded claims, missing source spans. Fix before promoting.
  7. Promote, then encode lessons. Write verified observations to hippocampus.md tagged source=brain-learn. The next Brain Cycle consolidates them. If the same correction lands N+ times across batches, file a prefrontal rule via the Brain Retro mechanism so the same mistake stops appearing in future extractions.
  8. Honesty over narrative. If extraction confidence is low, say so in the entry frontmatter. Do not paper over uncertainty with confident prose.

Reads: the source; existing cortex (for duplicate detection).
Writes: hippocampus.md (H### entries staged for the next cycle).
Budget: 160 tool calls, 1800 seconds (up to 30 minutes for deep extractions).
Ledger: retained per-batch as a manifest under logger/learning/learn-<hash>.json (source hash, observations promoted, observations rejected, user-correction count, model).

Brain Correct — immediate correction encoding

Trigger: explicit /brain-correct invocation, with arguments describing the correction. Used when a correction MUST be encoded immediately and cannot wait for /brain-retro at session end.

Why it exists. Brain Retro fires at session end, which can be hours away. Some corrections — especially recurring ones (the same mistake repeated three or more times) — need to be captured the moment the user types them. /brain-correct is the emergency-encode path. Without it, the same failure pattern can repeat several more times before Brain Retro gets a chance to lift it into a rule.

What it does:

  1. Read the correction text and any arguments from the user’s invocation.
  2. Identify the failure pattern: what did the agent do, what should it have done, what is the rule that would have prevented this.
  3. Write a prefrontal entry directly to cortex/prefrontal/L1/R###.md at strength 1 (fresh rule). Frontmatter records shape: brain-correct, the source message hash, and recurring_count: 1.
  4. If the correction text matches an existing R### entry (overlap coefficient ≥ 0.5 against the rule body), increment that entry’s recurring_count and bump strength by 1 instead of creating a duplicate. This is how Brain Correct surfaces patterns: repeated invocations against the same correction reinforce one rule rather than scatter many.
  5. Emit a learn_correct_applied log event with the new or reinforced rule ID.

Runs as a subagent — SubagentStop verifies

Brain Correct runs in a subagent. The parent invokes it with the user’s correction text and arguments. The subagent identifies the failure pattern and returns a proposed R### rule entry.

The SubagentStop hook then runs a single verifier check: the proposed R### entry validates against the prefrontal-rule schema (required fields, ID format, body present, source-message-hash captured). If schema validation fails, the rule does NOT land in cortex and a correct_verification_failed log event is emitted. Re-launch the subagent to retry.

Reads: the user’s correction text, optionally the recent conversation for context.
Writes: cortex/prefrontal/L1/R###.md (new entry or strength bump on existing).
Budget: 30 tool calls, 60 seconds. Much shorter than Brain Retro’s 80 / 300 because the scope is one correction, not a full session sweep.
Ledger: logger/learning/correct-NNNN.json per invocation. The same running stream that carries retro_summary events emits a correct_applied event on success.

Lineage with v1. Brain Correct is a v1 skill (referenced in cortex/prefrontal/L1/R140, R143, R145, R164; declared in cortex/broca/L1/og-broca-010 as “/brain-correct with arguments = encode correction immediately”). v2 keeps the same trigger and the same emergency-encode role; the SubagentStop verification pattern is new, inherited from Brain Learn.
