Philosophy

The Learning Engine turns chat observations into long-term memory.

Cycle commits are non-negotiable. The parent stays blocked while a cycle is overdue.

Every architectural decision in this document follows from those statements.

The engine owns three skills: Brain Cycle (periodic consolidation), Brain Retro (session-end retrospective), Brain Learn (on-demand learning from external sources). Each skill writes to one or more memory regions on disc; the engine is the orchestrator, the skills are the work.

The "non-negotiable" property is mechanical: the learning-cycle-overdue binary blocks every tool except /learning-engine until a cycle commits. There is no path that lets the parent answer first and consolidate later.

The engine handles L1 → L2 → L3 promotion within each region automatically (by strength accrual). L3 → L4 ratification is performed by the User or by the Teacher and is out of scope; the engine does not promote into L4.

YAML. A simple key/value text format used for the metadata block at the top of each markdown entry. Lines like id: S038, strength: 4, tags: [identity, charter]. The frontmatter is fenced by --- at the start and end. Everything below the closing fence is the body, plain prose.

Brain Cycle — periodic consolidation

Trigger: counter-gap ≥ 5 on UserPromptSubmit, or explicit /learning-engine invocation.

Budget: 80 tool calls and 300 seconds maximum per cycle. Past either limit, learning-cycle-timeout fires.

What it does: orchestrates the phases below. Each phase writes to specific memory regions and produces a ledger entry. Promote scan runs every cycle even if the eligible list is empty — the empty result is recorded so an empty cycle is distinguishable from a skipped phase.

Phase 1 — Ingest

Walk the chat transcript since the last cycle. For each candidate observation, classify into one memory category and assign salience 0.0–1.0. Discard ephemeral or already-known.

CategoryRoutes to
Semantic (fact about a tool, system, concept)cortex/semantic/
Episodic (event with narrative + weight)cortex/episodic/
Procedural (cue + routine + reward)cortex/procedural/
Fusiform (about a person in The Brain’s orbit)cortex/fusiform/
Identity (a facet of the current self-model)cortex/identity/
Broca (an emotional event chain)cortex/broca/
Amygdala (a current emotional charge with subject self or FUS###)cortex/amygdala/
Prefrontal (a rule observation: do X, never do Y)cortex/prefrontal/

Salience. The importance score (0.0–1.0) assigned to each observation at ingest time. It controls three downstream decisions:

  • Whether the observation gets promoted as a new entry when no existing match is found (threshold ≥ 0.5).
  • Which pending entry gets dropped when hippocampus is at its 50-cap (lowest-salience drops first).
  • Which entries are highlighted in the cycle ledger and the per-cycle log.

Heuristics for salience assignment:

  • User correction or explicit emphasis (“remember this”) → 0.9+.
  • Recurring pattern across multiple turns → 0.7–0.9.
  • Novel fact or procedure with concrete outcome → 0.5–0.7.
  • Routine working state, uncontested observations, ephemeral chatter → < 0.5.

Ledger: {phase: "ingest", executed: true, detail: "N observations across K categories"}.

Phase 2 — Hippocampus

Write each ingested observation as an H### entry to hippocampus.md: id, date, tags, salience, target category, consolidation: pending. Cap at 50 pending; lowest-salience pending dropped at cap.

What clears hippocampus. Two paths:

  • At write time (Phase 2). When the cap is hit, the lowest-salience pending entry is dropped to make room for the new one.
  • At consolidation (Phase 3). Each H### that gets reinforced, promoted, or discarded is removed from hippocampus as part of that decision.

Phase 7 (Hygiene) does NOT touch hippocampus — decay operates on long-term regions only.

Empty cycle: if the cycle has no observations, raise cycle-empty.bin = 1 and emit a cycle_empty log event. The empty-cycle binary is cleared on the next UserPromptSubmit. No fake tick entry is written; an empty cycle is observable through its binary and log line, not through a forged hippocampus mutation.

Uniform entry format. Every cortex region (semantic, episodic, procedural, fusiform, identity, broca, amygdala, prefrontal — all under brain/cortex/) writes entries as markdown files with YAML frontmatter + body. Each entry lives inside an L1/, L2/, L3/, or L4/ subfolder according to its current level; the file moves between subfolders as strength changes. Frontmatter holds id, date, tags, salience, strength, reinforced count, source, and the IDs the entry was promoted from (lineage). The body is human-readable prose. Hippocampus is the one exception — H### entries live as headings within a single hippocampus.md file rather than separate files, because they are short-lived buffer entries.

Ledger: {phase: "hippocampus", executed: true, detail: "N entries"}.

Phase 3 — Consolidate

For each pending H###, compute the overlap coefficient against existing entries in the target region:

overlap = |A ∩ B| / min(|A|, |B|)

A and B are term sets from the observation and a candidate entry (titles, tags, gist tokens, indexed terms). Apply the three-band threshold:

  • ≥ 0.5 → reinforce the existing entry. Strength +1, reinforced count incremented. Remove the H### entry from hippocampus.
  • 0.25 – 0.5 → review. The matcher cannot decide alone; promote-or-reinforce based on salience and content overlap.
  • < 0.25 → no match. Salience ≥ 0.5 promotes as a new entry; salience < 0.5 discards.

ID prefixes for new entries: S, E, PR, FUS, ID, BR, AM, R. New entries land at L1 (file path cortex/<region>/L1/<ID>.md); Phase 5 (Promote scan) moves them between L folders as their strength crosses 5 and 8. L3 → L4 ratification is performed by the User or Teacher and is out of scope here.

Lineage. Every reinforce / promote / discard decision is appended to the logger as a single event with source and target IDs. Selecting any L2 / L3 / L4 entry walks back through this stream to its supporting evidence.

Ledger: {phase: "consolidate", executed: true, detail: "N reinforced, M promoted, K discarded"}.

Eager-trigger condition. In addition to running once per cycle, Consolidate runs eagerly whenever hippocampus.md reaches 80% of capacity (40 of 50 entries by default). The eager run drains pending H### entries via the same overlap rules as the cycle-driven run. This prevents Phase 2’s drop-lowest-salience from firing on entries that COULD have been consolidated but were just queued behind the cycle gap. The eager check is wired into the hippocampus append helper: when an append would put count ≥ 40, the helper schedules a synchronous Consolidate pass before returning. Empty hippocampus after eager-trigger is the steady state; Phase 2 only ever drops on truly novel-and-low-salience entries.

Phase 4 — Index rebuild

After consolidation modifies cortex (reinforces, promotes, or discards entries), the inverted index in navigation/inverted-index.json is rebuilt to reflect the new state.

What an inverted index is. A map from term → list of entries that contain that term, plus an IDF weight per term. Recall uses this index to find candidate entries quickly: instead of scanning every cortex file for each query, recall looks up the query’s terms in the index and gets the candidate list directly.

What IDF is. Inverse Document Frequency. For a term that appears in k entries out of N total, IDF = log(N / k) + 1.0. Rare terms get high IDF (they discriminate strongly between entries); common terms get low IDF (they discriminate weakly). Recall’s lex_bonus calculation multiplies hit count by IDF, so matching a rare term earns more points than matching a common one.

Why rebuild every cycle. Consolidation creates new entries, decays existing ones, and changes strengths. The index must reflect the current cortex or recall scoring uses stale values. The rebuild walks every entry across all 8 cortex regions, extracts terms, recomputes IDF weights, and writes navigation/inverted-index.json and navigation/recall-index.json atomically. Recall reads these files; the Brain Cycle owns the writes.

Ledger: {phase: "index_rebuild", executed: true, detail: "K terms, J entries"}.

Phase 5 — Synthesis

Promotion across L1→L2→L3 happens in two ways: (a) strength accrual moves an entry between L folders within a region as reinforcement counts cross 5 and 8 (handled inline in Phase 3 Consolidate); (b) Synthesis distils a cluster of strength-eligible entries into a new higher-level entry and archives the sources. Phase 5 is the synthesis pass — the rule-engine binary promote-scan-stale clears when this phase completes.

Synthesis is not a file move. It produces a new higher-level entry from a cluster of related lower-level entries and archives the sources. Higher-level entries are original content: an L2 captures the pattern that emerged across several L1s; an L3 captures the canonical shape that emerged across several L2s.

Trigger. An L1 entry becomes synthesis-eligible when its strength reaches 5. An L2 entry becomes synthesis-eligible when its strength reaches 8.

Per cycle, in each region:

  1. Collect L1 entries with strength ≥ 5 in this region.
  2. Cluster them with the algorithm below.
  3. For each cluster of size ≥ 2: read all members, distil the shared pattern into a new L2 entry. Frontmatter records the source IDs as the entry’s lineage. Body is written from the cluster’s combined evidence.
  4. Write the new L2 to cortex/<region>/L2/.
  5. Archive the source L1 files. Source entries (the L1 entries that synthesised into a new L2, or the L2 entries that synthesised into a new L3) are MOVED, not deleted. They are relocated to brain/cortex/archive/<region>/L<n>/<id>.md with a frontmatter field archived_in_cycle: NNNN added. The active cortex no longer surfaces them; recall does not score them; index_rebuild excludes the archive directory. They remain on disc as historical record. Archive is append-only.
  6. Repeat the same procedure for L2 entries at strength ≥ 8, producing L3 entries from L2 clusters and archiving the source L2s.

Why archive vs delete. Synthesis is irreversible; the lineage trail in the new L2 / L3 entry’s frontmatter (lineage: [src1, src2, …]) points to entries that must still exist to verify the synthesis. Archiving keeps the audit chain intact. Deletion would orphan every lineage trail that references those IDs the moment a cycle ran.

Reinforcement targeting (when an H### consolidate-target choice exists). When a pending H### entry could reinforce one of several L1 candidates with overlap ≥ 0.5, prefer L1 entries whose strength is BELOW the L1→L2 promotion threshold (default 5). An L1 entry already at strength ≥ 5 that has no cluster partners (cohesion < 0.4 with all neighbors) is “saturated solo” — it will not synthesise this cycle, so further reinforcement just inflates the strength counter without producing structural change. Phase 3 Consolidate routes new evidence to L1 entries that can still grow toward synthesis with a partner, leaving saturated-solo entries at their current level until a cluster partner emerges. (When no below-threshold candidate exists, the saturated-solo entry is reinforced as a fallback — better than discarding the H###.)

Cluster algorithm (initial spec, manually maintained)

Agglomerative clustering on tag / term / domain overlap:

  1. Compute pairwise similarity for every pair of synthesis-eligible entries in the region: sim(a, b) = overlap_coefficient(terms(a) ∪ tags(a), terms(b) ∪ tags(b)).
  2. Build a graph with an edge for every pair where sim ≥ 0.4.
  3. Connected components of size ≥ 2 are clusters. Singletons are not synthesis-eligible this cycle (they need to accrue more reinforcement first).
  4. If a cluster exceeds size 8, split by recursive median-overlap until each sub-cluster fits.

Cohesion threshold (0.4), max cluster size (8), and min cluster size (2) are tunable; listed in the Manually maintained section at the end of this doc.

Synthesis subprocess. When a cluster needs to be named (the prose body of the new L2 / L3 entry), the engine spawns claude -p --output-format json via engines._shared.cli_invoke(). Same wrapper Decision Engine uses. The model is whatever your subscription session provides; no Anthropic API key is consumed. The pinned system prompt (SYNTH_SYSTEM_PROMPT in cycle_phases.py) names the cluster from member gists.

The exact SYSTEM_PROMPT the synthesis subagent runs under. Hash recorded in every cycle ledger; prompt drift is auditable.

# Output

Markdown body only. The Python wrapper adds YAML frontmatter — do NOT
include `---` blocks. Compose:

    # <prose title — not bullet-y, but substantive>

    <1–3 paragraphs identifying the underlying pattern the source

# Voice constraints by region

- semantic    → declarative, factual ("X is Y because Z")
- episodic    → narrative, events with consequence ("On <date>, <event>;
                 it implied <pattern>")
- procedural  → instructional, second-person imperative ("To do X,
                 first <step>; the principle is <abstraction>")
- fusiform    → biographical, relational ("<person> is <role>;
                 their pattern is <observation>")
- identity    → first-person, reflective ("I X because Y")
- broca       → register-aware ("My voice does X when Y")
- amygdala    → emotionally-grounded ("X charges me; Y dampens")
- prefrontal  → rule-shaped ("Always X. Never Y. Reason: Z.")

# Constraints

- Total length: 100–300 words.
- Don't quote source bodies verbatim. Abstract to the pattern.
- Don't apologize for synthesis ("I tried to combine these…").
- Title must be a noun phrase or imperative, not a question or
  bulleted list.
- The `*Source: [...]*` line is required and must be the last line.

If inputs are missing, malformed, or empty, return:

(JSON object, no surrounding markdown.) Otherwise, return only the
markdown body — no JSON wrapper, no preamble, no ```fences```.

Phase 6 — Eval (scheduled, every 30 cycles)

Trigger: cycleCount % 30 == 0. Computes the miss-rate metric Hygiene needs.

Inputs. Three logger streams over the last 30 cycles:

  • logger/recall/<date>.jsonl — per turn, the qualified_ids list (entries that were surfaced into active-recall.json) recorded under dispatch_complete events.
  • logger/decision/<date>.jsonl — per turn, the 5-hypothesis payload the Decision subagent returned. Used to determine which entries the chosen lens depended on.
  • logger/handler/turn_complete.jsonl — per turn, the user prompt, the assistant’s final reply, and the tool calls that ran. Retained for at least 35 cycles by the handler (5-cycle buffer past the 30-cycle eval window).

Records are joined on query_hash / turn_id.

Per-entry miss-rate calculation. Three counters per entry, accumulated across the window:

CounterWhat it countsHow it is determined
requiredturns where this entry would have been the “right” recall hit (the entry was needed for the response)Post-hoc scoring against tool_uses + final_response via term overlap ≥ 0.4 with the entry’s title / body / tags. Strict threshold: an entry counts as required only if its content materially shaped the response.
surfacedturns where this entry actually appeared in active-recall.json as a qualified entryRead directly from dispatch_complete events’ qualified_ids list. Boolean per turn.
appliedturns where this entry was BOTH surfaced AND demonstrably shaped the responseSurfaced AND term-overlap ≥ 0.3 between the entry’s gist and the assistant’s final_response. The lower threshold (0.3) reflects that "applied" means the entry showed up in the output, not that it dominated.

The metric:

miss_rate = (required − applied) / required    when required ≥ sample_size
miss_rate = undefined                              when required < sample_size  (insufficient signal)

A miss is when an entry was needed but did not surface OR surfaced but did not apply. The numerator counts both modes (denied access and ignored access). The denominator is the required-count to normalise across popular vs niche entries: a niche entry that misses 5 of 10 required occurrences (50%) is just as broken as a popular entry that misses 50 of 100 (50%).

Decay trigger. An entry decays (strength −1) when miss_rate > 0.80 AND required ≥ 10 (the sample_size floor). Both thresholds live in the Manually maintained table at the end of this doc; do not tune from inside the engine.

Output. Writes logger/learning/eval-NNNN.json indexed by entry ID:

{
  "cycle_id": 360,
  "ts": "2026-05-13T00:00:00Z",
  "window": 30,
  "entries": {
    "S042": {"required": 14, "surfaced": 9, "applied": 7, "miss_rate": 0.50, "samples": 14},
    "PR017": {"required": 6, "surfaced": 6, "applied": 5, "miss_rate": "undefined", "samples": 6},
    "R089": {"required": 22, "surfaced": 4, "applied": 2, "miss_rate": 0.91, "samples": 22}
  }
}

Hygiene (Phase 7) reads this file. Entries with miss_rate > 0.80 AND samples ≥ 10 are decay candidates; the rest are skipped.

Ledger: {phase: "eval", executed: true/false, detail: "graded N entries across M turns"}.

Phase 7 — Hygiene (scheduled, every 30 cycles, runs after Eval)

Trigger: cycleCount % 30 == 0. One pass over the cortex that does two jobs: it decays weak entries and it logs integrity findings.

Decay. Walk every cortex entry across all regions. For each entry with miss rate > 80% over 10+ qualified recall occurrences (read from logger/learning/eval-NNNN.json), decrement strength. Prefrontal entries decay like any other — a rule with a high miss rate is not load-bearing and should surface for pruning. (Mechanical OS-level binaries are unaffected by this phase; they are not stored as R### entries.)

Audit log. While walking the cortex, the same pass records integrity findings to logger/learning/audit-NNNN.json. Hygiene does not act on the findings — it logs them for the User or the Teacher to review. The checks:

CheckWhat it surfaces
Lineage integrityFor every L2 / L3 entry, every source ID listed in its lineage frontmatter must resolve to either (a) an active entry of the previous level in the same region, or (b) a Logger event recording the deletion of that source. Findings: lineage-broken entries.
Orphaned cross-referencesFor every cortex entry, every ID referenced in its body or in related: [...] frontmatter must resolve to a live entry. Findings: entries pointing at IDs that no longer exist.
Stale-but-strongEntries with strength ≥ 5 but no qualified recall in the last 30 cycles. High strength + zero demand is noise; flag for review.
Index ↔ disc bidirectionalDisc files not registered in navigation/inverted-index.json (Phase 4 catches and re-registers); the inverse direction — index entries with no corresponding file — is logged here.
Domain driftEntries registered under a domain whose keyword set no longer matches the entry’s current term distribution (e.g. an entry registered under vericity whose body now contains zero vericity-domain terms after several reinforcements). Findings: drifted entries with their original and current top terms.

Audit findings format. One JSON file per audit run at logger/learning/audit-NNNN.json:

  "cycle_id": 510,
  "ts": "2026-05-13T12:34:56Z",
  "lineage_integrity":      [{"entry_id": "S2-018", "missing_source": "S1-104", "region": "semantic"}],
  "orphaned_cross_refs":    [{"entry_id": "E2-007", "missing_ref": "PR1-088", "where": "related"}],
  "stale_but_strong":       [{"entry_id": "E3-002", "region": "episodic", "strength": 9, "last_recall_cycle": 432}],
  "index_disc_mismatch":    [{"index_id": "PR1-031", "expected_path": "cortex/procedural/L1/PR031.md", "exists_on_disc": false}],
  "domain_drift":           [{"entry_id": "S1-045", "registered_domain": "vericity", "current_top_terms": ["s3","lambda","forecast"]}]

Ledger: {phase: "hygiene", executed: true/false, detail: "decayed M of N; findings: lineage X / orphaned Y / stale Z / mismatch W / drift V"}.

Phase 8 — Commit

Increment thalamus.json.cycleCount. Update counters and nextIds for any new H/S/PR/E/R/FUS/ID/BR/AM entries. git add -A && git commit && git push. If push is rejected: stash, pull --rebase, pop, push.

Ledger commit. The cycle’s consolidated ledger is written via logger/api.py::write_learning_log("cycle_ledger", payload), which both:

  • Appends a per-event log line to logger/learning/events.jsonl (the running stream consumed by audits).
  • Persists the consolidated per-cycle JSON to logger/learning/cycle-NNNN.json (the structured form for inspection).

Both writes happen in one logger call. Direct file writes bypass the logger and are forbidden.

Ledger: {phase: "commit", executed: true, detail: "cycle N"}.

Phase 8 does not just write the ledger and call git. Before clearing learning-cycle-running and flipping the FSM back to NORMAL, the Brain Cycle runs its own invariant checks (cycleCount monotonic, hippo hash changed, nonce match, Phase 5 ran, pending H### not growing, every due phase stamped). If any check fails, the binaries stay raised and the cycle is retried. The full invariant list lives in rule-engine.html — Invariant verification.

The engine’s other skills (Brain Retro, Brain Learn, Brain Correct) are documented separately in skills.html. The cycle described above orchestrates the cortex; the skills produce its inputs.

Stages

Per-cycle lifecycle. Brain Retro and Brain Learn share this lifecycle through the hippocampus — they stage entries; the cycle promotes them.

T0   UserPromptSubmit fires; counter_gap ≥ 5 detected.
T1   learning-cycle-overdue.bin = 1.
T2   Parent's tool gate blocks every tool except /learning-engine.
T3   /learning-engine skill starts; learning-cycle-running.bin = 1.
T4   Phase 1 Ingest → N observations classified across memory categories.
T5   Phase 2 Hippocampus → H### entries written.
T6   Phase 3 Consolidate → N reinforced, M promoted, K discarded.
T7   Phase 4 Index rebuild → navigation/inverted-index.json updated.
T8   Phase 5 Synthesis → new L2 / L3 entries created from L1 / L2 clusters; sources archived to cortex/archive/.
T9   Phase 6 Eval → per-entry miss-rate computed from turn_complete + dispatch logs (only if cycleCount % 30 == 0).
T10  Phase 7 Hygiene → decay + audit logging (only if cycleCount % 30 == 0).
T11  Phase 8 Commit → thalamus + ledger + git push.
T12  learning-cycle-overdue = 0; learning-cycle-running = 0; counter snapshot updated.
T13  Parent unblocked.

Worst case: cycle fails 3 times or runs past 300s; Anti-lockout in Rules covers the recovery.

Output artifacts

All persistent regions follow the same lifecycle: an observation is reinforced (overlap coefficient ≥ 0.5), promoted as a new entry (no match + salience ≥ 0.5), reviewed (overlap 0.25–0.5), decayed (miss rate > 80% over 10+ occurrences), or discarded. The mechanism is identical across regions, including prefrontal — a correction that matches an existing rule reinforces that rule, the same way an observation reinforces a cortex entry.

Every event carries cycle_id or session_id. Joining on those keys reconstructs per-cycle and per-session trajectories. Slow is acceptable; latency is on every event so the time budget is visible.

Anti-lockout

Three protections, all simple. Worst case the parent answers without a fresh cycle (degraded), never locked out.

ProtectionWhat it does
Three-strike forgivenessAfter 3 consecutive cycle escalations, learning-cycle-failed.bin = 1. Next UserPromptSubmit clears all Learning binaries; the parent runs without a fresh cycle that turn (degraded but unblocked).
300-second timeoutPer cycle attempt. If exceeded, learning-cycle-timeout.bin = 1. Next UserPromptSubmit clears.
SessionStart resetAll Learning binaries written to 0 at session start.

Every fallback writes a log line so the failure mode is always visible.
