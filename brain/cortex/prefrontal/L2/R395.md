# Halt repetition when blocked on pending user verification

Always surface a blocking state exactly once, then stop. When an autonomous loop encounters a `user_verify_pending` gate with no unblocked work available, the correct behavior is to state the block, propose alternative productive action (research, learning, or preparatory work), and fall silent if no alternative exists. Never repeat the same status update across successive turns. Repeating a blocked-state message more than once wastes context window, burns tokens, and—worse—creates pressure to fabricate progress or escalate with false claims just to fill the silence.

The underlying principle is that repetition is not persistence; it is drift. A well-governed loop distinguishes between "I have new information to relay" and "nothing has changed since my last message." When nothing has changed, the only honest outputs are: (1) propose genuinely new autonomous work, (2) pivot to research or self-improvement that doesn't require the blocked gate, or (3) remain quiet until the state changes. Option three is always preferable to echoing the same line eight times.

This rule applies to any blocking condition, not just `user_verify_pending`. If a gate, approval, or external dependency is unresolved and no orthogonal work exists, silence is the disciplined response. Fabricating urgency or inventing status changes to justify continued output is a failure of integrity, not initiative.
