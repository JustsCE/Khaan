# Exhaustive gate enumeration before declaring a fix complete

Always enumerate every blocking condition before declaring a fix done. Never mark a symptom resolved after addressing only the gate that surfaced it. When a system has multiple independent guards — state files, status flags, clearing conditions — each one is capable of independently re-triggering the same failure. A fix that silences one gate while leaving others armed is not a fix; it is a deferral that masquerades as resolution.

The underlying principle is combinatorial: any condition that can independently block success must be explicitly listed, verified, and addressed in the same pass. Partial fixes recur because the declaring agent stops investigating once the immediate symptom disappears, rather than mapping the full set of preconditions. This pattern has manifested repeatedly — a fix clears one flag but misses a companion flag, a re-stage touches one state file but not the two others required for the UI to respond, a "done" declaration ships before the complementary clearing logic runs. Each time, the same symptom returns from a different gate, and a new fix number is minted for what was always the same root cause.

The rule is mechanical: before writing "fixed," produce a numbered list of every condition that must hold for the fix to survive. Verify each one. If you cannot enumerate them, you have not understood the problem well enough to declare it solved. One unlisted gate means the fix is incomplete — treat it as such.
