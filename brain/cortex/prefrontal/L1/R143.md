# R143: OAuth apps must be in "In production" mode

**Strengthens PR097.** Whenever Brain manages an OAuth integration where users are NOT going through Google's OAuth verification process for individual users, the OAuth consent-screen MUST be in "In production" publishing status, NOT "Testing".

## Why

Google Cloud OAuth apps in "Testing" status invalidate refresh_tokens after 7 days. This breaks any long-running integration that was working before (Gmail puller, Slack-style background jobs, etc.) silently — Brain doesn't fail loudly, the credentials just rot. Per PR097 + 4+ incidents now (Apr 18 setup → Apr 25 expiry being only the most recent), this is a recurring trap.

## Rule

For any new OAuth integration:
1. Right after creating the OAuth client, navigate to OAuth consent screen.
2. Under "Publishing status", click "PUBLISH APP". Confirm.
3. For single-user / personal apps, this is one click. No verification required.
4. For multi-user apps with sensitive scopes, verification IS required — flag this to user before continuing.

For any existing OAuth integration that breaks:
- ALWAYS check publishing status FIRST (`gcloud alpha iap oauth-clients` or Console).
- If Testing → publishing is the actual fix. Re-issuing tokens without publishing = guaranteed re-break in 7 days.

## Auto-refresh access tokens (separate concern)

`access_token` lifetime ~1h. Refreshing it is automatic via `refresh_token` and is harmless to do early. Cron `*/30 * * * *` running `/opt/brain-viz/secrets/gmail/refresh-token.sh` keeps the access token fresh and surfaces any deeper failure (invalid_grant) to logs. This does NOT solve the Testing-mode 7-day issue — only publishing the app does.
