# R131: HTTP 403 = Auth, Not Network

**Rule:** When a Claude.ai-hosted agent (Solo Time, scheduled routine) reports HTTP 403 hitting goodhealth.lv or any user-owned endpoint, the first hypothesis is **missing authentication credentials** (bearer token, API key), NOT sandbox outbound-HTTPS blocking.

**Why:** 2026-04-22: I told Justs "Solo Time can't reach goodhealth.lv because Anthropic sandbox blocks outbound HTTPS" — citing [[[[[[[[[[[[[[[[[[[PR-026 (resolved) (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)]. Justs pushed back: "but how come whatsapp agent is able to send messages?" He was correct — the WhatsApp MCP agent demonstrably reached goodhealth.lv via MCP endpoint. Actual root cause: **Solo Time worker doesn't have DASHBOARD_TOKEN env var set**, so the dashboard API rejects the request (403 Forbidden, not connection refused).

**How to apply:**
1. 403 response = server received request, rejected it (auth/authz)
2. Sandbox/network blocks produce different symptoms: `ECONNREFUSED`, DNS failure, timeout — NOT 403
3. Cross-check: does ANY other agent (same environment, different endpoint) succeed? If yes, it's not network blocking
4. Fix path: verify the auth token is set in the routine's env config (Routines UI → Settings → env vars)

**Severity:** Medium — wrong root-cause diagnosis misleads the user toward the wrong fix (adding allowlist entries instead of setting env vars).
