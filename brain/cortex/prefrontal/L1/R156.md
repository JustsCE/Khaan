# R156: Test-staging must touch host state.json

## The rule

When re-staging a project for cinematic testing (flipping `verified` → `user_verify_pending`), edit BOTH:
1. **Host** `~/.claude/brain/projects/<id>/state.json` (git-tracked)
2. Container `/brain/brain/projects/<id>/state.json`

Editing only the container is insufficient because the brain-viz container periodically `git pull`s from the host repo (or via webhook on push), and pulls overwrite container-only edits with the host's `verified` state.

The dashboard `/brain/dashboard/tasks/whatsapp.json` is gitignored — that one CAN be edited container-only.

## Why this rule exists

2026-04-28: TWICE in one session, the user reported that a re-staged project (mobile-viz specifically — the one I added back as a fresh source task) was NOT shimmering on /brain. Investigation revealed: state.json on host was `verified`, container's was `user_verify_pending` for a moment, then container did `git pull` → host's `verified` overwrote my staging.

User correction: "the task you just added isnt shimmering in silver - AGAIN!"

The first time it happened, I shrugged it off as a one-off race. The second time made it a pattern. Encoded.

## How to apply

In the `revert-projects.py` test-staging script (or any equivalent):

```python
import json, subprocess
for pid in PROJECTS:
    host_path = f'/home/ubuntu/.claude/brain/projects/{pid}/state.json'
    with open(host_path) as f: s = json.load(f)
    # ...flip status to user_verify_pending...
    with open(host_path, 'w') as f: json.dump(s, f, indent=2)        # 1) host
    subprocess.run(['docker','exec','-i','brain-viz-brain-viz-1','sh','-c',
                    f'cat > /brain/brain/projects/{pid}/state.json'],
                   input=json.dumps(s, indent=2).encode(), check=True)  # 2) container
```

After staging, optionally commit + push the host changes so future container pulls keep the staged state. The user's verify-clicks will write `verified` back via cmd_user_verify and gitPersistProject.
