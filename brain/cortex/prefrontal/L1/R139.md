### R139 — og Brain — Use mouse events for drag across sibling containers
**Added:** 2026-04-24 | **Strength:** 1 | **Source:** brain-retro rewire
**Tags:** correction, brain-retro, dnd, react, ui
**Rule:** When building drag-and-drop that must cross container boundaries (e.g., between sibling FloatingPanes with `position:absolute` + `overflow:hidden`), DO NOT use HTML5 drag API (`draggable`, `onDragStart`, `onDrop`). HTML5 drag events don't fire on unrelated DOM trees — only on the drag source and its descendants.
**Fix:** Use mouse events instead: `onMouseDown` → document-level `mousemove`/`mouseup` → `document.elementsFromPoint()` for hit-testing. Same pattern FloatingPane uses for its own drag. Works across any DOM structure, overflow, z-index.
**Defense:** Add a 5px drag threshold so clicks still fire. Add `data-droppable-*` attributes for hit-testing targets.
**Evidence:** Session 2026-04-24 — Spent 3 iterations trying to fix HTML5 drag across FloatingPanes (fixed dragend race, fixed guards, fixed ids). None worked because the events never fired on cross-pane targets. Switched to mouseevents, shipped working.
