# R154 -- og Brain — Justify Cadence; Prefer Push Over Poll

**Created:** 2026-04-27
**Status:** active
**Strength:** 1
**Reinforced:** 0
**Domain:** prefrontal (decision rule)

## Rule

When proposing scheduled / repeating work (cron, systemd timer, agent-triggered loop):

1. **Justify the cadence** — pick the LONGEST interval that still meets the actual user-visible requirement. Default to "eventually consistent" unless you can name a specific need for snappier response.
2. **Surface the cost/risk profile up front** — per-tick CPU, network calls, GH/API quota usage, interference with other agents. State explicitly that the cost is or isn't a concern, with reasoning.
3. **Prefer push-driven over poll-driven** when both are available. Polling is a fallback / drift-recovery mechanism, not a primary trigger.

## Why

- Aggressive polling spends limited resources (free tiers, API rate limits, CPU on shared hosts) for no user-visible benefit when the source can push.
- Justs explicitly pushed back on a 60s timer ("Cron every 60secs seems very aggressive. Wont it kill the server / incur costs / mess something up if agents are running?") — the question of cadence vs need wasn't justified, even though the per-tick cost was negligible.
- He then rejected polling entirely in favor of GitHub's native push webhook (A1 from a 4-option plan) — confirming the push-over-poll preference.

## Apply when

- Designing brain skills with timers
- Provisioning systemd services / cron jobs
- Choosing between webhook-driven and poll-driven sync
- Explaining ANY new background process to user

## Anti-pattern

"Poll every 60s, it's cheap." → Even if true, the user shouldn't have to stop you to make you justify it.
