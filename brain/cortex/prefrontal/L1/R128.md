# R128: Verify the Artifact Before Reporting Task Status

**Rule:** Before answering any status question ("did X happen?", "is Y done?", "what's pending?"), check the actual ground-truth artifact (repo, file, DB row, message ID, API response). Do NOT report from autonomy logs, intake notes, hippocampus, or memory alone — those can be stale, partial, or just wrong.

**Why:** 2026-04-22, Brain told Justs the Robson stats task was never executed because intake logs said "skipped — repo not authorized." The HTML report actually exists in JustsCE/stats. Justs had to correct: "why the fuck do I see an HTML report in the stats repo." Trusting the narrative log over the artifact is exactly the failure mode this rule blocks.

**How to apply:**
1. When asked about status, identify what artifact would PROVE the answer
2. Check that artifact via direct query (`gh api`, `git log`, `ls`, DB query, etc.) BEFORE answering
3. If the artifact and the log disagree, the artifact wins — and the log entry should be flagged as stale/wrong
4. If you can't check the artifact from this environment, say so explicitly rather than guessing from the log

**Severity:** High — confident wrong answers about status erode trust faster than admitting uncertainty.
