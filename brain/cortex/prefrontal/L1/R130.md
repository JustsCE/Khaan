# R130: Primary Format First, Alternates After

**Rule:** When user asks for a format conversion (Google Doc, DOCX, XLSX, PDF) AFTER work is underway, COMMIT and report the primary/original format first (PDF or repo artifact) — THEN offer the alternate as a follow-up. Never block delivery on mid-pipeline format chaining.

**Why:** 2026-04-22: user asked for PDF results report. I built it. User then asked "create a Google Doc too". I spent ~5 mins base64-encoding a 400KB DOCX for Drive upload (~564KB payload, ~140K tokens of context). User pivoted to "skip google doc just give me pdf" with visible frustration ("why does it take so long?"). The PDF had been ready the whole time.

**How to apply:**
1. When user adds a format request mid-work: FIRST push/deliver what's already built
2. THEN offer: "PDF is live at `<url>`. Want me to also generate a DOCX/Google Doc as follow-up?"
3. NEVER do: "hold on, let me convert first, then deliver both at once"
4. Format-conversion pipelines that require >30 seconds of tool calls or >20K tokens of payload MUST be deferred behind primary delivery

**Severity:** Medium — wastes user time + burns context window. Existing PR112 captures this but was reinforced today.
