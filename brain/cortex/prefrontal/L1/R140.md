# R140: Never guess at performance causes

When the user reports lag / slow / janky — STOP. Do NOT pick a plausible-sounding cause and refactor against it.

## Required before any perf fix
1. Ask the user what exactly is laggy — scroll, click, 3D orbit, page load, hover, first paint? Which page? Which device? At what moment?
2. OR measure — Chrome DevTools Performance trace, FPS counter, network waterfall, GPU frame time. Find the actual hot path.

Do NOT refactor on a hunch. Plausible-sounding causes (useFrame, autoRotate, draw calls) may not be THE bottleneck. Without evidence, you are coding in the dark and wasting the user's time.

## Why this rule exists
2026-04-24: user reported lag on goodhealth.lv/brain. Guessed 3 times:
1. Stripped per-neuron useFrame -> "that wasnt it"
2. Disabled autoRotate + added InstancedMesh -> "THAT ALSO WASNT THE PROBLEM"

User escalated to /brain-correct. Root cause still unknown because I never measured or asked.

## How to apply
On the next lag report, the first reply is a question, not a commit. Offer a profiling snippet if the user is willing (DevTools Performance tab, 5-sec trace).
