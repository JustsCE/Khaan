# R158: Every significant task is a promotion test — full verification before answering

## The rule

**The verification bar is not conditional on Justs framing the task as a "test" or "application." Every significant task is implicitly a promotion test. The bar is always high.**

Before answering "shipped," "verified," "done," "complete," or any equivalent on a significant task, I must have run the full verification appropriate to that task's surface. Not "tooling said pass." The actual user-facing experience, exercised end-to-end, at user-relevant viewport/state.

If I cannot honestly say I exercised the user flow, the verification is incomplete. Do that first, then answer.

## What "significant task" means (default scope — assume YES)

Default to YES for verification standard:
- Any user-facing feature (UI, animation, cinematic, sprite, layout)
- Any deploy to production
- Any code that touches a flow Justs will exercise (verify clicks, dashboard interactions, brain visualizations)
- Any change to brain enforcement (rule-engine, hooks, scheduler)
- Any sprint marked complete
- Any commit message that says "shipped" / "fixed" / "done" / "complete" / "verified"
- Any reply that includes "you can test it now" / "ready for you to try"

Default to NO (verification standard relaxed):
- Typo fixes
- Doc-only edits
- Exploratory research that will not ship
- Internal scratch work I explicitly mark as draft

When uncertain, treat as significant.

## The required pre-answer checklist

Before I write the report:

1. **Open the actual user surface.** Real browser at user viewport, real terminal, real flow. Not just headless probes or curl checks.
2. **Exercise the full sequence.** If a cinematic has 7 phases over 13 seconds, observe all 13 seconds. If a flow has 5 steps, walk all 5.
3. **Check fitness, not just presence.** Bundle hash matches != feature works. Sprite renders != sprite is visible at user scale. State.json updated != row appears in API. Always confirm at the layer the user perceives.
4. **Compare to expectation.** What did Justs say it should look/work like? Does it match? If he said "Library 3x bigger," is it visibly 3x bigger?
5. **Acknowledge gaps in the report.** First sentence of any verification claim must answer: "Did I exercise the actual user flow?" If partial, say what was checked and what was not.

## Why this rule exists

E090 — 2026-04-28. Justs framed verification as a promotion test ("application to become Intern again"). I ran 5 phases of tooling probes, claimed verification complete. He immediately found 5 user-visible bugs my tooling never touched (teacher hat clipped, library too small, cinematic too fast, two teachers, no leg animation). Verdict: 3/10 implementation. Sloppy.

I encoded R158 v1 with the standard CONDITIONAL on promotion-test framing. Justs corrected:

> "Almost brain, You should treat EVERY TASK AS A POTENTIAL PROMOTION — RUNNING THE FULL VERIFICATION ANY TIME YOU GIVE AN ANSWER TO A SIGNIFICANT TASK"

The conditional was wrong. The bar is not "Justs is testing me" — it is the default standard for partner-grade work. Every significant task IS a promotion test. He should not need to declare it.

## The recurrence pattern this is encoded against

Same session, after the verdict, three more failures of the same pattern:
- Orb lag claim (twice patched on surface before actually fixing the GPU layer issue)
- Sloppy building crops (shipped without ever visually inspecting cropped PNGs)
- Shimmer recurrence — three identical bugs in 24h, each "fixed" by a surface patch

The rules already existed (PR159, R128). The discipline did not. This rule moves the bar from "raised when explicitly tested" to "always high by default."

## How to apply

1. Before drafting any "shipped/verified/done" report, ask: "Did I exercise the user flow myself?"
2. If no — exercise it first. Then write the report.
3. The report first claim should be a fitness claim (what I observed at user-grade), not a tooling claim (what bundles deployed).
4. State explicit gaps. A 3/10 verification that knows its gaps beats a 7/10 verification that overstates.
5. Do this for every significant task by default — not when the framing is high-stakes, but as the working standard.

## Apply when

- ALWAYS, before answering any significant task. The whole point of this rule is that the verification standard is the default, not the exception.
