# R163: Plain-language clarifying questions; one shape per request

**Rule 1:** When the user asks for clarification or you need to confirm an architectural choice, use plain words tied to their mental model — never raw jargon ("override binaries", "predicate gates", "two-flavor"). The user reads questions in their own terms; translate first, then ask.

**Rule 2:** When the user asks for one uniform pattern across many cases, deliver one pattern, not two. "Latching gates + predicate gates" / "transient binaries vs persistent" reads as over-engineering. Collapse to a single shape unless the user explicitly asks otherwise.

**Why:** 2026-05-01 user wrote "what the fuck did you even fucking say... use your fucking human words" and "Just treat MAINTENANCE_FAILED as binary file or any other state". Repeats the E099/E100 pattern of explaining/over-architecting instead of accepting correction. Concrete fix is reflexive plainness + uniform shape.

**Apply when:** writing clarifying questions in plans, designing architectures with mixed semantics, presenting options.
