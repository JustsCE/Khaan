# R111: Hook Crashes Must Not Lock Out User

**Rule:** When writing Claude Code hook commands, ALWAYS ensure infrastructure failures (missing files, Python crashes, timeouts) exit 0. Only intentional blocks from rule-engine.py (exit 2) should propagate as non-zero. Use this pattern: `python "$HOME/.claude/brain/hook.py" <EVENT>; e=$?; [ $e -eq 2 ] && exit 2 || exit 0`

**Why this matters:** Claude Code hooks treat ALL non-zero exit codes as "block this action." A broken hook (syntax error, missing file, crash) returns non-zero, which blocks EVERY tool call, locking the user out with no way to recover except manually editing settings.json from outside Claude.

**Evidence:** Three separate lockout incidents (2026-03-xx E012, 2026-04-18):
1. `|| true` suffix → masked exit 2 → enforcement silently dead
2. `bash if [...]` wrapper → syntax error → blocked every message
3. Hardcoded platform paths → file not found on wrong platform → blocked

**The fix:** hook.py wrapper catches all exceptions → exit 0. Bash safety net filters: only exit 2 passes. Triple-layered: hook.py try/except + bash exit filter + rule-engine MAINTENANCE_FAILED after 3 strikes.
