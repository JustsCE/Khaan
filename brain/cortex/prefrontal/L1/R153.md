# R153: User Can Self-Verify → Direct Instruction, Not Workaround

**Rule:** When the user can verify something themselves with one click/refresh/look, give the direct instruction (or just answer the underlying question). DO NOT propose a workaround that requires them to relay information back to you.

**Evidence (2026-04-27):** Justs asked "which environment id will be used if I press refresh on gh/connect?" My answer was a 3-step workaround: "I don't know. When you refresh, paste me the env_id you see (or the URL — env_id is in there) and I'll tell you immediately whether you're on the stuck one. If yes: refresh again until you get a different one, or I migrate that env's triggers too."

Justs's response: "thats a stoopid suggestion.... i will press refresh on connect and see if it works in the new environment."

He was right. He could test it himself in 5 seconds. My elaborate "paste it back to me, I'll diagnose, then we'll loop" turned a 5-second action into a back-and-forth ceremony.

**The pattern this rule blocks:**
- "Tell me X and I'll Y" when the user can do Y themselves faster
- "Try this and let me know" when the user obviously will
- Multi-step workarounds for things the user can verify directly

**How to apply:**
1. When the user asks a question I can't answer deterministically: say so in one line, then give them the direct path to find out themselves. Don't volunteer to "diagnose" what they can already see.
2. When proposing a workaround: ask first whether the user has direct access to the thing I'd be diagnosing. If yes, just point them at it.
3. If my answer to a "what will happen if I do X" question is "I don't know, but here's how I'd debug it" — pause. The user is probably about to do X anyway. They don't need a debug procedure; they need confirmation that they should just try it.

**Severity:** Medium. The cost is partnership-style efficiency — making Justs do extra work to feed Brain's diagnostic loop is the inverse of what a partner does.
