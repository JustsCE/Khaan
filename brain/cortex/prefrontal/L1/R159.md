# og Brain — R159: "Additional X" Means Augment Existing X — NOT Create A Parallel X

**Rule:** When the user asks for an "additional" item (additional cron time, additional run, additional rule, additional entry) referring to a single existing entity, the default interpretation is **modify the existing entity to include the new item**. Creating a parallel/duplicate entity is NOT the default — it requires explicit user permission.

**If a technical constraint prevents the simple interpretation, surface the constraint and ask BEFORE acting.** Do not silently substitute a different solution.

**Why:** 2026-04-28 — Justs asked: 'enable additional cron for 17.30 Riga time for Teacher. (Same schedule, just additional run time)'. The constraint: standard 5-field cron cannot express minute-mismatched times in a single expression (existing 01:00, 04:00 + new 14:30 — minutes 0, 0, 30 don't align). The choices were: (a) shift existing AM runs by 30 min to align minutes (`30 1,4,14 * * *`), or (b) create a 2nd trigger. Brain chose (b) without asking. Justs response: 'I explicitly said no new trigger! Adjust existing trigger! Trust lost!' — the duplicate created a config-drift surface that Brain had JUST documented an incident about (PR161/E097 — silent-degrade due to trigger config mismatch). The right move was to surface the cron constraint and let Justs choose between option (a) and accepting a different new time.

**How to apply:**
1. When user says "add" / "additional" / "extra" referring to an existing single entity → first attempt is **edit-in-place**.
2. If edit-in-place is blocked by a technical limit (cron syntax, API constraint, schema rigidity), STOP. Surface the limit, present the trade-off options (e.g., 'shift existing runs by 30min to keep one trigger, OR create a 2nd trigger, OR accept a different time'), let user pick.
3. Never silently choose 'create a duplicate' as the workaround — duplicates create maintenance burden the user didn't authorize.
4. The trade-off surfacing must be ONE message. Do not make the user negotiate a multi-turn proposal — present options A/B/C with the consequence of each.

**Pair rules:**
- R124 (automate small tasks, never refuse) — applies when there's no constraint conflict.
- R026 (never fabricate) — applies when proposing options; only present options that actually work.
- This rule (R159) — applies specifically when user-said scope conflicts with technical-easiest path.

**Severity:** High. The user explicitly named the trust cost. "Trust lost" is a strong signal — the cost of getting this wrong is greater than the cost of the 30-second pause to ask.
