# R074: Always Test on Cheapest Tier Before Upgrading

When deploying to cloud services, ALWAYS start with the cheapest tier (Consumption/free) and test first. Only upgrade to Premium/paid tiers if the cheap tier demonstrably fails.

Justs will never approve buying something outright without testing the free option first. This applies to:
- Azure Function plans (Consumption before Premium)
- Database tiers
- Container hosting options
- Any paid service

**Evidence:** Brain created a Premium EP1 plan ($150+/mo) for CLIP scorer without testing if Consumption would work. The function is resumable and could handle 230s timeout with multiple runs.

**Rule:** Test cheapest → prove it fails → then upgrade. Never skip step 1.

## Extension (2026-04-26): Recommendation Order

This rule also governs the **order** in which fixes are presented:
- The FIRST option in any proposal MUST be zero-cost (or cheapest viable).
- Paid options come AFTER, with explicit cost called out.
- If only paid options exist, say so — don't bury cost in a footnote.

**Evidence (2026-04-26, E090):** When NAT-OPEN broke, Brain's first proposed fix was "allocate a new EIP (~$3.60/mo)" — completely missed that goodhealth.lv EC2 already had the right EIP and could act as instance NAT for $0. Justs: *"why the fuck do I need 2 IPs???? I dont want to fucking pay more!!!!!!!!!!!!!"*. Correct fix saved $32/mo.

The prompt-trigger: any AWS/infra recommendation. Before sending, check: "Have I listed a zero-cost option first?"
