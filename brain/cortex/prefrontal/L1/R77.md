# R77: Never Assume Undeployed From Local State

**Rule:** When the user says they deployed something to a cloud service (Azure Functions, AWS Lambda, Container Apps, etc.), VERIFY the cloud state directly before claiming it is not deployed. Local git status, untracked files, or missing commits tell you NOTHING about cloud deployment state.

**Why:** Brain almost derailed an entire session by claiming Azure Functions were "only local, never deployed" based on seeing untracked files in git status. User had deployed via `func azure functionapp publish` directly. If user had believed this, we would have wasted time re-deploying things that were already running.

**How to apply:**
1. ALWAYS check `az functionapp list`, `az functionapp function list`, or equivalent cloud CLI before claiming something is not deployed
2. Untracked files in git != undeployed. Direct CLI deploys dont touch git.
3. When user corrects with context like "we already tested Premium vs Consumption", that is evidence of deployment - trust it immediately
4. The correct diagnostic path is: check cloud state FIRST, then compare with local code for drift

**Severity:** Critical - wrong assumption here wastes entire sessions
