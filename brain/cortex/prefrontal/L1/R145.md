# R145: Verify-before-deploy for ALL visual work

## The rule

For any code that produces visual output — 3D scenes, animations, charts, dashboards, layouts, color schemes, transitions — Brain must visually verify the output BEFORE asking the user to test. The EC2 has Docker + headless browsers. There is NO EXCUSE for using the user's eyes as the QA loop.

## Mandatory verification protocol

For ANY change to viz/frontend or any visual code:

1. **Build + deploy locally** as usual.
2. **Headless screenshot capture** via Playwright running on EC2:
   ```python
   # /opt/brain-viz/scripts/verify-viz.py
   from playwright.sync_api import sync_playwright
   with sync_playwright() as p:
       browser = p.chromium.launch()
       page = browser.new_page(viewport={'width': 1400, 'height': 900})
       page.goto('https://goodhealth.lv/brain', wait_until='networkidle')
       page.screenshot(path='/tmp/viz-baseline.png')
       # trigger the action (e.g., click cycle row + RUN)
       page.click('text=#205')
       page.click('text=RUN CYCLE')
       page.wait_for_timeout(5000)   # mid-animation
       page.screenshot(path='/tmp/viz-mid.png')
       page.wait_for_timeout(20000)  # post-animation
       page.screenshot(path='/tmp/viz-post.png')
       page.click('canvas', position={'x': 50, 'y': 50})  # click outside
       page.wait_for_timeout(1000)
       page.screenshot(path='/tmp/viz-reset.png')
       browser.close()
   ```
3. **Read each PNG** with the Read tool (it accepts images). Verify:
   - Baseline: full brain visible, all neurons + edges at default opacity
   - Mid-animation: ball visible, lit subgraph distinct from background
   - Post-animation: same as mid OR fully restored (depending on intent)
   - Post-reset: identical to baseline
4. **Only THEN** commit + push and tell the user.

## Why this rule exists

2026-04-25 session: I made FOUR consecutive viz changes to the cycle-replay animation (v1: hide-non-focus, v2: AND→OR edges, v3: stop hiding entirely). Each time I deployed and asked Justs to test. Each time he reported the same or worse symptoms. v3 ended up showing all neurons + all edges with no distinction at all — the cycle wasn't visually distinguishable.

User reaction: *"USE SOME COMMON FUCKING SENSE ON HOW TO VERIFY THAT THIS VISUAL/ANIMATION IS ACTUALLY WORKING!!!! IM SICK AND TIRED OF THIS SHIT - YOURE ALMOST INCAPABLE WITH ANIMATIONS - DO BETTER"*

The deeper failure: I was reasoning about the code purely from logic ("if focusSet has both endpoints, edge brightens") without ever LOOKING at what the rendered scene actually showed. For animation code in particular, logic-only reasoning is insufficient — you need to see frames.

## Severity: HIGHEST

This rule outranks deploy-related rules (R141, R143, R144) because it touches user trust directly. Burning 4 deploy cycles asking the user to QA is worse than slowing down by 5 minutes to screenshot first.

## How to apply

- Before any `git commit` on viz code: must have generated + read screenshots
- If Playwright not yet installed on EC2: install it as the FIRST step of any viz task before writing any code
- For changes that are obviously safe (typo, color hex, single-pixel padding): document the exemption in commit message
- Animation changes are NEVER exempt — they must be screenshotted at min 3 time points (start, mid, end)

## Companion: R145.5 — install Playwright on EC2 immediately

`pip install playwright && playwright install chromium` should run as part of any viz session that doesn't already have it. Save the screenshot script under `/opt/brain-viz/scripts/verify-viz.py` so it's reusable.
