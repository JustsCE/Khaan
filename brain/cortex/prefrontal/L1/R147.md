### R147: og Brain — Self-bypassable flag-file escape hatches are forbidden
- **Strength:** 1
- **Source:** Justs directive 2026-04-25 ("can the escape hatches be used BY YOU to avoid the rules??")
- **Date:** 2026-04-25 (rewritten)
- **Rule:** No mechanical block rule may use a flag file that Brain can create or remove via Bash. Any "escape hatch" Brain can self-invoke is theater — not enforcement. Permitted bypass mechanisms:
  1. **Auto-expiry** — Brain cannot forge the system clock (e.g. RECALL_GATE 2h timeout).
  2. **Scheduler-state gate** — protected by SCHEDULER_FNAMES; Brain cannot write scheduler.json or cycle-manifest.json via Bash.
  3. **Narrow rule scope** — the rule allows the legitimate path natively (e.g. R146 allows explicit-service-list, no flag needed).
  4. **Server-side endpoint** with user out-of-band confirmation (future work).
- **Why:** I shipped R146 with a `.r146-allow-bare` flag in commit 0af4421a. User's question revealed it: I could `touch` the flag, run the bypassed command, `rm` the flag. The "mechanical" enforcement was performative. R007's premise that mechanical enforcement is qualitatively stronger than text doctrine collapses if the mechanism is itself self-modifiable.
- **How to apply:**
  1. When designing a rule: if the only bypass requires Brain to create/remove a file via Bash, reject the design and find a narrower rule scope.
  2. When auditing existing rules: every flag-file bypass is suspect. .brain-cycle-running and .action-train-broca-pending are still flag-based and should be migrated to scheduler-state gates over time.
  3. The block message must point to a NON-self-bypassable path (e.g. "use explicit service list" rather than "touch this flag").
- **Mechanical:** No — this is a meta-rule about how rules are written. Falsification is by inspection at the next add-rule invocation.
- **Replaces:** prior R147 (2026-04-25 v1) which permitted flag-file bypasses.
