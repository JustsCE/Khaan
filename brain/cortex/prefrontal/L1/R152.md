# R152: Verify Host Before Proposing SSH

**Rule:** When about to suggest "SSH into goodhealth.lv" (or any remote host) or include `ssh -i ...` in a plan, FIRST run `hostname && curl -s http://169.254.169.254/latest/meta-data/instance-id`. If output matches the target machine, you ARE there — drop the SSH and run commands locally. Companion procedure: PR149.

**Why:** 2026-04-26, Justs corrected this 9 times in one conversation before demoting Brain INTERN → HIGHSCHOOLER. The brain runs as a systemd service on goodhealth.lv (per PR076). Every Bash tool call executes ON the EC2. Proposing SSH into the machine you're already on is the strongest possible signal of self-unawareness.

**How to apply:**
1. Trigger: any plan/proposal containing the word "SSH", "ssh -i", or "into the EC2"
2. Before writing the plan, run: `hostname && curl -s http://169.254.169.254/latest/meta-data/instance-id`
3. If hostname starts with `ip-172-31-` AND metadata returns `i-09106d98eab1f2f12` → you are goodhealth.lv → drop SSH, run locally
4. If different host → SSH is appropriate, proceed
5. NEVER skip the verification because "I'm pretty sure"

**Evidence:** 2026-04-26 conversation. NAT-OPEN debug → NORDE recovery. Brain proposed SSH 4 times across the conversation before Justs's 9th correction. Mechanical check costs <1s; the alternative cost is trust.

**Refs:** PR149 (the procedure), E090 (the recovery this rule was learned from), PR076 (EC2 setup that established brain-on-EC2)

**Severity:** Critical for infrastructure work. INTERN → HIGHSCHOOLER demotion was the consequence.
