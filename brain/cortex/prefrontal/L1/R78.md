# R78: Stop Guessing at Runtime Errors — Reproduce Locally First

**Rule:** When debugging a frontend runtime crash (like Three.js/WebGL errors), do NOT guess at the cause and deploy multiple times hoping it works. Instead: reproduce locally with `npm run dev`, check the browser console, identify the exact line in unminified source, fix it, verify locally, THEN deploy once.

**Why:** Brain wasted 3 ACR builds (v8-dbsync-globe, v8-globe-fix, v9-no-drei) trying to fix a Three.js crash by guessing. Downgraded three.js (didn't help), removed drei (didn't help). Each ACR build takes 5+ minutes. Should have reproduced locally first and read the actual stack trace in dev mode.

**How to apply:**
1. For frontend runtime errors: `npm run dev` + open browser + check console = exact source line in seconds
2. Never deploy to prod to test a theory — test locally first
3. ACR builds are expensive (time + money) — treat each deploy as a last step, not a debugging step
4. Minified stack traces are nearly useless — always get the unminified version first

**Severity:** Critical — wasted 20+ minutes on 3 failed deploys
