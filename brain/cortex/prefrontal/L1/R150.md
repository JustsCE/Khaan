### R150: og Brain — Never Expand Scope When Porting/Migrating
- **Strength:** 1
- **Source:** [[[[[[[[[[[[[[[[[[[H-702 (resolved) (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (resolved)] (2026-04-26 user correction: "AI slop")
- **Date:** 2026-04-26
- **Rule:** When migrating from platform A to platform B, the new system has the SAME OPERATIONS as the old. Match 1:1. Do NOT add "while we are here" features, "future-proofing" actions, or convenience bundling. Improvements come AFTER migration is verified working.
- **Why:** During Lambda migration, I added 6 actions (restart-viz, disable-hooks, rotate-bypass-secret, etc.) when the existing GH workflows had only 4 (deploy-viz, ec2-recovery, brain-webhook, issue-bypass-token). User reaction (verbatim): "Why the fuck did you add 6 lambdas when git workflow had only 3 workflows+ the newone for Brain Bypass token?!?!?!? Brain youre overengineering simply shit. NEVER AND I MEAN NEVER OVERENGINEER WITH AI SLOP." Pattern: scope creep dressed as helpfulness.
- **How to apply:** Before any migration/port, count the existing operations. The new artifact must have the same count, the same names where possible, and zero "bonus" actions. If a new action seems obviously useful, write it down as a follow-up — do NOT bundle it into the migration. The migration is verified-working before any additions are even considered.
