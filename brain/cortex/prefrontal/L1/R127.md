# R127: Check git tracking before debugging app code

**Rule:** When files mysteriously reappear after deletion in a Docker container that runs `git reset --hard`, FIRST check if the files are tracked in git (`git show HEAD:<path>`). Only after ruling out git should you debug the application's write paths.

**Evidence:** 2026-04-21 — Spent 45 minutes adding debug logging, stack traces, sed edits, and process inspection to the Node.js backend, when the actual cause was `git reset --hard` restoring files from HEAD. One `git show HEAD:dashboard/tasks/slack.json` command would have found it in 10 seconds.

**How to apply:** When debugging "files keep coming back" in any git-synced environment:
1. `git show HEAD:<path>` — is it in git?
2. `git log --all -- <path>` — was it ever in git?
3. Only THEN look at application code.
