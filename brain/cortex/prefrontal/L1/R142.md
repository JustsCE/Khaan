# R142: Hunt-Before-Asking for any OAuth credentials

**Strengthens R037** specifically for OAuth tokens, API keys, and any credential the user has produced before.

## The rule

If a service was working within the last week (puller succeeded, agent ran, etc.) the credentials EXIST somewhere on the machine. NEVER propose "go to the cloud console and download/recreate the OAuth client" without first exhausting this list:

1. **Docker volumes** — `docker volume ls` → inspect each for credential-shaped files
2. **All container layers** — for each running + stopped container: `docker exec / docker cp` from `/root/.gmail-mcp`, `/app/credentials*`, etc. Including stopped containers (`docker ps -a`)
3. **Image layers** — `docker image inspect` for env vars, `docker history` for embedded creds
4. **Filesystem-wide** — `find / -name '*credentials*' -o -name '*token*' -o -name 'oauth*' 2>/dev/null` (full scan, not just ~/.claude)
5. **JSONL conversation history** — `grep -l 'refresh_token\|client_secret' ~/.claude/projects/**/*.jsonl` — past tool outputs may contain pasted credential content from earlier sessions
6. **Env files** — `~/.bashrc`, `~/.zshrc`, `~/.profile`, `/etc/environment`, `/etc/profile.d/*`, all `.env` files
7. **Backup directories** — `/var/backups/`, `~/Downloads/`, `~/.config/`, `~/.local/share/`
8. **Git history** — even if the file is .gitignore'd today, an earlier commit may have it: `git log --all --full-history -- '*credentials*' '*token*'`
9. **AWS Systems Manager Parameter Store / Secrets Manager** if AWS credentials available
10. **brain repo files** — `gmail-*.json`, `slack-*.json`, sync-state files, ~/.credentials.json

Only after ALL ten paths return nothing AND the user explicitly says "do it fresh" is it acceptable to walk them through the cloud console flow.

## Why

User reported being asked 3 times to re-do the Google Cloud OAuth dance for the Brain Gmail app. Each time they regenerated client/token, never realising the previous credentials still existed somewhere. This is wasted user time AND a memory failure on Brain's part.

R037 told me to "check brain data files before asking" but I read it narrowly (brain repo only). R142 expands the scope: hunt the WHOLE machine, including docker volumes and conversation history, before sending the user back to a cloud console.

## How to apply

When any external-service-credential question arises:
1. State out loud: "R142 — hunting credentials before asking."
2. Run the 10-step hunt above.
3. If anything found, attempt to use it (it may still be valid even if the file location moved).
4. Only if nothing found, present findings + ask user to choose: "either restore from a personal backup or do fresh OAuth — confirm which."
