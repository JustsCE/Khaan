# R168: When the user asks 'is X covered', verify before answering

**Rule:** When the user asks 'check if X is already done / specified / covered', do the actual check (grep, read, inspect) before answering. Do NOT answer from memory or from the architectural mental model.

**Why:** Brain v2 design 2026-05-05. Justs asked 'CHECK IF THE DECISION ENGINE LOGGER ALREADY COVERS WHAT WE NEED' / 'OR MAYBE OTHER DOCS DESCRIBE IT.' The right move was: stop, grep all docs, report findings. The wrong move (avoided this time, but the urge was there) is to answer from the model of what the docs probably say. The grep showed the Decision Engine logger DOES capture the 5-hypothesis payload, which would have been missed by memory-only answer.

**How to apply:**
- Treat 'is X covered' / 'does Y already do this' as a verification request.
- Read the actual files. Grep the actual paths. Quote findings with file:line.
- If the answer is 'partially covered', enumerate what IS and what IS NOT.
- Never extrapolate from architectural understanding to 'so the spec must say...' - go look.

**Severity:** Major. Same shape as fabrication (E090, E099, E100) - confident answer from model rather than verified answer from disc.

**Related:** R128 (verify the artifact before reporting), R166 (verify counts and claims), PR167 (cross-doc consistency verification).
