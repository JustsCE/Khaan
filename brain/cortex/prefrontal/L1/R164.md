# R164: Engine docs describe their own engine only

**Rule:** When writing a JustsCE/Brain engine doc (rule-engine.html, recall-engine.html, decision-engine.html, learning-engine.html, identity-engine.html, etc.), describe ONLY that engine. Do NOT do any of the following:

1. **Do not reference other engines by name** outside the strictly necessary minimum (the inputs table when reading another engine's published artifact, and the binary readers note when applicable).
2. **Do not describe what this engine ISN'T.** No carve-out paragraphs ("X stays here; Y goes elsewhere"). No "moved out" tables. No "this is what's NOT in scope" prose beyond a final "Out of scope" bullet list.
3. **Do not reference older designs.** No "older designs bundled X", no "twelve-phase pipeline", no "this engine carves them apart", no "older versions did Y", no "v1/v2/v3/v4" mentions.
4. **Do not fabricate architectural decisions for engines that don't exist yet.** If the Identity Engine isn't built, do not declare what "goes to" it. State the Learning Engine's behavior. Period.
5. **Do not use number-words for variable counts.** No "Six phases", "Five lenses", "Three protections" — the count may change. Use "Phases" / "the listed phases" / "the protections below".
6. **Do not compare this engine to other engines.** No "Unlike the Decision and Recall engines, this one...". No "the per-token brainpower demand is lower than Decision's...". State this engine's properties as absolute, not relative.

**Why:** 2026-05-03 — Justs called out the same failure pattern for the 5th time in one session ("WHY THE FUCK ARE YOU REFERRING TO OLDER DESIGNS THIS IS THE 5TH TIME I HAVE SAID DON'T REFER TO OLD DESIGN"). The Learning Engine doc had: count words ("Six phases"), other-engine references ("Unlike the Decision and Recall engines"), carve-outs ("Cortex management stays here; emotional state, self-model drift..."), older-design narration ("Older designs bundled cortex maintenance..."), and fabrications about Identity Engine that doesn't exist yet. Recurring failure across rule-engine.html, recall-engine.html, decision-engine.html, learning-engine.html.

**How to apply:**
- Before writing any prose in an engine doc, ask: "Am I describing THIS engine, or am I describing the system around it?" If the latter — delete.
- Pre-flight grep before commit: forbidden tokens are `Six|Five|Four|Three|Two`, `Older design|legacy|v[1-9]`, `Unlike|owned by other engines|live with the engines|do not run here|stays here`, every other engine's name except in the Inputs table and (where applicable) the Binaries readers note.
- If a rationale truly requires comparison or historical context, put it in the plan file or commit message, NOT in the engine doc body. The doc is a spec, not a design history.
- The Out-of-scope section, if used, lists at most 3-4 bullets of bare items. No prose, no "moved to X", no "owned by Y".

**Severity:** Critical. User has explicitly flagged this as a 5-time failure pattern. The next occurrence loses trust further.
