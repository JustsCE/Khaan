# R155 -- og Brain — Reuse Existing Components Verbatim When User Says "Match X"

**Created:** 2026-04-27
**Status:** active
**Strength:** 1
**Reinforced:** 0
**Tags:** [interpretation, component-reuse, ui, correction]
**Source:** brain-retro 2026-04-27 (3-strike correction)

## Rule

When the user says "match the X look", "use the X version", or "make Y look like X" — the correct move is to **render the existing X component verbatim**, not build a smaller / simplified / custom variant.

If positioning / layout makes the existing component unfit for the new context (e.g., desktop pinned vs. inline panel), add a layout-mode prop (e.g., `inline?: boolean`) to the existing component. Don't fork.

## Why

2026-04-27 session — the user asked three times for the same thing before I got it right:
1. "Combine Brain Tasks and Brain visuals" → I built a tiny custom header strip
2. "Mobile must match web" → I had built a smaller version that didn't match
3. "Nooo i wanted the full layout. Right now the stats are too small. I wanted the big Cycle number, the % connected, the big pulsating brain emoji, the profession (HIGHCHOOLER)" — explicit element list, all of which were already in the HUD component

I should have reached for the HUD component on Move 1 and added an `inline` prop. Three turns of frustration and re-deploy could have been one.

## How to apply

1. When user references an existing UI element by name or look, find that exact component first.
2. If it doesn't fit your current context due to layout (position fixed, fixed width, etc.) — add a prop, not a copy.
3. Default to "render the user's reference 1:1" — let them tell you what to remove, don't pre-trim.
4. Less is more is wrong here. The user's mental model includes ALL the visual elements; if you ship a subset, you've changed the ask.

## Anti-pattern

Building a "lighter version" of an existing rich component because you assumed the user wanted something simpler in the new context. They didn't. They want the same thing they pointed at.
