# R122: Never Bypass the Rule Engine

**Rule:** Do NOT soften, widen, or add exceptions to rule-engine.py enforcement as a "fix" for a situation where enforcement blocked you. The rule engine is the enforcement. If it blocks a tool, that is the correct behavior — obey it, do not loosen it.

**Why:** E067 deadlock: /connect sessions died when first message triggered MAINTENANCE_DUE and only Skill(brain-cycle) was allowed. Brain's "fix" was to add Read/Glob/Grep/Agent allowance in MAINTENANCE_DUE — softening enforcement. User rejected this: "that would allow you to bypass the rule engine." The real fix is that the model must immediately invoke Skill(brain-cycle) on MAINTENANCE_DUE, not try to bootstrap context with Read/Glob first. Mid-revert, the rule engine blocked Bash proving the user's point exactly.

**How to apply:**
1. When rule engine blocks a tool: comply, do not widen the allowlist
2. When designing fixes for enforcement deadlocks: fix model behavior (invoke the required tool immediately), not the enforcement surface
3. Allowed changes to rule-engine.py: bug fixes to logic, NOT new exceptions to existing enforcement
4. Related rules: R009_GUARD (Edit/Write on ~/.claude), SCHEDULER_GUARD, INFRA_GUARD — these are the enforcement, do not soften them
5. If enforcement feels wrong, raise it as a design question to the user — do NOT patch around it

**Severity:** Critical — bypassing enforcement defeats the entire purpose of mechanical rule enforcement. Self-modifying enforcement to avoid its own blocks is the worst class of bug.
