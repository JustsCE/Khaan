# S062 -- og Brain — Fine-grained GitHub PAT permission mapping

When working with fine-grained Personal Access Tokens (github.com/settings/tokens?type=beta), permission names differ from classic PATs:

| Classic scope | Fine-grained permission | What it enables |
|---|---|---|
| `repo` | **Contents: Read and write** | git push / pull on repo files |
| `workflow` | **Workflows: Read and write** | modify `.github/workflows/*.yml` via push |
| (part of `repo`) | **Actions: Read and write** | trigger/cancel workflow RUNS, read artifacts |
| `repo` | **Metadata: Read-only** (required, auto-added) | list repo info |
| n/a | **Secrets: Read and write** | manage Actions secrets |

**Critical distinction: Actions ≠ Workflows.**
- `Actions` = operate the workflow runs that already exist (trigger, cancel, read logs/artifacts)
- `Workflows` = modify the workflow *files* themselves (push yml changes)

These are separate permissions specifically because GitHub treats workflow file modification as security-sensitive.

**Scope changes vs regenerate:**
- Clicking "Update" after adding a permission does NOT rotate the token value. Secret stays valid, just with wider scope.
- Only "Regenerate token" rotates. If rotated, update every consumer (GitHub repo secrets, ~/.netrc, CI env vars).

- Permissions as of 2026-04-21: Actions r/w, Contents r/w, Metadata r, Secrets r/w, Workflows r/w
- Access: All repositories
