# Vericity transactions pipeline architecture and data lineage

The Vericity transactions pipeline operates entirely on flat-file storage rather than a relational database. Its source data consists of four CSV files spanning 2022–2025, totaling approximately 38 MB, hosted in S3. No `dbo.transactions` table exists in any connected database; the CSVs themselves are the authoritative data store. The schema carries rich transactional detail—transaction date, price, apartment area, room count, room address, cadastre numbers, and building-level metadata—but provides only string-form addresses with no embedded geocoordinates.

Geocoding is handled downstream via Nominatim, with deduplication applied at the building level to avoid redundant lookup calls against the same physical structure. The pipeline's output is a consolidated `Transactions.xlsx` file produced by a Lambda function, with the backend implementation located at `~/vericity-transactions/`. This architecture means that any schema evolution, data repair, or enrichment must happen either at the CSV layer in S3 or within the pipeline's transformation logic itself—there is no intermediate database to query or patch independently.
