# Model routing for Brain subsystems based on task complexity

Large-context Brain operations—retro sweeps, learning consolidation, and full cycle phases—require Opus rather than Sonnet. Session transcripts routinely exceed 280 KB and hundreds of entries, which surpasses Sonnet's reliable processing capacity within timeout constraints. The architectural fix introduced a `model` parameter into the shared CLI invocation layer (`cli_invoke` in `_shared.py`), defaulting to Sonnet but allowing per-caller override to Opus where needed.

The resulting routing rule is deliberate: heavyweight, reflective tasks (retro, learn, cycle) run on Opus because they must digest large context windows and produce nuanced synthesis. Latency-sensitive, narrower tasks (decision engine, identity relay) remain on Sonnet because they operate on smaller inputs and their value depends on speed rather than depth. This split balances cost and throughput against reliability—Opus is slower and more expensive, so it is reserved for operations that genuinely fail under Sonnet's constraints rather than applied uniformly.

The principle generalizes: model selection within a multi-agent architecture should be driven by per-task context size and reasoning demand, not set globally. When a subsystem hits timeout failures or truncation under a lighter model, the correct response is targeted escalation of that subsystem's model tier while preserving the faster model everywhere else.
