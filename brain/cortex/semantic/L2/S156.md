# Brain retro processing limits on large session transcripts

The brain retro engine encounters a hard timeout boundary when session transcripts exceed approximately 50KB in size. A full session transcript—on the order of 283KB with roughly 298 entries—surpasses Sonnet's 300-second processing window, causing the retro phase to fail silently or abort. This is a structural constraint of the model's throughput relative to the timeout, not a bug in the retro logic itself.

Two complementary mitigations address the problem. First, input truncation: capping the transcript to the last 30–50 entries, with each entry further trimmed to 500 characters, reduces the payload to a size Sonnet can handle within the timeout. Second, model escalation: routing the retro engine to Opus, which has greater processing capacity for large context windows, eliminates the timeout for full-length transcripts without requiring truncation. Both approaches were validated through successive fixes (FIX5 through FIX7), with truncation serving as the immediate stopgap and Opus migration as the durable solution.

The underlying principle is that any consolidation phase operating on unbounded session history must either bound its input or use a model whose throughput matches the worst-case transcript size. Retro is particularly vulnerable because it runs at session end, when transcripts are at their longest.
