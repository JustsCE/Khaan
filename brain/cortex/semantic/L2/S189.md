# Sonnet context-window throughput limits for session retrospectives

The `brain_retro.py` subsystem fails under Sonnet when a full session transcript grows beyond roughly 283 KB (approximately 298 entries). At that payload size, Sonnet cannot complete processing within the 300-second timeout, causing the retrospective to silently fail or hang. Two complementary mitigations have proven effective: routing oversized transcripts to Opus, which handles the larger context comfortably, and truncating the input to the most recent 30–50 entries as a defensive fallback before invoking the model at all. Truncation to the tail of the session is acceptable because retrospective value concentrates in recent interactions — earlier entries have typically already been consolidated by prior cycles.

The practical threshold is therefore a dual gate: if the transcript is under ~280 KB, Sonnet processes it within budget; above that line, the pipeline must either shed input or escalate to a higher-capacity model. This is not a Sonnet defect but a straightforward throughput ceiling — the model's per-token latency multiplied by a 298-entry transcript simply exceeds the fixed timeout window. Any future increase to the timeout or decrease in per-token latency shifts the threshold but does not eliminate the need for a size check, because session length is unbounded while timeout budgets are not.
