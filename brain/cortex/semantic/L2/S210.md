# MCP Service Topology on the EC2 Host

The infrastructure running on goodhealth.lv follows a uniform pattern for exposing MCP servers: each service runs as a Docker container on a dedicated port, fronted by an nginx reverse proxy that maps a route to the container, and registered via a corresponding entry in settings.json. Gmail occupies port 3005, WhatsApp port 3007, and both are reachable through nginx at predictable `/mcp/<service>/mcp` paths. This container-plus-proxy-plus-config triple is the canonical deployment unit for every MCP integration on the EC2 instance.

Not every MCP that can be deployed needs to be. The inventory distinguishes between working, necessary services—those that actively serve agentic functions like messaging and email—and unnecessary ones that were explored but found redundant. The integration lifecycle itself follows a phased pattern modeled after earlier Slack and Fireflies work: connect the MCP endpoint, validate it manually, then formalize its use. Gmail, for instance, exposes nineteen tools through this pipeline and reached operational status by following exactly that sequence.

The key architectural fact is that adding a new MCP service is a solved, repeatable process—container, port, nginx route, settings entry—rather than a bespoke integration each time. The discriminating decision is not how to deploy but whether a given service warrants deployment at all.
