# E040: Vericity Pipeline Migration Session

**Date:** 2026-04-10
**Duration:** Full day session
**Emotional tag:** frustrating start (30 messages to find Lambdas), smooth finish

## What happened
1. User asked to read Lambda scraping functions from personal AWS
2. Brain confused personal AWS with Picanova 5+ times, took 30 messages to find Lambdas
3. Discovered Teika sell district was missing from URL scraper (root cause of bdpbdb.html not being in DB)
4. Built combined ss_url_pipeline script with dynamic page detection
5. Deployed to AWS Lambda first (worked perfectly, 4042 URLs)
6. Migrated to Azure Functions -- hit multiple issues:
   - Flex Consumption same 230s limit as regular
   - ContentSettings dict vs object bug
   - UTF-8 streaming chunk boundary bug
   - Timer trigger timeout at 230s
7. Fixed all issues, deployed 3 functions (url-pipeline, data-scrape, images-invoke)
8. Data scrape: resumable with 200s time budget, 6 runs to process 2056 URLs
9. Images invoke: merged 34,238 rows in 9 seconds
10. Started 287K image migration from S3 to Azure Blob (24.2 GB)

## Lessons
- Always ask "which AWS account?" immediately
- Azure Consumption = 230s max regardless of settings
- Streaming blob reads need UTF-8 boundary handling
- Time-budget + resume pattern works well for long-running serverless
- SS.com rate limits after ~250 rapid requests -- use "all" pages (136 requests vs 427)
