# E059: Third Hook Lockout — Cross-Platform Fix Cascade

## What Happened
2026-04-18: User locked out of Claude Code for the third time due to broken hooks. Each "fix" introduced a new failure mode:

1. **`|| true` attempt:** Added to suppress errors on Linux where Windows paths don't exist. Side effect: ALL exit codes become 0, including exit 2 (enforcement blocks). Rule engine silently stopped enforcing anything. System appeared functional but was completely unenforced.

2. **`if [ -f ]; then` attempt:** Wrote command as `bash if [ -f "..."]; then bash ...; fi`. The `bash` prefix was doubled (hook runner already does `bash -c`), creating `bash if [...]` which is a syntax error. Every single message triggered the error, blocking all interaction.

3. **Hardcoded paths:** Windows paths (`C:/Users/...`) fail on Linux, Linux paths (`/home/ubuntu/...`) fail on Windows. Settings.json is git-synced between both.

User had to manually clear `"hooks": {}` from PowerShell to regain access.

## Root Cause
No abstraction layer between settings.json hook commands and platform-specific paths. No distinction between "hook crashed" and "hook intentionally blocked."

## Resolution
Created hook.py wrapper with 3-layer safety: Python expanduser for paths, try/except for crashes, bash exit-code filter. Only exit 2 passes through. Everything else → exit 0.

## Lesson
Hook modifications are HIGH RISK. The blast radius is total — a broken hook bricks the entire session. Always test before deploying, never use `|| true` on enforcement hooks, and design for crash-safety first.
