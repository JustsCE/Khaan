# E071 -- og Brain — Sentinel lockout root cause: .agent-mode not in .gitignore

## What happened
The .agent-mode sentinel file kept causing lockouts across sessions despite multiple fixes (E068 sentinel cleanup in prompt, auto-clear on stale). The real root cause: .agent-mode was never in .gitignore.

## The chain
1. Agent writes sentinel -> runs git add -A -> sentinel committed to branch
2. Agent calls merge_to_main -> sentinel lands on main
3. EC2 webhook pulls main -> sentinel on EC2 disk
4. Next /connect session -> hook sees sentinel -> LOCKED OUT

## The fix (2026-04-21)
1. Added brain/.agent-mode to .gitignore -- can NEVER enter git history
2. Changed stale timeout from hardcoded 600s to AGENT_BUDGET + 300s
3. Agent budget increased from 900s to 1800s

## Why previous fixes failed
- E068 fix: added rm to prompt Step 9 -- but agent ran out of budget before reaching Step 9
- Auto-clear fix: 600s timeout was shorter than 900s budget, so it cleared DURING the agent run
- Both were symptoms. The disease was that the sentinel was committable at all.

## Lesson
Ephemeral state files must be in .gitignore. If a file should not persist across sessions, it should be impossible to commit. Do not rely on cleanup steps in prompts -- prompts can be interrupted.
