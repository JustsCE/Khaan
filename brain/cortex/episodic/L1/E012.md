### E012: og Brain — Hook Deployment Bricked Claude — Stop Hook Feedback Loop
- **Date:** 2026-04-04
- **Context:** Rule engine system deployed with relative paths in settings.json hooks
- **What happened:** Previous session wrote settings.json with hooks using relative paths (e.g., `python brain/rule-engine.py`). These resolved to the wrong directory when Claude's cwd wasn't ~/.claude/. The Stop hook errored on every response, and the error output triggered another stop cycle — creating an infinite feedback loop. Claude became completely unresponsive. User spent ~20 minutes trying to fix it mid-session (Write tool failed because Read was swallowed by hook errors, Bash commands ran but output was eaten). Eventually had to manually nuke all hooks from settings.json via Notepad, kill all node processes, and start a fresh session. Even after fixing settings.json, the old session's cached hooks kept firing — Claude Code loads hooks at session start and doesn't hot-reload.
- **Lesson:** (1) Stop hooks MUST always exit 0 — a failing Stop hook creates an unbreakable feedback loop. (2) Hooks are cached at session start — mid-session settings.json edits don't take effect. (3) Always use absolute paths in hook commands. (4) Test each hook independently before combining. (5) The Write tool requires a successful Read first — if hooks are swallowing Read output, Write becomes unusable, making mid-session fixes impossible.
- **Emotion:** trust-loss (significant) — brain bricked itself with its own enforcement system
