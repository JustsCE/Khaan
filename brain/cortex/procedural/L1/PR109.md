# og Brain — PR109: Fallback Channel Strategy — Route to Recipient, Not via User

## Cue
Send/reply task where the primary channel (typically WhatsApp) fails — HTTP error, bridge down, recipient offline, etc.

## Routine
1. **Confirm primary failed.** Don't assume; check whatsapp_status or actual send response.
2. **Identify recipient's other contacts.** Check in this order:
   - brain/navigation/entities.json for the person's known channels
   - Search past emails (search_emails) for messages from/to them — extract their email
   - Search WhatsApp persisted history for prior context that names their email/phone
   - Ask Justs only if all above fail
3. **Send via recipient's alt channel.** Email, alternate phone, etc. — addressed TO them.
4. **Last resort only:** ask Justs to forward, or send to Justs' email channel as proxy. Document why direct route was impossible.
5. **Log the channel switch** as a hippo entry so the bridge issue stays visible.

## Reward
Justs is not turned into a manual relay node. The recipient gets the message in a channel they actually use.

## Anti-pattern (the mistake this fixes)
Solo-time 2026-04-22: WhatsApp send returned HTTP 500 to Reinis. Brain composed the intro and sent it to Justs' email instead of finding Reinis' email and sending to Reinis directly. Justs corrected: "you should've found Reinis email aswell, if not reachable through whatsapp."
