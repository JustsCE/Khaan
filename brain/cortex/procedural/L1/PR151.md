# og Brain — PR151: Diagnose Failing Agent Triggers via env_id Pattern

**Cue:** Scheduled agent triggers fire (per `updated_at` advancing) but produce no commits, OR get stuck "allocating cloud environment" in the Anthropic Routines UI.

**Routine:**

### Step 1 — Check trigger metadata for fire confirmation
```python
RemoteTrigger(action="get", trigger_id="trig_...")
```
Look at:
- `updated_at` — if advanced past your last edit time, the trigger DID fire (env was allocated, Claude session ran)
- `next_run_at` — when the next fire is expected

Two failure modes are distinguishable:
- **`updated_at` advanced + no commit** → env allocated, agent ran, but produced nothing (SETUP/hooks/git-push issue)
- **`updated_at` frozen at last edit** → env never allocated (platform issue OR poisoned env_id)

### Step 2 — If `updated_at` is frozen, check the Routines UI
Ask the user (or check yourself if accessible) what the trigger shows in the panel:
- "3 dots / allocating cloud environment" → poisoned env_id
- "Paused" → trigger disabled (rare if API says enabled=true)
- Error state → platform issue, escalate to Anthropic

### Step 3 — Compare env_ids across triggers
List triggers and group by `job_config.ccr.environment_id`. Cross-reference with which triggers HAVE produced commits recently. Pattern observed 2026-04-27:

| env_id | Triggers using it | Worked? |
|---|---|---|
| `env_01Mcp8H8E6dpR8c3KMxbKE3Z` | Teacher, Personal Triage, Work Triage | YES — produced commits |
| `env_01Tqq7QdHg2gLYR97yz24P9j` | Solo Time, Self-Test | NO — fired without artifacts, then stuck allocating |

The poisoned env was Solo Time's original — likely a stale container/volume that Anthropic could not reset.

### Step 4 — Migrate triggers to working env via API
```python
RemoteTrigger(action="update", trigger_id="...", body={
    "job_config": {"ccr": {"environment_id": "<working_env_id>", "events": [...], "session_context": {...}}}
})
```
Note: must include the FULL `job_config` payload (events, session_context). Partial update of just `environment_id` is not supported.

### Step 5 — Verify migration
Re-`get` the trigger and confirm `job_config.ccr.environment_id` matches the working env. Wait for next natural fire (or set a soon-fire cron) and check artifacts.

**Prerequisites:**
- Auth: must have Anthropic OAuth token configured (RemoteTrigger MCP handles this)
- Knowledge of which env_ids are "working" — derived empirically by which triggers produced commits

**Verification:**
- Trigger metadata `updated_at` should advance after next fire (proves run happened)
- Origin/main should have a new commit from the agent within budget+5min
- If still no commit: the issue is NOT env-level; investigate hooks (PR135) or git push auth

**Reward:**
- Triggers fire reliably on the working env
- Bad env_ids identified and avoided in future trigger creation
