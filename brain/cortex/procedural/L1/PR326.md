# Clear every gate flag at every exit point in brain_cycle.py

When `brain_cycle.py` finishes — whether by success, timeout, three-strike failure, or empty-cycle loop exhaustion — it must clear **all** gate flags: `learning-cycle-overdue`, `learning-cycle-running`, `learning-cycle-failed`, `learning-cycle-timeout`, and `cycle-empty`. Clearing only the flag that triggered the cycle, or only the happy-path flags, leaves sticky bits that permanently deadlock the agent because the gate system blocks every tool until the cycle completes cleanly.

To prevent this class of bug, treat gate cleanup as an invariant of the function's contract, not a reward for success. Place a single cleanup block that fires unconditionally — analogous to a `finally` clause — so that no exit path can skip it. The principle: any process that raises a blocking gate is responsible for lowering it on **every** termination path, including the ones you don't expect to hit. If a new flag is added to the gate vocabulary later, add it to the same cleanup block in the same commit; partial lists rot into partial deadlocks.

The original defect (FIX9) was that `run_cycle()` returning `None` on timeout or failure skipped the clearing logic entirely. FIX10 extended the fix by ensuring the success path also swept stale flags from prior failed runs, not just the current trigger. Both fixes reinforce the same rule: exhaustive cleanup is not optional, and "works on the happy path" is not tested until every sad path has been verified.
