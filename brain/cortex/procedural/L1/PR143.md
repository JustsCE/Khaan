# PR143: Recover Failed Multi-Service Deploy via Bundle-Hash Diff

**Cue:** A prior session claimed a deploy succeeded, but the user reports the change isn't visible. Or a session ended mid-deploy. You don't trust whether prod is actually running the latest code.

**Routine:**
1. Get the hash of the JS bundle the live site is serving:
   ```bash
   curl -s https://<host>/ | grep -oE 'assets/[^"]+\.(js|css)' | sort -u
   ```
2. Get the running container's image timestamp:
   ```bash
   docker inspect <container> --format '{{.Image}} created={{.Created}}'
   ```
3. Compare the image creation time against `git log -1 --format=%cI <expected-commit>`. If image predates the commit, the prior deploy never completed.
4. Confirm the source on disk matches the commit:
   ```bash
   diff -rq <repo>/src <prod-build-dir>/src 2>&1 | head
   ```
5. **Service-isolated rebuild (R146):**
   ```bash
   cd <compose-dir> && docker compose build <service> && docker compose up -d <service>
   ```
   NEVER bare `docker compose build` or `docker compose up -d` — that nukes nginx + every MCP container too.
6. Verify swap landed:
   ```bash
   docker inspect <container> --format '{{.State.StartedAt}} {{.Image}}'
   curl -s https://<host>/ | grep -oE 'assets/[^"]+\.(js|css)' | sort -u
   ```
   Bundle hash should differ from step 1; container started timestamp should be seconds-old.

**Prerequisites:** Shell access to the EC2 host. R146 enforces service-isolation mechanically.

**Verification:** New bundle hash served by prod ≠ old hash. Other containers (nginx, MCPs) untouched (`Up X minutes` unchanged).

**Reward:** Prod runs the actually-committed code. No 4th outage.

**Concrete recovery this procedure was extracted from:**
- 2026-04-25, viz v4 (commit ff74a1ca). Prior session ran `docker compose build --no-cache` (no service name) which crashed mid-build, leaving prod on v3 image from 08:21 UTC. Bundle was `index-SAqsyqCy.js`. After service-isolated rebuild → `index-DDU8q4Yi.js` served. nginx + 7 MCP containers stayed up untouched.
