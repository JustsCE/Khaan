# og Brain — PR113: HTML → DOCX Conversion (use pandoc, avoid Drive detour)

## Cue
Need to deliver a Word-compatible document (DOCX) from HTML content — either to commit to a repo or paste into Word.

## Routine
1. Build HTML with full content (use string composition or Jinja-style tokens)
2. `pandoc input.html -o output.docx` — default settings handle Latvian diacritics fine
3. For tables requiring cell merging, colspan control, or custom borders: use **python-docx** directly instead of pandoc HTML → DOCX (pandoc doesn't preserve cell merges well)
4. Commit the DOCX directly to the target repo

## Gotchas
- `--reference-doc=<(echo)` fails with "not enough bytes" error; omit reference-doc for default output
- Pandoc strips CSS borders; if you need them, use python-docx with OxmlElement for `<w:tcBorders>`
- For Latvian text, ensure source file has UTF-8 BOM or explicit encoding — some editors lose diacritics
- Pandoc converts images but loses some chart element types; for matplotlib output, pre-rasterize to PNG

## Anti-pattern: Google Docs auto-conversion detour
When user said "create a Google Doc", I spent considerable time base64-encoding a 400 KB DOCX for upload via `mcp__claude_ai_Google_Drive__create_file`, inflating context by ~140K tokens. Then user pivoted to "just give me the pdf". **Rule:** when asked for a secondary format mid-pipeline, deliver the primary first (PDF, repo commit) then offer the alternate format as a follow-up — don't block on format conversion.

## Reward
User gets a pasteable artifact in the expected format within minutes, without context-bloating base64 round-trips.
