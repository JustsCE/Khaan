### PR012: og Brain — Adding Rules to the Mechanical Rule Engine
- **Stage:** conscious
- **Learned:** 2026-04-04
- **Executions:** 5
- **Strength:** 1
- **Cue:** User says "always do X", "never do Y", "enforce", or wants mechanical enforcement of a behavior
- **Routine:**
  1. Read `brain/rules-engine.json` for current rules
  2. Read `brain/rule-engine.py` for available checker types: counter_gap_exempt, flag, path_guard, path_suffix, pattern_and_staleness, file_line_count, tool_output_pattern, always
  3. Define rule: id, name, trigger (PreToolUse/PostToolUse/UserPromptSubmit), matcher (tool regex), action (block/inject), check type, params
  4. For blocking rules: NEVER use matcher .* — list specific tools. Always exempt Read + Skill to prevent deadlocks (R029).
  5. Write to rules-engine.json via Bash (python)
  6. Update brain-diagnostic expected_rules list
  7. Update ARCHITECTURE.md rules table
  8. Test: trigger the condition and verify the rule fires
- **Prerequisites:** rule-engine.py must support the checker type. If not, add the checker function + register in CHECKERS dict.
- **Verification:** Trigger the rule condition and confirm it blocks or injects correctly.
- **Reward:** Behavior mechanically enforced — cannot be forgotten even if context compacts.
- **Errors:** R007 with .* matcher deadlocked all tools (E013). CYCLE_QUALITY as separate rule was redundant — merged into R007.
