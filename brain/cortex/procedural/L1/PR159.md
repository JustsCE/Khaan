# PR159 — og Brain — Visual primitives: oversize-by-default, then dial back

## Procedure

When shipping a visual primitive (icon, badge, flag, indicator, animated overlay), the verification standard is **rendered visual fitness**, not "compiled artifact present." R128 (verify artifact before reporting) is necessary but not sufficient for UI.

Concrete rule on initial sizing: **default to oversized.** Whatever the smallest reasonable interpretation of the spec is, ship something 1.5–2x that. Underscale fails silently — the element just doesn't communicate.

## Steps before declaring "ui shipped"

1. Build the primitive at a generous size relative to its container.
2. Build the bundle. Curl + grep to confirm it deployed (R128).
3. **Open the actual rendered page** (or have Justs open it) and look at the element next to its actual neighbors. If it can't be confidently identified by glance at user-zoom, it's too small.
4. Iterate: dial down only if it visibly dominates. Most "looks tiny" feedback wouldn't have happened if step 3 had run.

## Anti-pattern

- Choosing a size from arithmetic on container dimensions ("row is 36px so flag is 22px") without ever loading the page. The arithmetic is right; the perception is off because of contrast, surrounding density, and visual weight of neighbors.
- Calling it shipped because the bundle contains the keyframes. Bundle correctness is upstream of visual correctness.

## Source incident

2026-04-28 morning: shipped silver verify-pennant at 28x22px next to a ~36px row in Brain pane. The flag was visually invisible. Justs corrected: "wayyy to small compared to the row." Bundle was correct, deploy was correct, the size was wrong.
