# og Brain — Gmail MCP Attachment Retrieval — Container Workaround

The Gmail MCP's `download_attachment` tool has a known volume-mount issue: passing `savePath=/downloads` sometimes fails with ENOENT. Fall back to omitting `savePath`, which saves the file to the container root.

## Symptom
"Failed to download attachment: ENOENT: no such file or directory, open '/downloads/FILE.docx'"

## Workaround
1. Call the tool without savePath — file goes to container's / (root):
   mcp__claude_ai_WhatsApp__download_attachment(messageId=..., attachmentId=..., filename="FILE.docx")
   Returns: "Saved to: /FILE.docx"
2. Copy out of container to host /tmp:
   docker cp brain-viz-mcp-gmail-1:/FILE.docx /tmp/FILE.docx

## Why it works
The Gmail MCP runs in container `brain-viz-mcp-gmail-1` on goodhealth.lv EC2. The container's root `/` is writable (root user). Volume mount `/home/ubuntu/.claude/downloads:/downloads` exists but occasionally fails to resolve — container-root fallback always works.

## When to use
- When savePath=/downloads returns ENOENT
- When ~/.claude/downloads/ is empty after a successful download_attachment report
- Any retrieval of Gmail attachments that must land on host filesystem for further processing

## Permanent fix (deferred)
The MCP container's volume-mount config should be debugged — possibly a docker-compose issue. Until then, this workaround is reliable.
