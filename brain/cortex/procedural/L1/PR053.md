# PR053: Vericity Azure Functions Pipeline Validation and Fix

**Cue:** User reports stale data on vericity.ai, or need to verify the scraping pipeline is working.

**Routine:**
1. Check Azure Blob for recent output:
   ```bash
   CONN="$(az functionapp config appsettings list --name vericity-ss-scraper --resource-group inudat --query "[?name=='SS_STORAGE_CONNECTION'].value" -o tsv)"
   az storage blob list --connection-string "$CONN" --container-name ss-listings --prefix "url_cache/" -o tsv --query "[].{name:name,modified:properties.lastModified}" | tail -5
   ```
2. Check App Insights for recent invocations:
   ```bash
   APPID=$(az monitor app-insights component show --app vericity-db-sync --resource-group inudat --query "appId" -o tsv)
   # Use REST API with temp file for query (Windows shell escaping issues)
   ```
3. Check Azure SQL data freshness:
   ```sql
   SELECT MAX(scraped_Date) FROM dbo.listings_ss;  -- raw data (should be today)
   SELECT MAX(scraped_Date) FROM dbo.listings_fact; -- enriched (rebuilt by db-sync)
   ```
   - Gotcha: listings_ss can be fresh while listings_fact is stale — means rebuild step isn't running
4. If db-sync deployed version is outdated, redeploy:
   ```bash
   cd scripts/azure-functions/db-sync && func azure functionapp publish vericity-db-sync --python
   ```
5. If listings_fact needs immediate rebuild, run SQL directly with the REBUILD_FACT_SQL from admin.py
   - Gotcha: Must run ENSURE_FACT_COLUMNS_SQL first to add missing columns

**Prerequisites:** Azure CLI logged in, INUDAT_SQL_PASSWORD env var set, SQL Server driver available locally (use "SQL Server" not "ODBC Driver 18" on Windows)

**Verification:**
- `SELECT COUNT(*), MAX(scraped_Date) FROM dbo.listings_fact` shows today's date
- `SELECT TOP 1 * FROM dbo.refresh_log ORDER BY refreshed_at DESC` shows recent entry

**Reward:** Pipeline healthy, fresh data on vericity.ai
