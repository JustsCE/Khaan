# PR150: First-Failure Date Discovery in AWS Lambda

## Cue
- Lambda has been failing "for a while"
- Need to know EXACT day it broke
- Suspect a config change or external event on a specific date
- Fixing requires confirming the cause-effect window

## Routine

### Step 1: List daily log streams (Lambda creates one per day)
```bash
python -m awscli logs describe-log-streams \
  --log-group-name /aws/lambda/<NAME> \
  --profile personal --region eu-north-1 \
  --order-by LastEventTime --descending --max-items 60 \
  --query "logStreams[].{Stream:logStreamName,Last:lastEventTimestamp}" \
  --output text
```

### Step 2: Pick a date range bracketing the suspected break

Use --log-stream-name-prefix to find streams for specific dates:
```bash
python -m awscli logs describe-log-streams \
  --log-group-name /aws/lambda/<NAME> \
  --log-stream-name-prefix "2026/03/3" \
  --profile personal --region eu-north-1 \
  --query "logStreams[].logStreamName" --output text
```

### Step 3: Read each stream's content (NB: must use --start-from-head AND --start-time 0)
```bash
python -m awscli logs get-log-events \
  --log-group-name /aws/lambda/<NAME> \
  --log-stream-name '2026/03/30/[$LATEST]<id>' \
  --profile personal --region eu-north-1 \
  --start-from-head --start-time 0 > /tmp/log.json 2>&1

python3 -c "import json; d=json.load(open('/tmp/log.json')); [print(e['message'][:250]) for e in d.get('events',[])[:10]]"
```

### Step 4: Binary-search around the boundary
Compare success markers (e.g. "Success", "Route added") vs failure markers (ERROR, traceback). Narrow until you find: last good day | first bad day.

## Gotchas

- **Quote the dollar sign in stream names:** `'2026/03/30/[$LATEST]xxx'` (single quotes). `[$LATEST]` is literal in stream names.
- **Without `--start-from-head --start-time 0`:** events array returns empty even though stream has data. Default behavior is to return tail-only with a recent time window.
- **Stream prefix searches:** `--log-stream-name-prefix "2026/03/"` returns alphabetical, not chronological — sort manually if needed.
- **Empty grep results:** if you grep for "ERROR" and get nothing, the format may use tab-separated multiline messages. Inspect raw output first.

## Verification
- Compare consecutive days' last log line:
  - Success: e.g. `Created NAT Gateway → NAT Gateway available → Route added`
  - Failure: e.g. `Created NAT Gateway → [ERROR] WaiterError ... terminal failure state`
- The boundary day where this flips IS the first-failure date.

## Reward
You can now correlate the break date with: deployments, IAM changes, EIP reassignments, AWS regional events, etc. — the exact root cause.

## Real-world use
2026-04-26 (E090): Walked NAT-OPEN streams from April 26 backwards to March 27. Found:
- March 30 ✅ "NAT Gateway available, Route added"
- March 31 ✅ same
- April 1 ❌ "WaiterError: terminal failure state"
- Boundary: April 1 (matched the missing S3 data exactly)

Confirmed cause: EIP `eipalloc-04e5d283862b56a1e` got reassigned to EC2 around April 1, breaking NAT-OPEN's hardcoded reference.
