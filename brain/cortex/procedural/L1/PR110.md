# og Brain — PR110: Verify the Artifact, Not the Log, Before Reporting Status

## Cue
Any question about whether a task was done, what's outstanding, or what state something is in. Examples: "did X happen?", "is Y done?", "what's pending?"

## Routine
1. **Identify the artifact** that would PROVE the task was done — a file on disk, a row in a DB, a commit in a repo, a sent message ID, an API response.
2. **Check that artifact directly** before answering. Concrete checks:
   - Repo state: `gh api repos/OWNER/REPO/contents` or `git log`
   - File on disk: `ls -la PATH`
   - WhatsApp/email send: search the persisted store for the actual message ID
   - DB record: query the table
3. **Only after the artifact is verified** report status. If the artifact disagrees with the log/intake/memory, trust the artifact.
4. If the artifact can't be checked from this environment, say so explicitly: "Logs say X; I can't verify the actual repo from here — do you want me to check via Y?"

## Reward
Justs doesn't have to correct status reports. Brain says "yes/no/unsure" with grounding, not confident-but-wrong claims sourced from stale logs.

## Anti-pattern (the failure this fixes)
2026-04-22: Justs asked what Robson stats was supposed to do. Brain read autonomy-log-2026-04-21.md and autonomy-log-2026-04-22.md, both said "skipped — repo not authorized," and confidently told Justs the work had never run. The autonomy logs were either outdated, written by a session that never tried, or the work happened in a path the logs don't capture. Either way: trusting the log over checking the repo was the bug.
