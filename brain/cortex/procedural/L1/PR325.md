# Clear every failure flag on every exit path to prevent self-inflicted deadlock

When a binary flag such as `learning-cycle-overdue`, `learning-cycle-failed`, or `learning-cycle-running` is set to `1` by a gating system, it persists across turns until explicitly zeroed. If the function that triggers the flag — in this case `run_cycle()` — exits without clearing it on every code path (success, failure, empty result, timeout, loop exhaustion), the flag stays stuck. A stuck flag cascades: the FSM enters `MAINTENANCE_FAILED`, the rule engine blocks all Bash and Edit calls, and the agent loses the ability to self-recover. The only escape becomes manual terminal intervention to reset the FSM state and zero the bins — a pattern that has recurred repeatedly, with escalation counts reaching into the hundreds.

To prevent this class of deadlock, follow two invariants. First, treat flag cleanup as a `finally` obligation: every function that sets or inherits a failure flag must clear it at *all* exit points, not just the happy path. If `run_cycle()` can return `None`, that path must still zero `learning-cycle-overdue` and siblings before returning. Second, the success path of any engine must clear *all* stale failure flags from prior runs, not just its own — because a previous failure's residue can block a subsequent success from ever executing. Design flag lifecycles so that no single stuck bit can lock the entire tool surface. If a state like `MAINTENANCE_FAILED` blocks all write tools, the agent is definitionally unable to repair itself, making the lockout permanent until a human intervenes.
