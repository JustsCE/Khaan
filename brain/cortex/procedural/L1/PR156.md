# PR156 -- og Brain — Check Redundancy After Adding New UI Surface

**Created:** 2026-04-27
**Status:** active
**Strength:** 1
**Reinforced:** 0
**Domain:** procedural (UX coherence)

## Rule

When you add a new visualization, panel, or surface that displays information ALREADY visible elsewhere in the UI: in the SAME pull request, scan for now-redundant copies and remove them.

Do NOT ask the user "want me to remove the duplicate?" — that's a round-trip the user shouldn't need to make. Detect, decide, act.

## Procedure

1. After implementing the addition, identify what data the new surface displays.
2. Grep / inspect the rest of the UI for the same data being rendered elsewhere.
3. For each duplicate found:
   - If the new surface is more discoverable / draggable / closeable / contextual → drop the older one.
   - If the older one provides genuinely different value (always-visible vs panel-gated, mobile-only vs cross-device) → keep both, but mention the choice in commit message.
4. Bundle the cleanup in the same commit as the feature.

## When to confirm with user (rare)

Only if BOTH copies have a defensible reason to coexist that you can't resolve from context. Then state your call as one line + ship: "Note: keeping both X and Y because [reason]; revert if you want only one." Don't make it a question.
