# og Brain — PR153: Safe Destructive Migration — copy, verify, delete (in that order)

**Cue:** About to perform any data migration that involves `mv`, `rm -rf`, or replacing source files after copying. Especially when shells/sudo/HEREDOCs are involved.

**Routine:**

### Step 1 — copy with explicit success check

Do not trust globs to expand correctly when the calling shell lacks read perms (e.g. `sudo cp $VOL/dir/*.json` — the glob expands as the calling user, not root). Either:
- Use `sudo bash -c 'cp ...*.json...'` so the entire chain runs under root
- Use `find ... -exec cp {} ...`
- List files explicitly

After cp/mv: capture exit code OR run a verification step.

### Step 2 — verify destination

For each source file, confirm it landed at the destination with the expected size:
```bash
for f in src/*.json; do
  base=$(basename "$f")
  src_size=$(stat -c%s "$f")
  dst_size=$(stat -c%s "dst/$base")
  [ "$src_size" = "$dst_size" ] || { echo "MISMATCH: $base"; exit 1; }
done
```

For ANY rm/mv on the source: the line right above it should be a fail-fast check that proves the destination is good.

### Step 3 — delete

Only after step 2 confirms.

### Anti-patterns to avoid

- `sudo cp $VOL/path/*.json dst/ && sudo rm -rf $VOL/path/` (chained without verify; glob may not expand)
- `sudo cp ... ; sudo rm -rf ...` (semicolon; rm runs even if cp failed)
- HEREDOC inside sudo where vars expand at the wrong scope
- "I'll recover from undelete if needed" — ext4 + Docker volumes have no live-undelete (fs cannot be unmounted)

**Verification:**
- Destination has expected file count + sizes
- Source still exists until verification passes
- Exit code propagation tested (no silent continue)
