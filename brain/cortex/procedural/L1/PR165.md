# PR165: Brain Engine Modularization Plan (leanout v4)

**Procedure:** When the engine is being refactored, the design philosophy is:

1. **Engine = DISTRIBUTOR + VERIFIER, NOT executor.** The engine routes events and verifies invariants. It does not run cycles, decay, or rebuilds itself — those live in their own modules called by skills.

2. **Module split.** Target: ~12 focused modules (runtime, scheduler, guards, recall, audits, ledger, cli, dispatcher, harness). Each module owns one concern.

3. **Bash carve-outs stay inline.** R009_GUARD style 3-line guards remain in the dispatcher; do not over-modularize tiny security checks.

4. **Externalize logs.** Logs live in a sibling `logger/` folder, not embedded in engine.

5. **Guard naming by intent.** Rename guards by what they protect, not what they block: USE_BASH, CYCLE_SECURITY, ENGINE_SECURITY, SERVER_SECURITY.

6. **Kill action_train.** Removed from runtime.

7. **SubagentStop captures land in hippocampus.** Processed at next cycle. Pattern consistency over convenience.

8. **Harness adapter layer.** ~200 lines of glue lets other LLMs plug in.

**Why:** The 1562-line orchestrator conflated routing, verification, execution, logging, and security. Splitting clarifies ownership and lets each piece be tested/replaced independently. Plan file at plans/leanout.md (gitignored).

**How to apply:** When touching engine code during the leanout, place the change in the module whose concern it owns. Do not add a new responsibility to an existing module without splitting it.
