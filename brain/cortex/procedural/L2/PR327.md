# Routing large session transcripts through Opus to prevent retro timeouts

When a session transcript exceeds approximately 100KB — the threshold discovered at 283KB with 298 entries — Sonnet reliably times out at the 300-second ceiling during retrospective processing. Chunking the transcript into smaller pieces does not solve the problem; a chunked attempt on Sonnet lost 62% of the data (13 of 21 chunks timed out), making the output unreliable. To process full retro sweeps on large sessions, switch the model to Opus via the `model` parameter in `cli_invoke`. Truncating to the last 30–50 entries is an acceptable fallback when Opus is unavailable, but it sacrifices coverage of earlier session events.

The principle is capacity-aware routing: match the model to the payload size. Sonnet remains the correct choice for latency-sensitive, smaller-context tasks — identity relay, decision engine — but any pipeline stage that must digest a full session transcript (retro, identity boot, cycle phases) should default to Opus. The implementation path is: add a `model` parameter to the CLI invocation layer so callers can override per-task, then set retro and related phases to request Opus explicitly. Do not attempt to rescue Sonnet performance on large transcripts through chunking; the failure rate is too high to trust.
