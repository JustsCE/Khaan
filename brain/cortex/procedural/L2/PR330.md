# Recovering the WhatsApp MCP Bridge After Silent Disconnects

To send a WhatsApp message through the Brain, first confirm the wa-bridge container is healthy by hitting `https://goodhealth.lv/wa-bridge` (or the relevant health endpoint). A 500 response means the Baileys session inside the container has silently dropped — a condition that survives indefinitely without intervention because the bridge does not auto-reconnect after a container restart or an idle timeout.

When the bridge returns 500 errors, restart the wa-bridge container (`docker restart wa-bridge` or the equivalent compose command). This forces Baileys to re-establish its WhatsApp Web session. Do not attempt to debug the internal state; a clean restart is the proven recovery path. The disconnect can recur multiple times in a single working session, so treat any unexpected 500 from the bridge as a disconnect signal rather than an application bug.

The broader principle: MCP bridges that wrap persistent-session protocols (WhatsApp Web, in this case) are fragile at the session layer. Always gate outbound message workflows on an explicit health check, and default to a container restart as the first remediation step before investigating deeper causes.
