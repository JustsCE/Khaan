# Hardening Sub-Model Prompts Against Conversational Drift

When a smaller model (e.g., Sonnet) receives the user's raw message inside a relay or composition pipeline, its default instinct is to *answer* that message conversationally rather than perform its assigned structured task. This failure mode is not occasional — it is the dominant behavior unless the system prompt contains explicit, redundant countermeasures. A single instruction like "return JSON" is insufficient; the model must be told, in several overlapping ways, that it is **not** in a conversation: "You are NOT having a conversation with the user. Do NOT answer or respond to the user's message. Your ONLY job is to [structured task]. Return ONLY valid JSON."

To guard any sub-model relay against conversational drift, stack three constraints in the system prompt: (1) a role anchor stating the model is a composition engine, not a chatbot; (2) an explicit prohibition on answering the user's message; and (3) a format lock demanding only the target schema. All three are necessary — removing any one allows the model to slip back into conversational mode. The principle is that user-message content inside a pipeline prompt acts as an adversarial injection against structured output; the defense is prompt-level immunization, not post-hoc parsing. Additionally, ensure that pipeline state (hypothesis bins, navigation files) is consumed and cleared within the same turn to prevent stale context from compounding the drift.
