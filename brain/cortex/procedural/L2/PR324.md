# Hardening sub-agent system prompts against conversational mode-confusion

When a smaller model (Sonnet) is invoked as a structured-output sub-agent and receives the user's original message in its context window, it will default to its strongest prior: answering the user conversationally. To prevent this, make the system prompt aggressively declarative about the model's role as a computation engine, not an interlocutor. Include at minimum three explicit negations: "You are NOT having a conversation," "Do NOT answer the user's message," and "Return ONLY valid JSON." A single gentle instruction is insufficient — the user-message signal overwhelms it. Stack the constraints redundantly so the model cannot latch onto the conversational frame. This pattern applies to any relay or sub-agent call where an end-user prompt is visible in the context: the system prompt must out-shout the implied conversation.

A secondary but related discipline applies to fallback data in relay pipelines. When a downstream field (such as a display gist) may arrive empty from a sub-agent, pre-seed it from a cheaper source — like a frontmatter title — before the main processing loop runs. This avoids blank-field propagation without requiring another model call. The principle is the same: never trust a sub-agent's output to be complete; always have a deterministic fallback that runs before the stochastic one.

Timeout budgets for relay calls should also account for real-world latency: if a Sonnet relay completes in 18–19 seconds, a 60-second timeout is dangerously tight once retries or prompt-hardening overhead are factored in. Double the timeout to 120 seconds to absorb variance without masking genuine failures.
