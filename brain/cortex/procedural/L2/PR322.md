# Debugging the Hook–Recall–Decision Pipeline: Diagnose, Patch, Verify In-Session

When a recall entry surfaces with an empty gist, the root cause is almost always a missing fallback in the parse chain. To fix gist blanks, locate the `_parse_entry()` function in `recall.py` and add a fallback that reads the YAML frontmatter `title:` field when the primary gist extraction returns an empty string. This ensures that L2 synthesis entries — which lack inline summaries — still produce non-empty gist values for specificity scoring and for display in `additionalContext`. Always verify the fix by reading live gist values from real cortex files, not by trusting the code path alone.

When the Decision Engine's hypothesis bins block tool execution after consumption, the fix belongs at the consumption boundary, not in the gate itself. At the end of `_compose_turn_context()` in `hook.py`, zero each bin with `write_bin(f'decision-hypothesis-{i}', 0)` immediately after the hypotheses have been composed into context. The bins are set to 1 by `dispatch()`, consumed into `additionalContext`, then must be zeroed before `PreToolUse` fires — otherwise the gate sees stale "pending" state and blocks reads. Confirm the fix by attempting a `Read` tool call in the same session; if it succeeds without a gate block, the clearing works.

For subprocess reliability — particularly the identity relay — harden the system prompt to demand JSON-only output and set tight completion constraints. The principle across all three fix classes is the same: diagnose by tracing the data flow from write-point to gate-check, patch at the exact boundary where state transitions, and verify within the live session rather than deferring to a later cycle.
