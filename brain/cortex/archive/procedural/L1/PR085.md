### PR085: og Brain — WhatsApp — Sending Messages via Bridge
- **Domain:** Infrastructure / WhatsApp
- **Learned:** 2026-04-17
- **Strength:** 1
- **Source:** brain-retro

**Cue:** Sending a WhatsApp message from the Brain.

**Routine:**
1. Ensure wa-bridge is connected: curl https://goodhealth.lv/whatsapp/status (status: connected)
2. Send via: curl -s -X POST "https://goodhealth.lv/whatsapp/send" -H "Content-Type: application/json" -d '{"to": "<phone_or_jid>", "message": "<text>"}'
3. Phone number format: just digits (e.g. "37125667732"), bridge auto-appends @s.whatsapp.net
- **Gotcha:** After container restart/reconnect, sock variable may be stale — restart container if send returns "Cannot read properties of undefined"
- **Gotcha (CORRECTED 2026-04-20):** LID format JIDs (e.g. 65069106917478@lid) DO work for sending. Both manual and scheduled agent runs confirmed delivery to LID addresses. Original warning was wrong.
- **Gotcha:** Rate limited to 1.5s between sends
- **Verification:** Response is {"ok": true, "to": "..."}
- **Apply when:** Sending WhatsApp messages programmatically
- **Gotcha (2026-04-22):** Bridge can return HTTP 500 on all JIDs while status=connected. Both @lid and @s.whatsapp.net formats fail. Read works, write broken. Possible cause: stale Baileys sock or WA protocol version expiry. Fallback: use email (send_email MCP).
