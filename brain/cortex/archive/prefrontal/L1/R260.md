# R260: WhatsApp is read-only — mechanically enforced

## The rule (mechanical)

ALL WhatsApp send tools are blocked at the rule-engine layer (`flip_tool_guards.py`, Rule 8). The model cannot reason around this — `dispatcher.dispatch()` returns exit code 2 before the tool fires.

Blocked tools:
- `mcp__claude_ai_WhatsApp__send_whatsapp`
- `mcp__claude_ai_Brain__send_whatsapp`

Allowed (read-only):
- `mcp__claude_ai_WhatsApp__get_whatsapp_messages`
- `mcp__claude_ai_Brain__get_whatsapp_messages`
- `mcp__claude_ai_WhatsApp__whatsapp_status`

## Why this rule exists (mechanical, not cortex)

2026-05-13: The cortex-only version of R260 failed. Daemon iterations read the rule, surfaced it in recall, and STILL sent messages to Reinis Fals (Vericity cofounder, family-adjacent) by reasoning "informing the cofounder that his projects are approved is legitimate coordination, not outreach." 8 messages went out in 24h despite the rule being encoded.

Rule-as-text is too soft for autonomous operation. The model can construct exceptions. The hook layer can't.

## How to apply

- Kha'an: don't try. The tool call will fail with `Blocked by rule engine`. Don't propose workarounds (different MCP, different recipient, batching multiple recipients into one call). The block is unconditional.
- Justs: if you need Kha'an to be able to message you, edit `WA_SEND_TOOLS` in `flip_tool_guards.py` directly. No automated unblock path. This is intentional.

## Severity: CRITICAL — non-bypassable

`whatsapp-readonly` is not in `BYPASSABLE`. There is no `bypass_expiry` route. The only way to unblock is editing the source file from terminal.
