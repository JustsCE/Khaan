# R157: Re-staging requires THREE writes — host state.json, container state.json, AND source task status=open

## The rule

To re-stage a project for verify-cinematic testing, the row only renders shimmering on `/brain` when ALL THREE conditions hold:

1. **Host** `~/.claude/brain/projects/<id>/state.json` → `status: "user_verify_pending"` (unconditionally, not just when status was 'verified')
2. **Container** `/brain/brain/projects/<id>/state.json` → same
3. **Container** `/brain/brain/dashboard/tasks/whatsapp.json` → source task `status: "open"`, `tags: ["VERIFY", "PROJECT"]`

If condition 3 is missing, the row is FILTERED OUT of `/api/dashboard/brain-tasks` (`if (task.status !== 'open') continue;`) — it never reaches the frontend. No row, no shimmer to apply.

R156 covered (1) and (2). This rule extends with (3).

## Why this rule exists (third recurrence)

2026-04-28, three sessions in a row, Justs saw the same broken visual: 1 of 3 staged rows shimmered, the other 2 didn't. Each time the surface diagnosis was different, but the deep cause is the same: re-staging is a **transactional three-step write**, and my scripts kept missing one.

After the user verifies a project via the cinematic, `cmd_user_verify` flips state.json to verified AND `/api/dashboard/projects/:id/verify` closes the source task (`status: done`). On re-stage my script's flip of state.json was conditional `if status == 'verified'` — but if status was already `user_verify_pending`, OR if I'd already written it, the flip became a no-op. Meanwhile the source task stayed `status: done` because nothing in my script reset it. The row vanished from the API, no shimmer.

## How to apply — canonical re-stage script (UNCONDITIONAL writes + post-stage API verify)

```python
import json, subprocess, urllib.request

PROJECTS = ['mog6gyrdc3g7', 'mobile-viz', 'mocj93a79unn']
CONTAINER = 'brain-viz-brain-viz-1'
WA = '/brain/brain/dashboard/tasks/whatsapp.json'

# 1+2: state.json on host AND container — ALWAYS overwrite, never conditional
for pid in PROJECTS:
    host = f'/home/ubuntu/.claude/brain/projects/{pid}/state.json'
    with open(host) as f: s = json.load(f)
    s['status'] = 'user_verify_pending'
    s['round'] = 4
    s['verifiedAt'] = None
    s['userVerifyNote'] = None
    with open(host, 'w') as f: json.dump(s, f, indent=2)
    subprocess.run(['docker','exec','-i',CONTAINER,'sh','-c',
                    f'cat > /brain/brain/projects/{pid}/state.json'],
                   input=json.dumps(s, indent=2).encode(), check=True)

# 3: source task status=open + tags=[VERIFY,PROJECT] — ALWAYS, even if currently done
raw = subprocess.check_output(['docker','exec',CONTAINER,'cat',WA]).decode()
ts = json.loads(raw); seen = set(); new = []
for t in ts:
    if t['id'] in seen: continue
    seen.add(t['id'])
    if t['id'] in PROJECTS:
        t['status'] = 'open'   # 'done' would filter it out of brain-tasks
        t['tags'] = ['VERIFY','PROJECT']
    new.append(t)
if 'mobile-viz' not in seen:
    new.append({'id':'mobile-viz','source':'whatsapp','title':'Simplify Mobile View',
                'status':'open','tags':['VERIFY','PROJECT'],'priority':2,
                'createdAt':'2026-04-24T20:42:39.626515+00:00','createdBy':'pretest',
                'people':{'from':'Justs Ceplis','assignedTo':'Brain'},'confidence':1,
                'description':'staged'})
subprocess.run(['docker','exec','-i',CONTAINER,'sh','-c',f'cat > {WA}'],
               input=json.dumps(new, indent=2).encode(), check=True)

# 4: VERIFY post-condition via API. FAIL LOUD if any row missing or wrong status.
data = json.loads(urllib.request.urlopen('http://localhost:3001/api/dashboard/brain-tasks').read())
rows = {it['id']: it for it in data['project'] if it['id'] in PROJECTS}
missing = [p for p in PROJECTS if p not in rows]
wrong = [p for p,r in rows.items() if r.get('projectStatus') != 'user_verify_pending']
if missing or wrong:
    raise SystemExit(f'STAGE FAILED missing={missing} wrong={wrong}')
print('STAGED OK')
```

## Apply when

- Any time staging a project for verify-cinematic testing
- First instinct after "shimmer not appearing" — check API output before frontend logic. Bug 3/3 times was data-state drift, not rendering.
