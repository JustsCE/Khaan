Philosophy

The Rule Engine is the OS of the brain.

Every alert is one file. 0 is silent, 1 is fire.

Bypassing means doing the work, which is the point.

Every architectural decision in this document follows from those three statements.

The engine routes hook events. It reads binary files. It blocks when any file holds 1. It does not compute the alert, run the cycle, perform the audit, or score the recall — those belong to the cognitive engines (Brain Cycle, Recall, Identity, Decision) that raise and clear their own binaries. The Rule Engine has read access only.

Faking state.json values does not silence an alert — the alert lives in its own binary file, and only the legitimate opener can flip it back to 0. The only way past the gate is doing the actual work the gate is supposed to enforce.

Binaries

Fundamentals

  • One alert per file. Every binary lives in binaries/<name>.bin holding 0 (silent) or 1 (alert). Nothing else in the file.
  • Block on PreToolUse only. On every PreToolUse event, the engine reads the blocking binary set and the rule enforcement results. If any are raised, the engine exits with code 2 and surfaces the message. The user can always talk — UserPromptSubmit is passthrough.
  • Engine never writes. Only the cognitive engines and the bypass mechanism write to binaries. The Rule Engine has read access only.
  • Atomic writes. Every write is tmp-file + os.rename. No partial reads possible.
  • Session reset. SessionStart writes 0 to every binary. Every session begins from a known clean state.

Ten of the 23 binaries above share the same recovery pattern. They are raised by engine failures and automatically cleared on the next UserPromptSubmit, so the brain is never permanently locked out — degraded but unblocked.

The pattern is three-strike forgiveness: the engine retries up to 3 times, then raises the binary and the brain runs without that engine for one turn. The next UserPromptSubmit clears it and the engine tries again fresh.

Rule Enforcement

The brain has rules stored as text in cortex/prefrontal/. Most are followed via recall — the brain reads the rule and trusts itself to obey it. A small subset cannot be trusted to recall alone. These are the mechanically-enforced prefrontal rules: enforced at the OS level by PreToolUse predicates that block the tool call before it executes.

Each rule is evaluated by engines/flip_tool_guards.py :: evaluate() on every PreToolUse event. When a rule fires, the gate name is returned to the dispatcher and merged into the raised list. No .bin file is written — the enforcement is stateless and per-call.

Rule 1 — Brain files via Bash only (use-bash-block)

Claude Code’s Edit, Write, MultiEdit, and NotebookEdit tools trigger permission prompts when targeting files under brain/. The brain’s own engine processes write brain files via Bash (Python scripts), which run inside the engine process and do not trigger permission dialogs. This rule forces the brain to use Bash for brain-internal writes, preventing permission dialog interruptions during normal operation.

Trigger: An Edit, Write, MultiEdit, or NotebookEdit tool call targets any file path under ~/.claude/brain/.

Effect: This also protects the binaries/ directory itself — since it lives under brain/, any attempt to Edit a .bin file directly is blocked. The brain can only modify binaries through engines/bin_io.py :: write_binary(), which runs inside engine processes via Bash.

Bypass: yes, via HMAC token.

Rule 2 — Engine source protected (engine-security-block)

The engine source files at engines/*.py must not be modified during a session. Since Rule 1 already blocks Edit/Write/MultiEdit on all files under brain/ (which includes engines/), this rule only needs to catch Bash commands that redirect output into engine files — for example > engines/recall.py, tee engines/dispatcher.py, or python -c "open('engines/...')".

Trigger: A Bash command contains output redirection (>, >>, tee) or Python open() targeting engines/*.py.

Bypass: yes, via HMAC token.

Rule 3 — Cycle state protected (cycle-security-block)

Cycle-state files (state.json, brain_cycle.py, cycle_phases.py) can only be written while a cycle is actively running. Outside a cycle, these files are read-only.

Trigger: A Bash command writes to a cycle-state file AND state.json :: state is not CYCLE_RUNNING.

Bypass: yes, via HMAC token.

Rule 4 — Docker service must be named (server-security-block)

The brain visualisation at goodhealth.lv runs via Docker alongside other services. A bare docker compose up, down, or restart without an explicit service name would affect all containers, potentially taking down unrelated services.

Trigger: A Bash command matches docker compose up|down|restart with no service name following.

Bypass: no — rephrase the command with an explicit service name.

Rule 5 — No file-permission changes (permissions-block)

File permissions and ownership under brain/ must not be changed. chmod changes who can read, write, or execute a file. chown changes who owns it. Either could lock out the engine processes (by removing their read/execute access) or expose secrets (by making files world-readable). This rule is not redundant with Rule 1 — chmod and chown are Bash commands, not Edit/Write operations, so Rule 1 does not catch them.

Trigger: A Bash command runs chmod or chown (with or without sudo) and the command references a path under brain/.

Bypass: yes, via HMAC token.

Cycle gate — learning-cycle-overdue

This is not a behavioural rule — it is a cycle gate. When learning-cycle-overdue.bin == 1, every tool is blocked except the /brain-cycle skill (or its direct Bash invocation python3 -m engines.brain_cycle). This forces the brain to run its consolidation cycle before doing anything else. The binary is raised by the message-gap check in hook.py and cleared by a successful cycle commit.

Bypass

A bypass token clears one guard for a short time window. Tokens are minted externally — the brain cannot mint its own. The HMAC secret lives at ~/.claude/.bypass.secret (file mode 0400, stored outside brain/ so the brain cannot read or write it).

When consumed, the token is verified by engines/flip_bypass.py :: consume(). On success, the named guard is cleared and the bypass is recorded in state.json :: bypass_expiry. When the time window closes, the guard is automatically re-raised.

Invariant verification

Each cognitive engine owns its own invariant checks. Failure to verify keeps the engine’s binaries raised; the dispatcher only reads the binaries — it does not run the checks. The lists below are the contract every engine must enforce before clearing its gates.

Brain Cycle (6 invariants)

Verified by engines/brain_cycle.py at Phase 8 commit, before clearing learning-cycle-running and flipping the FSM back to NORMAL.

#Invariant
1state.json :: cycleCount incremented by exactly +1 since cycle start.
2cortex/hippocampus.md hash differs from the value captured at cycle start.
3state.json :: active_cycle_nonce matches the nonce written into the cycle-start manifest.
4Phase 5 ran this cycle — schedule.last_promotion_scan_cycle == cycleCount.
5Pending H### count is not strictly growing across the last 3 cycles (consolidation is keeping up).
6Every due phase carries schedule.last_<phase>_cycle == cycleCount (no due phase skipped).

If any check fails, the cycle binaries stay raised and the cycle retries. promote-scan-stale clears only when invariant 4 holds.

Decision (5 invariants)

Verified by decision-engine/flip-decision-verify.py on the SubagentStop event, against the JSON envelope returned by the Decision subprocess.

#Invariant
1hypotheses[] contains exactly 5 entries.
2Lens labels are [Literal, Identity-shaped, Recall-shaped, Contrarian, Minimal-action] — each present exactly once, no duplicates, no extras.
3Source-hash fields match the active-recall.json and active-identity.json files that existed at subprocess launch.
4query_hash matches the user message (sha256 of lower-cased text).
5Tool-uses audit: the subprocess’s tool_uses array includes a Read on every cited evidence ID’s file, plus reads of both active-*.json inputs.

If any check fails, active-decision.json is not written; decision-failed.bin is raised; the next UserPromptSubmit clears it (Anti-lockout).

Recall (2 invariants)

Verified by engines/recall.py before writing active-recall.json and clearing recall binaries.

#Invariant
1Every entry in qualified_entries[] has final_score >= 2.0 (the Recall floor).
2navigation/active-recall.json is written atomically (tmp + os.rename); never observed half-written.

Identity (2 invariants)

Verified by engines/identity.py — Boot at session start, Relay per turn.

#Invariant
1brain/boot-identity.md is present and parses on session start (Boot precondition for Relay to mirror the kernel).
2navigation/active-identity.json is written atomically (tmp + os.rename); never observed half-written.

Dispatcher

Every hook fired by the Anthropic harness enters hook.py, which calls engines/dispatcher.py :: dispatch(). The dispatcher reads binaries, merges rule enforcement results, and returns an exit code.

Routing table

Hook eventBehaviour
SessionStartpassthrough
UserPromptSubmitpassthrough
PreToolUseblock if any binary or rule enforcement gate is raised
PostToolUsepassthrough
SubagentStoppassthrough
Stoppassthrough — logs turn_complete.jsonl (user prompt + assistant response + tool uses)

Only PreToolUse blocks. UserPromptSubmit is passthrough because the user must always be allowed to talk — degraded brain state restricts the agent, not the user’s input. SessionStart is passthrough because Identity Boot fires at SessionStart — blocking it would be self-defeating.

Engine

def dispatch(event, payload, repo_root):
    restore_expired(repo_root)           # re-raise any expired bypass windows

    per_call_gates = []
    if event == "PreToolUse":
        per_call_gates = flip_tool_guards.evaluate(payload, repo_root)

    spec = ROUTING.get(event)
    if spec is None:
        return 0                         # unknown events pass

    raised = []
    for bname in spec["blocking_binaries"]:
        if read_binary(repo_root, bname) == 1:
            raised.append(bname)
    raised.extend(per_call_gates)        # merge rule enforcement gates

    blocks = bool(raised) and spec["behaviour"].startswith("block")
    return 2 if blocks else 0

No imports of engine logic. No knowledge of which binary belongs to which engine. The dispatcher’s job ends at “read files, evaluate rules, block if any raised.”

state.json

The Rule Engine’s private state file at brain/state.json. The cognitive engines read and write it; the Rule Engine (dispatcher) never opens it. All writes are atomic (tmp-file + os.rename).

FSM state. One of four values: NORMAL (healthy), MAINTENANCE_DUE (cycle overdue, waiting for /brain-cycle), CYCLE_RUNNING (cycle in progress), or MAINTENANCE_FAILED (cycle failed 3+ times, degraded). Transitions are driven by hook.py on message-gap detection and by the cycle engine on commit or failure.

Counters. message_counter increments on every UserPromptSubmit. last_brain_update records the counter value at the last successful cycle commit. The gap between them triggers MAINTENANCE_DUE when it reaches 5.

Cycle nonce. A one-time token (active_cycle_nonce) held for the duration of a cycle. Prevents two cycles from running concurrently. Set at cycle start, cleared on commit or failure.

Escalation count. Consecutive cycle failures. At 3, the FSM falls to MAINTENANCE_FAILED.

Bypass expiry. A map of guard name → ISO 8601 expiry timestamp. Tracks active bypass windows for the rule enforcement gates. restore_expired() re-raises guards whose window has closed.

Timestamps. last_session_start, last_transition, last_cycle_end_ts, last_cycle_outcome — diagnostic fields for debugging and cycle health monitoring.
