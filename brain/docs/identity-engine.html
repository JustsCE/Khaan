Philosophy

The Identity Engine compiles The Brain’s self into two payloads on every session.

One is the constant: who The Brain is. The other is the situational: what the current turn calls for.

No tool runs until the latest payload is published.

Every architectural decision in this document follows from those statements.

Identity, voice, and emotional charges live as many small markdown files in three regions on disc. They cannot all fit in context every turn. The Identity Engine is the synthesizer: it picks the entries that matter right now, composes them into a single payload, and hands the result downstream. The skills implement this — one runs once at session start (constant kernel), the other runs once per turn, composing the situational slice that Decision reads alongside Recall’s qualified set.

The "no tool until published" property is mechanical. PreToolUse blocks the parent on the Decision Engine’s hypothesis binaries; the Decision Engine cannot raise those without a current active-identity.json; therefore the parent literally cannot use any tool until Identity Relay finishes for this turn.

Inputs

The engine reads three memory regions for synthesis, one for lookups during scoring, plus a runtime signal.

SourceFormatWhat is read
cortex/identity/L<n>/ID###.mdmarkdown with YAML frontmatter + bodySelf-model facets. Boot reads L1–L4 (L4 dominates by score). Relay reads L1 / L2 / L3 (L4 already in kernel).
cortex/broca/L<n>/BR###.mdmarkdown with YAML frontmatter + bodyEmotional event chains. Same Boot/Relay split.
cortex/amygdala/L<n>/AM###.mdmarkdown with YAML frontmatter + bodyCurrent emotional charges. Same Boot/Relay split.
cortex/fusiform/L<n>/FUS###.mdmarkdown with YAML frontmatter + bodyPerson lookup for charge_bonus (read-only; not a synthesis target)
thalamus.jsonJSONCounter context for recency calculations

YAML. A simple key/value text format used for the metadata block at the top of each markdown entry. Lines like id: ID042, strength: 4, tags: [doctrine, core]. The frontmatter is fenced by --- at the start and end. Everything below the closing fence is the body, plain prose.

Skills

Two skills, same synthesis core, different cadence and scoring. Boot scores L1–L4 across identity / broca / amygdala (L4 dominates by level_bonus, but strong L3 / L2 sub-traits join the kernel as supporting structure). Relay scores L1–L3 only — L4 is mirrored from the kernel rather than re-scored. Both skills compose their final payload via a claude -p --output-format json subprocess wrapped by engines._shared.cli_invoke() — the same mechanism the Decision Engine uses. The CLI selects the model from the active subscription (typically Opus when available); no Anthropic API key is consumed.

Identity Boot — once per session

Trigger: SessionStart hook. Boot runs synchronously inside the hook handler before the harness returns control to the assistant; the kernel reaches the parent as additionalContext on the very first turn.

Reads: cortex/identity/{L1,L2,L3,L4}/, cortex/broca/{L1,L2,L3,L4}/, cortex/amygdala/{L1,L2,L3,L4}/, thalamus.json (counter context).
Writes: brain/boot-identity.md.
Budget: 60 tool calls, 180 seconds. Score floor 1.5.

How it works:

  1. Walk identity / broca / amygdala across L1–L4.
  2. Score each entry with the signals below. No relevance signal — there is no current message at session start.
  3. Rank descending. Drop entries below the 1.5 score floor.
  4. Compose the kernel via claude -p --output-format json: pass the ranked entries to the subagent, which produces a structured kernel grouped by region (L4 entries always at the top of each region group).
  5. Write the resulting IdentitySnapshot to brain/boot-identity.md as YAML frontmatter + body, and return the kernel as additionalContext.

Boot scoring signals

SignalFormulaCap
level_bonusL1 +0.5, L2 +1.5, L3 +3.0, L4 +5.0+5.0
recency_bonus+1.0 if reinforced within last 5 cycles; +0.5 if within 20 cycles+1.0
strength_bonus+0.3 per strength point+2.0

L4’s +5.0 is intentionally above L3’s +3.0 + the maximum recency + strength caps (1.0 + 2.0 = 3.0). Even a stale L4 entry with no recency / strength contribution outranks the most-recently-reinforced L3 entry. L4 = ratified core; everything else can support but cannot supplant.

Worked examples (Boot).

  • An L4 facet, strength 10, reinforced 3 cycles ago: level_bonus +5.0 + recency_bonus +1.0 + strength_bonus +2.0 = 8.0. Top of the kernel.
  • An L4 facet at strength 8, reinforced 50 cycles ago: level_bonus +5.0 + 0 + +2.0 = 7.0. Still in the kernel; L4 alone is enough.
  • An L3 entry at strength 8, reinforced 3 cycles ago: level_bonus +3.0 + recency_bonus +1.0 + strength_bonus +2.0 = 6.0. Joins the kernel as a strong sub-trait beneath L4.
  • An L1 facet at strength 1, reinforced 50 cycles ago: level_bonus +0.5 + 0 + +0.3 = 0.8. Below the 1.5 floor; dropped.

The kernel is small and slow-moving at the L4 layer (L4 promotion happens only on explicit ratification — User or Teacher) and adapts at the L3 / L2 layers as cortex synthesis runs. Session-over-session, the L4 spine is near-identical; the supporting layers drift with reinforcement.

Output schema (Boot IdentitySnapshot): markdown with YAML frontmatter holding session_id, ts, kernel_size, source_hashes, model, plus a body composed of three sections (Identity / Voice / Charges). Diagnostic; the actual delivery channel is the SessionStart hook return value.

Why not write to CLAUDE.md. CLAUDE.md is read by Claude Code at session start, before our hooks fire. If Boot wrote to CLAUDE.md, the change would appear in the NEXT session, not this one. The additionalContext path makes the kernel available immediately. boot-identity.md on disc is for diagnostics and the next-session warm-start path, not the delivery mechanism.

Identity Relay — every turn

Trigger: Relay runs once per turn, before the Decision Engine composes its hypotheses. Decision reads Relay’s output (active-identity.json) the same way it reads Recall’s. Relay does not have its own hook handler.

Reads: cortex/identity/{L1,L2,L3}/, cortex/broca/{L1,L2,L3}/, cortex/amygdala/{L1,L2,L3}/, cortex/fusiform/L<n>/ (FUS lookup for charge_bonus), current user message, thalamus.json, boot-identity.md (for the kernel mirror).
Writes: navigation/active-identity.json.
Budget: 60 tool calls, 60 seconds. Score floor 1.5.

How it works:

  1. Walk identity / broca / amygdala across L1–L3 only. (L4 is excluded — already in the kernel.)
  2. Score each entry with the signals below.
  3. Rank descending. Drop entries below the 1.5 score floor.
  4. Synthesize the situational layer via claude -p --output-format json: pass the ranked entries to the subagent, which composes a single prose paragraph capturing the current voice / mood / situational concerns. The output is a synthesized read of what the brain’s self is bringing to this turn — not a list of entry references.
  5. Write the resulting IdentitySnapshot to navigation/active-identity.json. The snapshot carries the L4 kernel (mirrored verbatim from boot-identity.md) AND the situational paragraph in one file.

Relay scoring signals

SignalFormulaCap
level_bonusL1 +0.5, L2 +1.5, L3 +3.0 (folder = level)+3.0
recency_bonus+1.0 if reinforced within last 5 cycles; +0.5 if within 20 cycles+1.0
strength_bonus+0.3 per strength point+2.0
relevance_bonus+2.0 if the entry shares terms with the current message (overlap coefficient against message tokens, threshold 0.25)+2.0
charge_bonus+1.0 if amygdala entry has subject matching a person named in the current message (FUS### lookup)+1.0

Worked examples (Relay).

  • An L3 entry at strength 8, reinforced 3 cycles ago, in a turn that shares terms with its body: level_bonus +3.0 + recency_bonus +1.0 + strength_bonus +2.0 + relevance_bonus +2.0 = 8.0. Strong contributor to the situational synthesis.
  • An L1 amygdala entry with subject: FUS012, in a turn where the user names that person: level_bonus +0.5 + charge_bonus +1.0 = 1.5. Just clears the floor — the charge is felt this turn.
  • An L1 broca entry, no recency, no relevance, no charge match: level_bonus +0.5 + strength_bonus +0.3 = 0.8. Below the 1.5 floor; dropped.

Output schema (Relay IdentitySnapshot, JSON):

{
  "session_id":   "...",
  "query_hash":   "sha256(lower(message))",
  "ts":           "...",
  "kernel":       [{"id": "ID-final-3", "title": "...", "gist": "..."}, ...],
  "situational":  "...synthesized prose paragraph...",
  "source_hashes": {"identity": "...", "broca": "...", "amygdala": "..."},
  "model":         "claude-cli (subscription, no API key)"

Mechanical enforcement — the gate that protects every turn

Boot runs synchronously inside SessionStart; the kernel reaches the parent as additionalContext on hook return and is system-context for every turn of the session. Relay runs once per turn before the Decision Engine composes — its output is what Decision reads as active-identity.json. Until both payloads are current, the parent cannot use any tool.

The two enforcement points:

  1. SessionStart hook return. Boot runs inside the hook handler; the harness waits on hook return before the assistant sees its first message. The hook returns the kernel as additionalContext; the kernel is in-context from message 1.
  2. PreToolUse on decision-hypothesis-{1..5}. The parent’s tools stay blocked until all five hypothesis binaries clear, and Decision will not clear them without a current active-identity.json. Relay must publish before any tool runs.

Stages

Two lifecycles — one per session, one per turn.

Per-session lifecycle (Boot)

T0   SessionStart fires.
T1   Identity Boot reads cortex/{identity,broca,amygdala}/{L1,L2,L3,L4}/, thalamus.json.
T2   Score + rank entries (L4 always tops); drop entries below the 1.5 floor.
T3   claude -p subprocess composes the kernel from the ranked entries.
T4   Write boot-identity.md.
T5   SessionStart hook returns; kernel injected as additionalContext.
T6   Assistant ready. Boot kernel is system context for every turn this session.

Per-turn lifecycle (Relay)

T0   UserPromptSubmit fires the Decision Engine skill.
T1   Identity Relay launches for this turn.
T2   Identity Relay reads cortex/identity/{L1,L2,L3}/, cortex/broca/{L1,L2,L3}/,
     cortex/amygdala/{L1,L2,L3}/, cortex/fusiform/, current user message, boot-identity.md.
T3   Score + rank L1-L3 entries; drop entries below the 1.5 floor.
T4   claude -p subprocess synthesizes the situational paragraph from the ranked entries.
T5   Write active-identity.json (kernel mirror + situational paragraph).
T6   Decision Engine reads active-identity.json and proceeds.
T7   PreToolUse gate releases when decision-hypothesis-{1..5} all clear; parent unblocked.

Worst case for either lifecycle: skill fails 3 times or runs past its budget; Anti-lockout covers the recovery (degraded but unblocked).

Infrastructure

Hooks, binaries, output artifacts.

Hooks (owned by the Identity Engine)

Hook eventHandler fileWhat it does
SessionStartidentity-engine/identity_boot.pyRun Boot synchronously: read cortex L1–L4, score, compose kernel via claude -p, write boot-identity.md, return kernel as additionalContext.

Relay has no hook handler — it is invoked by the Decision Engine skill as a Python thread, not a registered hook event.

Binaries

BinaryTrigger to raise (1)Cleared by
identity-boot-failed3 consecutive Boot failuresNext UserPromptSubmit (auto-recovery)
identity-relay-failed3 consecutive Relay failuresNext UserPromptSubmit (auto-recovery)
identity-boot-timeoutBoot exceeds 180sNext UserPromptSubmit (auto-recovery)
identity-relay-timeoutRelay exceeds 60sNext UserPromptSubmit (auto-recovery)

No pending binary surface — Boot is gated by SessionStart hook-return semantics; Relay is gated transitively by the Decision Engine’s decision-hypothesis-{1..5} binaries on PreToolUse. Failure binaries above are the only Identity-owned binaries.

Output artifacts

  • brain/boot-identity.md — constant per session. Loaded as system context.
  • navigation/active-identity.json — refreshed per turn. Consumed by the Decision Engine.

Anti-lockout

Three protections, all simple. Worst case the session continues without a kernel or the turn continues without active-identity (degraded), never locked out.

ProtectionWhat it does
Three-strike forgivenessAfter 3 consecutive failures of either skill, the corresponding *-failed binary fires. The next UserPromptSubmit clears all Identity binaries.
Per-skill timeoutBoot 180s → identity-boot-timeout.bin = 1. Relay 60s → identity-relay-timeout.bin = 1. Cleared on the next UserPromptSubmit.
SessionStart resetAll Identity binaries written to 0 at session start.

Every fallback writes a log line so the failure mode is always visible.
