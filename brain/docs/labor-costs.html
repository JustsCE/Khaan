Overview

Labor costs represent personnel costs (direct labor, contracted services) distributed across shops and affiliates in the GEO Controlling (GLC) model. Unlike material and shipping — which are distributed as a simple % of revenue — labor costs use personnel distribution tables that split headcount cost by cost center, then by shop type and destination, based on manually maintained percentage splits.

The process has two parts:

  1. Loading: Base Labor Costs Excel file is prepared by Shemek (Przemysław Lamparski) and provides the monthly absolute labor cost inputs. These feed into SP_CM2_EBITDA_costs_in_ASSA (see LucaNet Load Process doc).
  2. Coefficient adjustment: The labor coefficient in GLC is adjusted to match the actual absolute amount from LucaNet data for months where real data exists.

Base file reference: Base for Labor costs.xlsx — BI SharePoint → Share folder.

Source Data

The primary source is Base for Labor costs.xlsx (BI SharePoint → Share folder). It contains:

  • SKU-level labor cost inputs
  • Packaging split percentages (updated to 20% in early 2026, down from 40%)
  • Cost center assignments

Shemek prepares and updates this file. Justs reviews it with Shemek in the Wednesday session before processing. The Wednesday coordination also ensures any months that have not yet been loaded (e.g. Q1 has not been loaded as of late April 2026) are identified and prioritized.

Personnel Distribution Tables

Labor costs are distributed to shops and affiliates via inline distribution tables defined inside SP_CM2_EBITDA_costs_in_ASSA. These are not in a separate config file — they live directly in the stored procedure code.

The distribution logic uses a CASE WHEN year BETWEEN ... THEN [table] pattern to apply the correct percentages for each time period:

PeriodDistribution approach
Pre-2022Single-column split, simple percentages.
2023–2024Split by shop type (D2C core business, Reseller, Enterprise, etc.).
2025+New percentages, same shop-type split approach.
2026+ (pending)New section needed — add new inline table + new CASE WHEN branch (not yet done as of April 2026).

The inline tables map cost center ID → percentage split across shop types and destinations. The reference for cost center IDs is the GL data information file → "cost center IDs" sheet (list of all cost center + department + entity combinations).

Yearly Maintenance — Adding a New Period

When a new year begins and the personnel cost percentages change, a new inline table section must be added to the stored procedure:

  1. Obtain the new year's personnel cost split percentages from Shemek / HR. These define what % of each cost center's labor cost goes to each shop type / destination.
  2. Open SP_CM2_EBITDA_costs_in_ASSA in the database editor.
  3. Add a new inline table (follow the pattern of existing PS_table1, PS_table2, etc.).
  4. Add a new CASE WHEN branch: 
    CASE
  WHEN year BETWEEN 2023 AND 2024 THEN [use PS_table_2023]
  WHEN year BETWEEN 2025 AND 2025 THEN [use PS_table_2025]
  WHEN year >= 2026               THEN [use PS_table_2026]  -- add this
END
  5. Test by running SP_CM2 against a known month and comparing output totals against LucaNet absolute figures.

Helper reference: GL data information file → "cost center IDs" sheet. Confirm the file location with Justs / Justus Kaiser before editing the procedure.

Labor Coefficient Adjustment

After loading labor costs for a month via SP_CM2, the labor coefficient in the GLC model must be updated to match the real absolute amount from LucaNet.

Why: The CM2 SP uses the last 3 months' average absolute labor cost to project future months. For months with real LucaNet data, the coefficient must reflect the actual figure — not an average — so GLC shows the right absolute amount.

Process:

  1. After SP_CM2 runs, open GLC Alerts Controlling Excel (BI SharePoint → Power BI alerts folder).
  2. For the month with real LucaNet data: compare the labor absolute from LucaNet vs. what GLC currently shows.
  3. Adjust the labor coefficient so the GLC absolute amount matches LucaNet.
  4. Double-check: confirm the coefficient is a ratio/percentage (e.g. ~9%), not an absolute value accidentally entered as a coefficient (the formula has a known confusion risk — see Known Issues).

Primary check is absolute, not percentage: For labor, the % of revenue benchmark (~9.4% per Alisa) is secondary. Always compare against the absolute LucaNet amount first — the ratio naturally varies month to month based on revenue fluctuations.

Validation

After loading and coefficient adjustment:

  • Data Quality tab in GEO Controlling BI report: Check labor rows — compare absolute amounts for months with LucaNet data. Small deviations (~0.5–2%) are normal.
  • GCS alerts Slack channel: Watch for labor-specific alerts the morning after processing.
  • Cross-check with LucaNet directly: For a spot-check, compare the GLC labor total for the loaded month against the lucanet_costs_pivot labor rows. Any material discrepancy (e.g. GLC shows 931k EUR, LucaNet shows 987k EUR) should be investigated before the next monthly cycle.

Example: In October 2025, GLC showed 931k EUR labor while LucaNet showed 987k EUR — a 6% gap that was flagged by PMU. The root cause was that Q1 had not been fully loaded yet, affecting the 3-month average used to project remaining months. Always confirm which months are fully loaded before comparing totals.

Packaging Share Change (2026)

As of early 2026, the labor cost packaging share changed from 40% → 20%. This affects how labor costs are split between direct production and packaging in the distribution tables.

The Wednesday session with Shemek coordinates this change. If the Base for Labor costs.xlsx has been updated with the new 20% split but the SP_CM2 inline table has not yet been updated for 2026 (see Yearly Maintenance section), there will be a discrepancy between what the base file implies and what the procedure calculates. Confirm both are aligned before running the load.
