Overview

The Forecast pipeline ingests daily actuals from Redshift, accepts manual forecasts submitted by demand planners as Excel files via Slack, validates and combines those submissions, calculates a full daily forecast.csv for all unforecasted products, and generates review workbooks at shop, shoptype, and region level. Everything is triggered by Slack slash commands backed by AWS Lambda functions.

Source repo: JustsCE/Forecast — 11 Lambda functions, Python 3.12, S3 as the data bus.

Architecture

Slack slash commands
        |
  FORECAST-review-trigger  (API Gateway → Lambda dispatcher)
        |
        ├── /submit_forecast    → FORECAST-submit-verify → FORECAST-submit
        ├── /review_shop        → FORECAST-review-trigger-verify → FORECAST-review
        ├── /review_shoptype    → FORECAST-review-shoptype
        ├── /review_region      → FORECAST-review-region
        ├── /calculate_forecast → FORECAST-calculate
        └── /verify_forecast    → FORECAST-verify

Daily (EventBridge → Step Functions):
  FORECAST-actuals → FORECAST-inputs

All 11 Lambdas are independent. FORECAST-review-trigger is the Slack entry point — it routes and returns HTTP 200 within 3 seconds to satisfy Slack's timeout, then invokes workers asynchronously.

Daily Schedule

A Step Functions state machine FORECAST-steps runs every morning at 08:00 local time (05:00 UTC):

Start → FORECAST-actuals → FORECAST-inputs → End
  • FORECAST-actuals — Streams raw actuals from Redshift into actuals.csv via S3 multipart upload. Batches of 50,000 rows, 6 MB minimum part size. Aborts the multipart upload cleanly on any failure (no orphaned MPU state).
  • FORECAST-inputs — Syncs Forecast_Input.xlsx from SharePoint via Microsoft Graph API, then reads both the Excel and actuals.csv and applies all three mapping passes (product → forecast_product, shop → forecasted_shop, format/frame → format_clean). Overwrites actuals.csv with the fully-mapped version. Also writes new_products.csv and new_formats.csv surfacing unmapped items.

Submit Flow

  1. Planner uploads an Excel file to the Slack forecast channel and types /submit_forecast.
  2. FORECAST-review-trigger routes to FORECAST-submit-verify.
  3. FORECAST-submit-verify downloads the file from Slack using the bot token, timestamps it (YYYYMMDD_HHMMSS), uploads to s3://bi-automations/Forecast/Seperate Forecasts/submit_[shop|shoptype|region]_NAME_timestamp.xlsx.
  4. Validation (shop files): forecast for current month ≥ actuals; format row totals match Forecast row when both exist; no gaps in the forecasted month range.
  5. Validation (shoptype files): partial coverage check (shops with some months filled, others na); forecast ≥ actuals for current month.
  6. Validation (region files): pass through — no validation.
  7. On failure: post grouped error report to Slack, delete bad file from S3.
  8. On success: post summary (per-product forecast totals, YoY %, delta vs. previous version) to Slack, then invoke FORECAST-submit.
  9. FORECAST-submit reads all valid submit files, resolves priority (shop overrides shoptype overrides region for the same product/shop/period), distributes forecasts to format/frame level, extrapolates missing months, writes seperate_forecasts_combined.csv.

File naming regex: ^submit_(shop|shoptype|region)_NAME(_YYYYMMDD_HHMMSS)?( (\d+))*.xlsx$. Files without a timestamp are still accepted (older uploads). Latest file per shop wins.

Calculate Flow

  1. Reads actuals.csv, Forecast_Input.xlsx (anchor weeks, current month), and seperate_forecasts_combined.csv.
  2. Remaining products: identifies all products not covered by any submit file for the forecast period.
  3. Computes distribution shares from the last 3 full months of actuals (and a Q4 reference window for stability):
    • Format shares per product × destination × shoptype
    • PCS region shares per product × format × destination
    • Day-of-week (DOW) shares per product × destination
    • Shoptype shares per product × destination
  4. Share stability check: compares last-N-months vs. last-Q4 shares. Combinations present in only one window are flagged — signals that a format or shoptype mix may have changed significantly.
  5. Applies shares to monthly totals → daily rows for remaining products.
  6. Combines remaining-product rows with submitted forecasts.
  7. Uploads forecast.csv to S3 and to SharePoint via Microsoft Graph API.
  8. Posts completion summary to Slack via response_url.

Review Flow

Shop-level (/review_shop <name>)

Validates shop name against actuals (via FORECAST-review-trigger-verify) before generating. Workbook uploaded to Slack:

  • Summary sheet: top 15 products by YTD volume (FY t-1 ≥ 1,000). Columns: % of Total YTD, YTD t, YTD t-1, YTD % growth, FY t-1, FY t-2, FY % growth, Last 3W % growth, last 3 months MoM growth. "Other products" row + TOTAL. Destination row below. Revenue-weighted % of Total when revenue data is available.
  • Per-product sheets: Calculations block (metrics transposed); Last 6 weeks block (weekly actuals, LY, YoY %); Monthly block M-2 to M+11 (Actuals, Actuals LY, YoY %, F-YoY %, Forecast — blue fill for forecast months); Available formats block (format/frame breakdown with blue-cell forecast values).

Shoptype-level (/review_shoptype <name>)

Same structure but with dual-region (EU+RoW / US+CA) summary tables and top-5 shop breakdowns per product sheet. Shop-level forecast rows use =IFERROR((1+F-YoY%)*Actuals LY, 0). "Other [shoptype]" aggregates shops not individually listed. Blue fill convention: direct user-input cells only; derived rows (Other, product totals) do not get blue.

Region-level (/review_region <name>)

Supports EU+RoW and US+CA. Column layout per product sheet: TOTAL | gap | TOTAL shoptype1 | shop1 | shop2 | Other shoptype1 | gap | TOTAL shoptype2 | .... Shop columns are grouped (collapsible, hidden by default). Both week table (last 15 weeks) and month table (M-2 to M+5). Conditional formatting: red < −10% YoY, green > +10%.

Verify Flow

  1. Reads cutoff week from Forecast_Input.xlsx info sheet C2.
  2. Reads forecast.csv (all rows — both "remaining" and "separate" source types).
  3. Reads actuals.csv; determines current year t.
  4. Actuals shop remapping (residual approach): named shops get their own rows; "Other [shoptype]" = shoptype total actuals minus sum of named shops. Handles cases like ORWO where FORECAST-inputs groups sub-shops into a single forecasted_shop.
  5. Builds aggregations by (destination_region, forecasted_shop, forecast_product, week): Actuals t, Actuals LY (week-shifted to align with t), Forecast t.
  6. Products sorted by YTD actuals (descending). Only shops present in forecast.csv shown.
  7. Summary sheet: product × destination (no shop), split by region. Pre-cutoff cells grayed; cutoff-week border marker.
  8. Per-product sheets: shop breakdown per region.
