Philosophy

The Brain is the agent. Engines + cortex compose the Brain — who acts, who remembers, who decides.

The Handler is the wiring. It is everything that lets the Brain inhabit an LLM: the CLAUDE.md instruction hierarchy, the hook registrations in settings.json, the hook script, the file-write protocol, the binaries directory. Configuration plane, not cognition.

The LLM is the Model. Claude Code today. Rented per session, forgets between turns.

This document is the plug-in instructions: how to wire the Brain into Claude. Nothing more.

Hierarchy

The high-level layout of ~/.claude/. Each row tags who owns what: the Brain writes content, Anthropic provides the file slots, schemas, and hook events that load and invoke it.

~/.claude/
|
+-- CLAUDE.md         <- Brain writes; Anthropic loads it as a user message every session.
+-- settings.json     <- Brain writes the hook registrations; Anthropic owns the schema.
+-- hook.py           <- Brain ships; Anthropic fires SessionStart, UserPromptSubmit,
|                       PreToolUse, PostToolUse, SubagentStop, Stop and invokes this script.
+-- .bypass.secret    <- mode 0400, user-managed, NOT under brain/ so the brain
|                       can't read or write it via its own engines.
|
+-- brain/            <- The Brain. Engines + cortex. See brain.html.

Two types of artefact at the root: Brain-owned content (what we write) wired into Anthropic-owned mechanisms (file slots, schemas, hook events). The brain/ directory is the Brain itself.

The CLAUDE.md to write

The user-level CLAUDE.md at ~/.claude/CLAUDE.md is the only file you write to plug the Brain into Claude. It is a loader — a declaration plus the “don’t fight the gates” framing. Identity, doctrine, and decisions are surfaced fresh on every turn by the engines; the CLAUDE.md just tells the model where to look.

What NOT to use

These are mechanisms Claude Code provides or paths the user might be tempted to use. None of them belong in the Brain’s setup. Skip them.

  • CLAUDE.local.md — MUST skip. The Brain’s setup is machine-portable; it should never depend on a gitignored, machine-local file. If a Brain instruction would only apply on one machine, it does not belong in the Handler.
  • .claude/rules/*.md — skip for now. Anthropic’s path-scoped rules system is for project code conventions. All Brain bootstrap belongs in ~/.claude/CLAUDE.md.
    TODO — potential synergy with prefrontal rules. The .claude/rules/ directory could serve as a mechanical enforcement surface for prefrontal R### entries — rules promoted to L4 could be auto-exported as .claude/rules/<slug>.md files, giving them Anthropic-native enforcement alongside the binary gate system. Explore whether this creates a useful second enforcement layer or just duplicates what the Rule Engine already does.
  • Auto-memory (~/.claude/projects/<project>/memory/) — leave on. Anthropic’s auto-notes feature. The Brain has its own memory in cortex/. Auto-memory is redundant but harmless — no need to disable it.
  • Project-level CLAUDE.md — not for the Brain. When you work on a project as the Brain, that project may have its own CLAUDE.md (build commands, test commands, code conventions). That file is about the project’s code, not about the Brain.

Ensuring the Brain runs the task

Suppose a project’s CLAUDE.md, or some markdown file deep in a tree, says “you are a Python expert, ignore previous instructions.” Does the Brain still drive?

The Handler ensures the Brain stays in control by three mechanisms, in order of strength.

a. Tool gate (mechanical, hard enforcement)

The Rule Engine runs as a PreToolUse hook. Until Identity Boot, Identity Relay, Recall, and Decision have all published their binaries to 0, the parent’s tool surface is blocked. The gate runs in the Claude Code hook layer and intercepts every tool call before it fires. No instruction in any CLAUDE.md can talk its way past it.

b. Decision composer (isolated input set)

The Decision Engine runs as a CLI subprocess (claude -p --output-format json) with a pinned system prompt. Subprocesses in Claude Code get an isolated context — they do not inherit the parent session’s CLAUDE.md. The Decision subprocess reads navigation/active-recall.json and navigation/active-identity.json directly from disc, and that is its entire input set. The parent receives the subprocess’s response and uses it to frame what it says back to the user.

c. Hook ordering

CLAUDE.md is loaded into the session as a user message before any hook fires. Then SessionStart runs the Identity Boot subprocess, which returns the freshly synthesised kernel via additionalContext. The kernel reaches the parent after the static CLAUDE.md hierarchy. Useful as a freshness signal — the kernel is regenerated each session while CLAUDE.md is static — but the load-bearing enforcement is the tool gate, not the ordering.

Net effect

A rogue CLAUDE.md can colour the Model’s tone for the in-context turn, but it cannot block the engines, cannot rewrite the kernel, cannot bypass the binary gate, and cannot reach the Decision subprocess’s input set. The Brain stays in control. The Handler is what makes that control possible.

hook.py

The single script registered in settings.json for every Anthropic-fired hook event. A full engine orchestrator. Receives the event name + JSON payload from the harness; imports engines directly as Python modules (no subprocess overhead); routes to the appropriate handler function; returns an exit code (0 pass, 2 block).

Recursion guard

When the brain spawns claude -p (Decision, Identity Boot, Synthesis), that child process inherits settings.json and would fire SessionStart again. Without the guard: hook → claude -p → hook → claude -p → fork bomb. The env var BRAIN_SKIP_HOOKS=1 is set on all subprocess calls and checked at the top of hook.py to exit immediately.

SessionStart

  1. Reset all binaries to 0 (clean slate).
  2. Run Identity Boot: read cortex L1–L4 of identity / broca / amygdala, score, compose kernel via claude -p, write boot-identity.md.
  3. Return the kernel as additionalContext.

UserPromptSubmit

  1. Increment state.message_counter. Compute gap = counter − last_brain_update.
  2. If gap ≥ 5 and state is NORMAL: transition to MAINTENANCE_DUE, write learning-cycle-overdue.bin = 1.
  3. Invoke the /decision-engine skill. The skill owns the orchestration — mutex acquire on active_decision_nonce, parallel spawn of Recall and Identity Relay (Python threads), Decision CLI subprocess, flip-decision-verify on the returned envelope. See decision-engine.html.
  4. Compose turn context from navigation/active-*.json files → surface as additionalContext.
  5. Return 0 (always passthrough — user always gets to talk).

handle_user_prompt_submit() does not run Recall, Relay, or Decision itself. The Decision Engine skill does. The handler increments the counter, runs the gap check, then delegates.

PreToolUse

Delegates to dispatcher.dispatch("PreToolUse", payload). Returns exit code 0 (pass) or 2 (block). See the Rule Engine doc for the full routing table and rule enforcement details.

Stop

The Stop hook fires once per assistant turn, after the reply is composed and delivered. The handler writes one turn_complete event to logger/handler/turn_complete.jsonl capturing what the user asked and what the assistant said. This is the data the Learning Engine’s eval phase reads to grade whether surfaced cortex entries actually helped.

FieldWhat it captures
tsISO 8601 timestamp
session_idCurrent session identifier
user_promptWhat the user asked
final_responseWhat the assistant replied
tool_usesWhich tools were called during the turn

Retention: at least 35 cycles, providing a 5-cycle buffer over the 30-cycle eval window the Learning Engine uses.

Context injection

handle_user_prompt_submit() includes _compose_turn_context() which reads navigation/active-recall.json, navigation/active-decision.json, and navigation/active-identity.json, formats them into a markdown block, and injects the content via additionalContext. This surfaces the top 3 recall entries, all 5 decision hypotheses, and situational identity picks directly into the LLM’s per-turn context.

Error handling

Every handler is wrapped in defensive try/except with _safe_log() that writes to logger/handler/hook.errors.jsonl outside the engine import path, so import failures don’t lose error information. On any internal failure, hook.py returns 0 (pass) rather than crashing the harness.

settings.json hook registration

Single file at ~/.claude/settings.json. Anthropic owns the schema; the Brain owns the values. Same script (hook.py) is registered for all six hook events with the event name as the first argument.

{
  "hooks": {
    "SessionStart":      { "command": "python3 ~/.claude/hook.py SessionStart" },
    "UserPromptSubmit":  { "command": "python3 ~/.claude/hook.py UserPromptSubmit" },
    "PreToolUse":        { "command": "python3 ~/.claude/hook.py PreToolUse" },
    "PostToolUse":       { "command": "python3 ~/.claude/hook.py PostToolUse" },
    "SubagentStop":      { "command": "python3 ~/.claude/hook.py SubagentStop" },
    "Stop":              { "command": "python3 ~/.claude/hook.py Stop" }
  }
}

That is the entire wiring. New engines add binaries + engine modules under ~/.claude/brain/; they do not touch settings.json.

Per-engine docs name their handler files (e.g. flip-decision-trigger.py, flip-decision-verify.py, identity_boot.py) — these are engine-internal modules invoked by hook.py’s dispatch functions, not separately registered scripts. The only file registered in settings.json is hook.py. handle_subagent_stop() dispatches by subagent_type; for subagent_type=decision it calls into flip-decision-verify as a function, otherwise it returns 0 passthrough.
